## 1. Capability correctness

Adversarial frame: *can a caller invoke any privileged operation introduced by ADR-0026 / T-014 without holding the capability the security model requires? Does the new dispatch chain widen any existing capability check?*

- **No new capability operation.** OK — ADR-0026 / T-014 added `register_idle` (a kernel-internal boot-time registration call), the `idle: Option<TaskHandle>` field, and dispatch-chain updates to consult `idle` only when `s.ready.dequeue()` returns `None`. None of these introduces a userspace-reachable surface or a capability check; the kernel-internal nature of the change is identical to ADR-0022's original `add_task`-for-idle pattern from a capability-system standpoint.
- **`register_idle` is set-once at boot.** OK — the function asserts `s.idle.is_none()` *unconditionally* (release builds included; γ tightened this from `debug_assert!` to `assert!` per coderabbitai's PR #12 review-round finding). Boot-time programming error becomes a loud, named panic rather than a silent overwrite. The single-idle invariant ADR-0026 relies on cannot be violated by any caller — the only way to break it is for a future BSP to call `register_idle` twice, which fails build-then-panic.
- **Dispatch chain `ready.dequeue().or(s.idle)` introduces no new authority.** OK — both halves of the chain return `Option<TaskHandle>`, and `TaskHandle` is the same opaque slot index used pre-ADR-0026. The chain only changes *which* handle gets dispatched, not *whether* the dispatcher checks any capability before dispatch (it doesn't — capabilities gate IPC operations and future syscalls; they don't gate cooperative-yield dispatch decisions, which are kernel-internal).
- **All Phase A + B0 + B1 (2026-04-28) invariants intact.** OK — re-verified:
  - `cap_copy` peer-depth, `cap_derive` saturating depth cap, `cap_revoke` BFS reachability, `cap_take` ordering, `CapRights::from_raw` reserved-bit masking, `CapabilityTable::new` build-time capacity assertion (γ migrated to inline `const { assert!(...) }` form — equivalent semantics, hard build failure on violation either way).
  - `ipc_send` cap-take-before-state-mutation atomicity, `ipc_recv` `ReceiverTableFull` pre-flight, `IpcQueues::reset_if_stale_generation` slot-reuse reset, `ipc_notify` `&CapabilityTable` immutability vs `&mut` for send/recv.
  - `Scheduler::start` empty-queue panic (now panics only when *both* ready empty AND `s.idle` is `None` per ADR-0026; the boot-time programming-error semantics are preserved).

## 2. Trust boundaries

Adversarial frame: *what new boundaries did ADR-0026 / T-014 introduce, and what untrusted bytes (or values) cross them?*

- **No userspace/kernel boundary added.** OK — same as 2026-04-28. Syscalls remain Phase B5+ work.
- **The new `register_idle` site introduces no new trust boundary.** OK — `register_idle` is a `pub unsafe fn` in `kernel::sched`, called exactly once from `bsp-qemu-virt::kernel_entry` during boot before any task runs. The only "untrusted" inputs (in the boot-time-discipline sense) are the `TaskHandle` (locally allocated from the BSP-owned `TaskArena`), the `entry: fn() -> !` (a BSP-internal function pointer), and the `stack_top: *mut u8` (BSP-controlled). All four are kernel-internal at boot; per the same reasoning that applies to `add_task`, the inputs are trusted-by-boot-discipline.
- **Dispatch fallback path is symmetric with the regular path.** OK — the chain consumed in `start_prelude` / `yield_now` / `ipc_recv_and_yield` reads `s.idle` only when `s.ready.dequeue()` returns `None`. The handle that `s.idle` resolves to is the same opaque `TaskHandle` slot index the FIFO would have provided; no new validation or trust-decision is introduced at dispatch time.
- **No new MMIO / asm surface.** OK — T-014 is pure Rust on the kernel + BSP side; no new audit entries; no GIC / vector-table / timer changes since 2026-04-28.

## 3. Memory safety

Adversarial frame: *can a hostile caller, or an unexpected interrupt arrival, cause any existing audited unsafe region (UNSAFE-2026-0001..0021) to violate the invariants it claims, in light of the new T-014 surface?*

- **Kernel crate's `unsafe` surface unchanged in count and shape by T-014.** OK — `register_idle` follows the exact same momentary-`&mut Scheduler<C>` discipline as `add_task` and the three IPC bridge functions: `let s = unsafe { &mut *sched }` inside an inner block that ends before any caller acquires another momentary `&mut`. The discipline is the one ADR-0021 codified and that UNSAFE-2026-0014 audits; the third Amendment to UNSAFE-2026-0014 (added 2026-05-06) names `register_idle` as the new sanctioned site explicitly. **Adversarial probe:** can a future caller of `register_idle` accidentally hold a `&mut Scheduler<C>` across the function? The function takes `*mut Scheduler<C>`, not `&mut`, so the only `&mut` materialised is inside the inner block; the function returns before the caller acquires any further reference. ADR-0021's non-aliasing rule holds vacuously for `register_idle`'s body and is a structural property of the BSP `kernel_entry`'s call sequence.

- **`assert!(s.idle.is_none())` cannot be bypassed.** OK — γ replaced the prior `debug_assert!` with an unconditional `assert!`. Release builds now panic on a duplicate `register_idle` call rather than silently overwriting. The `s.idle` slot becomes a *write-once* location for the lifetime of the kernel; a future `unregister_idle` API (none planned) would need its own audit entry. **Adversarial probe:** can `s.idle` be written by any path other than `register_idle`? `Scheduler::idle` is `pub(crate)`-default (no explicit visibility on the field; module-private to `kernel::sched`); only `register_idle` writes it. `cargo grep "\.idle\s*=\s"` returns one production hit (`register_idle`) plus the test-only direct-write at `kernel/src/sched/mod.rs:1490+` inside the regression-guard test. ✓

- **The post-T-014 smoke trace produces no observable aliasing or UB.** OK — Miri 152/152 clean across HEAD; the three new T-014 host tests (including the regression guard `unblock_after_yield_dispatches_unblocked_receiver_not_idle`) exercise the dispatch chain with `FakeCpu` + raw-pointer Scheduler discipline and pass under Miri's Stacked Borrows. The full QEMU smoke run produces zero events under `-d int,unimp,guest_errors` for the entire ~5.8–6.3 ms boot-to-end window; `cargo +nightly miri test` was re-run after every PR-round commit.

- **UNSAFE-2026-0019 / 0020 — partial verification under sustained execution.** OK (re-verified) — the 2026-05-06 *partial-verification* + *post-T-014 smoke* Amendments record that the setup sites (GIC `init`, `MSR VBAR_EL1`, `MSR DAIFClr`) are now confirmed under sustained execution with `DAIF.I` unmasked across the full demo run. The IRQ-take + dispatch path remains unverified (no v1 caller arms a deadline); `Pending QEMU smoke verification` status persists for both. Forward-only: full clearance gates on a future preemption-using task that arms a real deadline.

- **UNSAFE-2026-0021 — no observable verification possible from v1.** OK (re-verified) — the 2026-05-06 *no-verification* Amendment records that the timer-write site is unreachable in the v1 demo (no `arm_deadline` caller); the post-T-014 smoke confirms this conclusion (the smoke trace contains zero events; the timer-IRQ path covered by UNSAFE-2026-0021 is dormant). Forward-only: same as 0019/0020.

- **`Aarch64TaskContext` compile-time size guard added (γ).** OK — `const _: () = assert!(core::mem::size_of::<Aarch64TaskContext>() == 168);` mirrors the 2026-04-28 `TrapFrame` guard; turns any future asm/Rust-`repr(C)` drift into a build-time failure. **Adversarial probe:** is the size proof complete? `(10 + 1 + 1 + 1) × 8 + 8 × 8 = 168` matches the `naked_asm!` byte offsets `0/80/88/96/104` in `context_switch_asm`; Track I §ABI boundary at the comprehensive review verified the field-offset audit independently. The guard is the strongest possible safety net.

- **`SchedQueue::new` compile-time `N > 0` assertion added (γ).** OK — turns a `SchedQueue<0>` instantiation into a build-time hard error. The wrap arithmetic in `enqueue` / `dequeue` was already correct under N>0 by short-circuit; the assertion lifts the type-level invariant from "implicit short-circuit" to "structural property". **Adversarial probe:** can any `unsafe` block in the kernel rely on `N > 0` and be widened by the change? The new `assert!` *narrows* the contract (more callers fail at build), so any previously-safe code stays safe; no `unsafe` block widens.

- **`ContextSwitch +Send+Sync` super-trait bound added (γ).** OK — closes the seam Track B flagged: every other HAL trait carried the bound; `ContextSwitch` was the outlier. Adding the bound forces all implementors (production: `QemuVirtCpu`; tests: kernel-private `FakeCpu`) to be `Send + Sync`. Both are zero-sized markers / no-state structs; both already satisfy the bound implicitly. The bound is a future-proofing measure for multi-core safety, not a current safety fix.

## 4. Kernel-mode discipline

Adversarial frame: *does any new code path violate the principle that kernel-mode work be minimal, deterministic, and panic-only-on-impossibility?*

- **`yield_now`'s new "current is idle" branch is constant-time.** OK — `let current_is_idle = s.idle == Some(current_handle)` is a single `Option`-equality compare; `s.task_states[current_idx] = TaskState::Ready` is a single store. The conditional `s.ready.enqueue(current_handle)` is skipped when current is idle, otherwise a bounded enqueue (the SchedQueue's wrap-math has constant cost). The dispatch chain `ready.dequeue().or(s.idle)` is two `Option`-method calls. No new loops; no new allocation; no new panic on the structurally-reachable paths.

- **`start_prelude`'s panic message tightens, but the panic surface does not widen.** OK — the message changed from "scheduler start called with empty ready queue" to "scheduler start called with empty ready queue and no idle task". The `should_panic(expected = "empty ready queue")` test still binds (substring match); the structural-reachability of the panic is *narrower* now (registering only an idle task is sufficient to avoid the panic), so v1's actual reachability is reduced, not widened.

- **`ipc_recv_and_yield`'s Phase 2 Deadlock rollback still leaves the endpoint in `RecvWaiting`.** OK (carry-forward from Phase A + 2026-04-28). The code-review Track A non-blocker on this is queued for δ as an ADR. Today's smoke verification does not change the rollback behaviour: with idle registered, the Deadlock path remains structurally unreachable in v1; the asymmetry is benign.

- **Idle's `WFI` activation is structurally unreachable in v1's cooperative IPC demo.** OK — both `task_a` and `task_b` end with `loop { core::hint::spin_loop() }` (no `yield_now`), so the dispatcher never reaches the empty-queue + `s.idle` fallback during the demo's tail. The post-T-014 smoke trace confirms: zero exception events, zero IRQ entries for the entire ~6 ms run; idle does not run. **Adversarial probe — what if a future task body yields in its tail?** Idle becomes the dispatcher's fallback; `WFI` issues; with no IRQ source armed, the core sleeps until a future `arm_deadline` caller wakes it. Memory-safety: untouched. Liveness: depends on the IRQ-source design that B5+ syscalls bring.

- **`#![deny(clippy::panic)]` etc. in `kernel/src/lib.rs` is preserved.** OK — γ trimmed the layered-deny-list comment to clarify "these layer onto `[workspace.lints]`; do NOT re-state workspace denies here", but did not change the kernel-stricter denies themselves. Track J §J-OBS1's "lift to workspace lints" is queued as a future hygiene item; the current per-kernel placement is correct.
