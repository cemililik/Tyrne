## 1. Capability correctness

Adversarial frame: *can a caller invoke any privileged operation introduced by ADR-0035 / T-017 / ADR-0028 / T-018 without holding the capability the security model requires? Does the new AddressSpace cap surface widen any existing capability check or open a revocation-cascade gap?*

- **New `CapKind::AddressSpace` + `CapObject::AddressSpace(AddressSpaceHandle)` cap variants land additively.** OK — the variant-set is open-ended by design per [ADR-0014](../../../decisions/0014-capability-representation.md) + [ADR-0016 §"Per-type arena pattern"](../../../decisions/0016-kernel-object-storage.md); no §Revision notes change required. The kind discriminator + typed-handle object discipline gives compile-time wrong-kind protection (`CapObject::Endpoint(h)` cannot accidentally route to an `AddressSpace`-expecting wrapper); enforced by `resolve_address_space_cap`'s match-against-`CapObject::AddressSpace(_)` returning `CapError::WrongKind` on miss. **Adversarial probe — can a caller construct a forged `AddressSpaceHandle`?** No: `AddressSpaceHandle` is `pub` (cross-crate visible) but its constructor is `pub(crate) const fn test_handle(...)` only; production paths construct handles exclusively via `AddressSpaceArena::allocate` (returns the index + generation tag), which is the canonical capability-creation site. The handle's generation tag prevents use-after-free (slot reuse bumps generation; stale handle lookup returns `StaleHandle`). Same discipline as `EndpointHandle` / `TaskHandle`; verified by `arena_get_with_stale_handle_returns_none` host test.
- **`cap_create_address_space` is the only cap-minting site and enforces the full discipline.** OK — the wrapper at [`kernel/src/mm/address_space.rs`](../../../../kernel/src/mm/address_space.rs) walks an eight-step preflight chain that closes every leak path under v1's single-core cooperative semantics: (1) lookup parent cap by handle, (2a) check `parent.kind == CapKind::AddressSpace`, (2b) check `parent.rights ⊇ CapRights::DERIVE`, (2c) check `new_rights ⊆ parent.rights` (no-widening), (2d) check `parent_depth + 1 ≤ MAX_DERIVATION_DEPTH` via `CapabilityTable::depth_of`, (3) check `arena.is_full()` / `table.is_full()` capacity preflight, (4) `pmm.alloc_frame()` (first irreversible step), (5) `Mmu::create_address_space(root)` zero-fills the L0 (UNSAFE-2026-0026 contract), (6) `arena.allocate(AddressSpace::from_mmu_address_space(inner))` (preflight-guaranteed succeeds), (7) `table.cap_derive(parent_cap, new_rights, CapObject::AddressSpace(handle))` (preflight-guaranteed succeeds). The wrapper uses [`CapabilityTable::cap_derive`](../../../../kernel/src/cap/table.rs) (NOT `insert_root`) so the new cap is wired into the derivation tree as a child of the parent — revoking the parent cascades to the child per [ADR-0014](../../../decisions/0014-capability-representation.md). **Adversarial probe — can a caller bypass the DERIVE check?** No: step 2b runs unconditionally; `CapRights::empty()` parent cap fails with `CapError::InsufficientRights`. Verified by `cap_create_address_space_rejects_missing_derive` host test. **Adversarial probe — can a deeply-derived parent leak a PMM frame on overflow?** No: step 2d's depth preflight runs *before* step 4's PMM alloc; `DerivationTooDeep` returns with PMM untouched. Closed in commit `8b9f52e`; verified by `cap_create_rejects_too_deep_parent_without_consuming_pmm` host test.
- **`cap_map` / `cap_unmap` check kind only, not per-operation rights bits — known v1 limitation.** **Flagged (non-blocking; deferred to B5+ ADR).** [`resolve_address_space_cap`](../../../../kernel/src/mm/address_space.rs:411) at line 411 checks only `CapObject::AddressSpace(_)`; the rights bits are not consulted before `mmu.map`/`unmap` mutates the page tables. A caller holding any AddressSpace cap (even one minted with `CapRights::empty()` via the rights-narrowing path in `cap_create_address_space`) currently gets full mapping authority. **v1 risk surface:** the kernel-init holds the bootstrap AS cap with all four rights; no userspace exists; the gap is unreachable from any attacker-observable path. **Adversarial probe in a future world:** a B5+ task that mints a narrowed AS cap for a userspace process and hands it across an IPC channel would surface the gap — the recipient could map arbitrary VAs in that AS regardless of the narrowing. **Mitigation forward:** the gap is documented inline at [`resolve_address_space_cap`'s rustdoc](../../../../kernel/src/mm/address_space.rs) and flagged in the [business retrospective §Adjustments](../business-reviews/2026-05-14-B3-closure.md#adjustments) as the B5+ `MemoryRegion` + per-operation rights set extension ADR. Adding `CapRights::MAP` / `UNMAP` / `ACTIVATE` is an ADR-level change (rights set is settled by ADR-0014); deferred to the same ADR that introduces `MemoryRegionCap` for frame-ownership tracking.
- **`cap_revoke` cascades correctly through the new AddressSpace branch.** OK — re-verified: [`CapabilityTable::cap_revoke`](../../../../kernel/src/cap/table.rs) walks the derivation tree BFS-order and calls `free_slot` on every descendant (bumps generation + clears entry). The new `CapKind::AddressSpace` variant is no special case — the tree-walk is kind-agnostic. The regression test `cap_create_uses_cap_derive_so_revoke_parent_invalidates_child` pins this: mint child via `cap_create_address_space`, revoke parent, assert child lookup returns `CapError::InvalidHandle`. **Adversarial probe — does revocation leak any kernel object?** Revoking a parent cap that was the only handle on the child's AS frame does NOT free the PMM frame (the arena slot stays live; `AddressSpaceArena` doesn't reference-count). This is a **known v1 limitation** mirroring the existing `Endpoint` / `Notification` story — destroying the underlying kernel object is a separate (Phase-B4+) concern from revoking caps that reference it. Recorded in [ADR-0028 §"Open questions"](../../../decisions/0028-address-space-data-structure.md); not a closure blocker.
- **Phase A + B0 + B1 + B2 invariants intact.** OK — re-verified against post-PR-#28 source:
  - `cap_copy` peer-depth, `cap_derive` saturating depth cap (now exposed via `depth_of` getter), `cap_revoke` BFS reachability with kind-agnostic walk, `cap_take` ordering, `CapRights::from_raw` reserved-bit masking, `CapabilityTable::new` build-time capacity assertion — unchanged.
  - `ipc_send` cap-take-before-state-mutation atomicity, `ipc_recv` `ReceiverTableFull` pre-flight, `IpcQueues::reset_if_stale_generation` slot-reuse reset, `ipc_notify` `&CapabilityTable` immutability vs `&mut` for send/recv — unchanged.
  - T-015's `ipc_cancel_recv` recovery primitive (PR #17) re-verified: still non-generic, still no new audit entry.
  - T-018's activation-hook closure-threading does NOT widen the `unsafe fn` scheduler bridge — the closures take `AddressSpaceHandle` (a typed handle, not a raw pointer); the closure body's `&AS_ARENA` + `&MMU` borrows are scoped to the post-`&mut Scheduler<C>`-drop instant per the UNSAFE-2026-0014 fifth Amendment's "no cross-referent alias" invariant.

## 2. Trust boundaries

Adversarial frame: *what new boundaries did T-017 / T-018 introduce, and what untrusted bytes (or values) cross them?*

- **No userspace/kernel boundary added.** OK — same as B2. Syscalls remain Phase B5+ work; AddressSpace cap surface is kernel-internal in v1.
- **The `FrameProvider` trait boundary is now load-bearing at runtime.** The B2 closure flagged "the future PMM ADR will need to formalise the non-aliased-frame contract"; T-017 honors this. `tyrne_hal::FrameProvider` requires `alloc_frame()` to return frames that are page-aligned, exclusively-owned, and zero-initialised; `Pmm::alloc_frame` honors all three (alignment is by construction from the extent's base + index; exclusivity comes from the bitmap bit being newly set; zero-fill is UNSAFE-2026-0026). The trait method is `fn alloc_frame(&mut self) -> Option<PhysFrame>`; `Option::None` is returned on exhaustion (no panic). **Adversarial probe — could a misbehaving `FrameProvider` impl return a frame that's already mapped?** The kernel relies on every `alloc_frame` returning a fresh frame not currently part of any AS's page tables. `Pmm::alloc_frame` enforces this via the bitmap: a bit set to `1` means "allocated"; the scan only returns bits set to `0`. The bootstrap reservation (`__boot_pt_l0`/`l1`/`l2_low`/`l2_high` + kernel image + boot stack) marks 161 frames as already-allocated, preventing the PMM from handing them out. **Adversarial probe — could a wrong reservation list cause overlap with the live MMU regime?** `Pmm::new` checks `(a) extent alignment, (b) reservations non-overlapping, (c) reservations inside extent, (d) reservation count ≤ MAX_RESERVATIONS`; mis-config returns `PmmError::*` before any allocation. The reservation list itself is BSP-coded (`bsp-qemu-virt/src/main.rs` derives it from linker symbols); the trust boundary lives at the BSP-author / linker-script-author level, not user-input level. Same shape as `Pl011Uart::new(MMIO_BASE)` — kernel-discipline, not type-system enforcement.
- **The `AddressSpace<M>` arena slot is the only mediation between cap-lookup and `mmu` mutation.** OK — `cap_map`'s flow is: `cap_handle → table.lookup → CapObject::AddressSpace(handle) → arena.get_mut(handle) → &mut AddressSpace<M> → mmu.map(&mut as_.inner, va, pa, flags, frames)`. Every step is a typed pointer dereference inside the arena; no raw pointer manipulation by the caller; no path to alias a different AS's L0 frame. **Adversarial probe — could a caller race two `cap_map` calls into the same AS arena slot?** v1's single-core cooperative model precludes this (no preemption; no SMP). SMP-readiness for `arena.get_mut` is a forward-flagged concern in [ADR-0028 §"Open questions"](../../../decisions/0028-address-space-data-structure.md); not a v1 closure blocker.
- **The activation-hook closure-threading introduces no new boundary.** OK — the closure (`bsp-qemu-virt/src/main.rs::activate_address_space`) is constructed by the BSP and threaded into the scheduler entry points (`start`, `yield_now`, `ipc_recv_and_yield`, `ipc_send_and_yield`) as a `FnOnce(AddressSpaceHandle)` argument. The scheduler treats the closure as opaque; the BSP closure's body reads `&AS_ARENA` + `&MMU` `StaticCell`s and invokes `Mmu::activate(&as_.inner)`. No untrusted input crosses any boundary; the `AddressSpaceHandle` is a typed kernel-internal value with generation-tag staleness protection.
- **No new MMIO surface added.** OK — T-017 + T-018 are kernel + BSP code; no new device range, no new memory-mapped peripheral.

## 3. Memory safety

Adversarial frame: *can a hostile caller, or an unexpected control-flow sequence, cause any audited unsafe region (UNSAFE-2026-0001..0026) to violate the invariants it claims, in light of the new T-017 + T-018 surface?*

- **UNSAFE-2026-0026 (PMM frame-zeroing) lands cleanly under `unsafe-policy.md §3`.** OK — the entry covers `core::ptr::write_bytes(frame_ptr, 0, PAGE_SIZE)` at `Pmm::alloc_frame`'s commit point, with three explicit invariants: (a) frame is page-aligned (constructor enforces `extent.base % PAGE_SIZE == 0`; index_to_pa = base + i*PAGE_SIZE preserves alignment); (b) frame is exclusively owned (bitmap-bit transition `0 → 1` is the single ownership-acquire point; no aliasing because nobody else holds a PhysFrame for the same PA); (c) target is RAM, not MMIO (`Pmm::new` rejects an extent whose `[base, base+N*PAGE_SIZE)` overlaps any device range — currently informal BSP-author convention; flagged for future formalisation). The "Rejected alternatives" subsection records why `MaybeUninit::zeroed_array` + `Box::leak` was rejected (no heap available in `no_std` kernel + would require trait-object boxing through `FrameProvider`'s `&mut dyn` signature). **Adversarial probe — could a caller observe the pre-zero contents of a freshly-allocated frame?** The PMM commits ownership (bitmap-bit set) *before* the zero-fill, but the zero-fill happens unconditionally before the frame is returned to the caller; no escape path between bit-set and zero-fill. The audit's smoke-verification status is "exercised at every boot" — the `tyrne: pmm initialized` banner depends on the alloc path executing during arena ramp-up.
- **UNSAFE-2026-0014 fifth Amendment (activation hook + BSP closure) is scope-additive.** OK — the umbrella's existing invariant "every momentary `&mut` to scheduler state drops before any cross-referent borrow" extends cleanly to the activation-hook site. The scheduler-side hook is structured as: `let next_handle = (&*sched).task_address_space_handles[next_idx]; { let s = unsafe { &mut *sched }; /* update task_states[current_idx] = Blocked etc */ } /* &mut Scheduler<C> drops here */ activate_address_space(next_handle); cpu.context_switch(...)`. The closure body's `&AS_ARENA` + `&MMU` borrows are taken *after* the `&mut Scheduler<C>` drops — no cross-referent alias. Verified by `yield_now_activates_when_tasks_differ_in_address_space` host test (covers row 3 of ADR-0028's §Simulation table) + `ipc_recv_and_yield_with_idle_as_current_returns_deadlock` regression test (covers the post-fix `idle == current` self-dispatch guard added in review-round 3).
- **The post-T-018 smoke trace produces no observable aliasing or UB.** OK — `cargo host-test` 226/226 clean across HEAD `47b0a86` + the closure-trio housekeeping commit `6334881` (all 41 new tests are post-T-018 host tests; covered Miri-clean as the kernel crate runs under `cargo +nightly miri test`). The full QEMU smoke run produces the post-T-018 trace through `tyrne: all tasks complete` cleanly; the only `-d guest_errors` events are the pre-existing PL011 "data written to disabled UART" warnings (526 events with the new banners — exactly one event per UART byte, no non-PL011 events). No new fault classes; no Translation / Permission / Unallocated-Instruction events.
- **The `cap_create_address_space` leak-path closure discipline is now load-bearing.** OK — every fallible step runs before the unrecoverable PMM commitment (step 4). The pattern is verified by the depth-preflight regression test; future kernel-object creation paths (B4 userspace AS, B5 `MemoryRegion`) inherit the same discipline. **Adversarial probe — could a panic between step 4 and step 7 leak a PMM frame?** A kernel panic is fail-stop; the leak is technically "everything is leaked" because the kernel halts. v1 has no panic recovery; the discipline is "no panic on the structurally-reachable path" (`#![deny(clippy::panic)]` etc. in `kernel/src/lib.rs`). No new panic-on-reachable-path is introduced by T-017 or T-018; verified by `cargo kernel-clippy -D warnings` clean.
- **The `ipc_recv_and_yield` Phase-1-mutates-before-Phase-2-checks endpoint-state leak is closed.** OK — the original code path could enter Phase 1 (mutate endpoint state from `Idle` → `RecvWaiting`), then return `Err(SchedError::NoCurrentTask)` at Phase 2's pre-check without reverting the endpoint mutation. Closed in review-round 3 by moving the `current.is_none()` check to *before* Phase 1; regression test `ipc_recv_and_yield_with_no_current_task_leaves_endpoint_idle` pins the invariant.
- **The `ipc_recv_and_yield` self-dispatch UB when `idle == current` is closed.** OK — the dispatcher's `s.ready.dequeue().or(s.idle)` could return `s.current` when `idle` was promoted to running (an idle-task-self-yield scenario); the subsequent split-borrow `let cur_ctx = &mut *ctx_ptr.add(current_idx); let nxt_ctx = &*ctx_ptr.add(next_idx);` would alias the same context slot under release-mode optimisation (debug-mode `debug_assert_ne!(current_idx, next_idx)` would catch it). Closed in review-round 3 by adding an `idle_h != current_handle` guard before the dispatch path; regression test `ipc_recv_and_yield_with_idle_as_current_returns_deadlock` pins the invariant. The mirror site in `yield_now` was already covered by the existing `yield_now_with_no_current_returns_error` test.
- **The kernel + BSP `unsafe` entry count.** Pre-T-017: 25 entries (0001..0025 with 0012 Removed; 24 active). Post-T-018: 26 entries (+1 new; UNSAFE-2026-0026). Existing entries' bodies unchanged except UNSAFE-2026-0014 (5th Amendment, scope-additive) and UNSAFE-2026-0025 (body-correction Amendment for `MmuError::BlockMapped` variant split). **Adversarial probe — does any prior `unsafe` block widen by the new T-018 paths?** No: T-018's only new `unsafe`-touching site is the `Mmu::create_address_space` call (HAL trait method declared `unsafe fn` per [ADR-0009](../../../decisions/0009-mmu-trait.md)) in `cap_create_address_space` step 5; the call-site SAFETY block enumerates the same preconditions ADR-0028's §Simulation row 1 + UNSAFE-2026-0026's invariants already cover (page-aligned, zero-filled, exclusively owned). No new sanctioned site widens any existing umbrella.

## 4. Kernel-mode discipline

Adversarial frame: *does any new code path violate the principle that kernel-mode work be minimal, deterministic, and panic-only-on-impossibility?*

- **`Pmm::alloc_frame` is bounded by extent size.** OK — the scan is O(N/64) word-stepping through the bitmap (N = 32,768 frames for QEMU virt's 128 MiB extent; the bitmap is `[u64; N/64]` = `[u64; 512]`). The hint pointer rewinds on `free_frame` and advances on `alloc_frame`, making the expected-case scan O(1) when alloc/free are interleaved and bounded by allocation density when they aren't. No allocation; no MMIO; no panic on the structurally-reachable path. v1's worst case (32,768-iteration scan) is sub-microsecond on Cortex-A72.
- **`Pmm::new` is constant-time in reservation count.** OK — O(N + R) where R is the reservation list length (currently ≤ 4 in v1 BSP). Linear in N to mark the bitmap; linear in R to apply reservations. No allocation; rejects invalid input with `PmmError::*` before mutating state.
- **`cap_create_address_space` is constant-time in arena + table state.** OK — bounded by `MAX_DERIVATION_DEPTH = 16` (depth preflight) and `ADDRESS_SPACE_ARENA_CAPACITY = 8` (arena preflight) and `CAP_TABLE_CAPACITY = 32` (table preflight); all three checks are O(1). The PMM alloc is O(N/64) per the above; `Mmu::create_address_space` is BSP-defined (`QemuVirtMmu::create_address_space` is a trivial struct wrap, no MMIO). Total path is O(1) + O(N/64) — bounded.
- **`cap_map` / `cap_unmap` paths are O(4) page-table walks.** OK — inherited from T-016 (UNSAFE-2026-0025); the cap-gated wrapper adds: cap lookup (O(1)), arena lookup (O(1)), mmu invocation (O(4)). No loops; no new bounds.
- **Activation hook is single-call, structurally bounded.** OK — `address_space_activation_target` is a pure two-`Option<AddressSpaceHandle>`-comparison; `FnOnce` closure is invoked at most once per scheduler invocation; the closure body's `Mmu::activate` is constant-time on QEMU virt (writes TTBR0_EL1, issues fixed barrier + TLBI sequence). **Adversarial probe — could a future BSP closure leak a panic on a "wrong handle" path?** [`kernel/src/mm/address_space.rs::activate_address_space_handle`](../../../../kernel/src/mm/address_space.rs:356) at line 356 uses `debug_assert!(false, ...)` on a stale handle (dev-time diagnostic) and falls through to a release-mode no-op. **Trade-off accepted in review-round 3 finding F6:** panicking from inside the activation hook's `IrqGuard` scope would leave the kernel in a silent-halt state (the panic handler may not reach the console under interrupts-disabled discipline); the release-build silent no-op is the lesser-of-two-evils choice. The CodeRabbit reviewer accepted this as a "reasonable compromise". v1's `address_space_activation_target` always returns a live handle (all tasks share bootstrap AS); stale-handle case is structurally unreachable.
- **`#![deny(clippy::panic)]` etc. discipline preserved.** OK — T-017 + T-018 add zero panic sites on the kernel-reachable path (test code is allowed pragmas per file-level `#![cfg(test)]`); `cargo kernel-clippy -D warnings` clean at HEAD.
- **DAIF discipline preserved.** OK — T-018's activation hook fires *inside* the scheduler's `IrqGuard` scope (`disable_irqs` + `restore_irq_state` bracket), so the closure body executes with IRQs masked. The closure's `Mmu::activate` writes TTBR0_EL1 + issues barriers; ARM ARM guarantees the architectural ordering even with IRQs masked. **Adversarial probe — could an IRQ fire between the `&mut Scheduler<C>` drop and the activation closure?** No: the `IrqGuard` is declared at the scheduler entry's outer scope and lives until *after* the activation closure + context_switch return; the `&mut Scheduler<C>` drop is purely a borrow-checker artifact, not an IRQ-enable boundary.
- **All allocation paths return typed errors.** OK — `AddressSpaceError::OutOfFrames` / `ArenaFull` / `CapError(_)` cover every failure mode; no `panic!` / `unwrap` / `expect` on the kernel-reachable creation path. Verified by `cap_create_address_space_returns_out_of_frames_on_pmm_exhaustion` + 4 other rejection tests.

## 8. Threat-model impact

- **No new threat-model claim introduced by T-017 + T-018 themselves.** The AddressSpace kernel object is *infrastructure* for future security wins (per-task isolation, kernel-image section permissions, guard-page stacks) — none of which v1 actually enables. The infrastructure does shift one thing: every `Mmu::map`/`unmap` is now reachable from a capability surface (`cap_map` / `cap_unmap`), so the future B5+ rights-set extension ADR has a concrete API to constrain.
- **The `cap_create_address_space` revocation-cascade property is a real defensive win.** Before T-018, only `Endpoint`, `Notification`, and `Task` caps had derivation-tree semantics; AddressSpace caps could not be revoked transitively because they didn't exist. T-018 brings AS caps into the same revocation model — when B5+ introduces per-task AS caps, killing a process by revoking its root cap will cascade to every mapping cap derived from it. The regression test pins this; the test would have caught the pre-fix `insert_root` bug (the cap was minted as a derivation-tree root instead of a child) before merge.
- **The depth-preflight defense is a defense-in-depth win.** A future attacker who managed to chain `cap_derive` calls up to `MAX_DERIVATION_DEPTH` could, in the pre-fix world, trip a PMM leak by triggering `DerivationTooDeep` after PMM alloc. Closed structurally — the depth preflight runs before PMM commitment.
- **The `MmuError::BlockMapped` variant is an API-honesty win, not a threat-model win.** The pre-fix `MmuError::AlreadyMapped` was returned for both "leaf-collision" and "block-collision" cases; the post-fix `BlockMapped` lets callers distinguish "this VA is in a 2 MiB block, not a 4 KiB page" from "this VA has an existing leaf mapping". Disambiguation aids future block-split logic (B3+); no threat-model shift in v1.
- **The `.claude/skills/` → `.agents/skills/` migration is operational, not security-relevant.** The agent skill files are docs; no kernel code references them. The migration affects discoverability for agent runners; trade-off (Claude Code default auto-discovery path no longer used) is recorded in the business retrospective.
- **ADR-0033 (high-half) + ADR-0034 (section permissions) placeholders unchanged.** Both still slot-reserved per ADR-0027 §Decision outcome (a). Gates unchanged: ADR-0033 opens when B5 surfaces per-task TTBR0 swap; ADR-0034 opens with the first attacker-observable execution context (B5+ first userspace).
- **The bot-review-round arc surfaced one B5+-relevant gap: per-operation AS cap rights.** Round-3 F2 noted that `cap_map` / `cap_unmap` check kind only; the deferral rationale (rights set is settled by ADR-0014; new variants need an ADR) is sound. The escalation path is the future ADR pairing `CapRights::{MAP, UNMAP, ACTIVATE}` with `CapKind::MemoryRegion`; named in the [business retrospective §Adjustments](../business-reviews/2026-05-14-B3-closure.md#adjustments) as the natural B5+ opener.

## Verdict

**Approve.**

All eight axes pass; T-017 + T-018 land the AddressSpace kernel-object scaffold + cap-gated `Mmu::map`/`unmap` surface without introducing any new attack surface, capability widening, memory-safety hazard, or threat-model shift. The audit-log discipline holds (UNSAFE-2026-0026 introduced under Operation / Invariants / Rejected-alternatives shape; UNSAFE-2026-0014 fifth Amendment is scope-additive; UNSAFE-2026-0025's body-correction Amendment for the `MmuError::BlockMapped` variant split respects the introducing-commit-boundary discipline because the variant was forward-flagged in the introducing commit's "Future direction" paragraph). The five-round PR #28 review arc surfaced and closed three load-bearing memory-safety items (PMM leak via depth-preflight gap; `ipc_recv_and_yield` endpoint-state leak via Phase-1-mutates-before-Phase-2-checks; `ipc_recv_and_yield` self-dispatch UB when `idle == current`) — all closed before merge with regression tests pinning the post-fix invariants.

The post-T-018 smoke trace is the closure's most important verification artefact — recorded verbatim in the [business retrospective §"Smoke trace"](../business-reviews/2026-05-14-B3-closure.md). The AddressSpace kernel-object scaffold is now live; B4 work can build on a capability-mediated MMU surface with the depth-preflight + leak-path-closure disciplines already in place.
