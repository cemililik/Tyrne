## 1. Capability correctness

Adversarial frame: *can a caller invoke any privileged operation introduced by ADR-0027 / T-016 without holding the capability the security model requires? Does the new MMU surface widen any existing capability check?*

- **No new capability operation.** OK — T-016 added the `MapperFlush` newtype + return-type extension on `Mmu::map` / `unmap`; the BSP-side `QemuVirtMmu` impl + `mmu_bootstrap` bootstrap routine; the `.boot_pt` linker reservation. None of these introduces a userspace-reachable surface or a capability check; the entire MMU surface is kernel-internal infrastructure. The future `MemoryRegionCap` grant path (B5+) will sit on top of these primitives, but that ADR is unwritten and out of scope for B2.
- **`MapperFlush::new` is `pub` (cross-crate visible) but cannot be misused for memory unsoundness.** OK — the constructor signature is `pub const fn new(va: VirtAddr) -> Self`, which `bsp-qemu-virt::QemuVirtMmu` requires to mint tokens at every `Mmu::map`/`unmap` return. The doc-comment records that "kernel code never constructs tokens directly" by convention. **Adversarial probe:** can a misbehaving caller mint a `MapperFlush` for a VA they don't own and trigger a TLB invalidate that affects another mapping? The token's only "power" is to call `Mmu::invalidate_tlb_address(va)` — which is a TLB hint, not a memory-safety operation. A spurious invalidate degrades performance (the next translation walks the page tables again instead of using a cached TLB entry) but cannot violate any memory-safety invariant. Soundness analysis recorded in the [`MapperFlush::new` doc-comment](../../../../hal/src/mmu/mod.rs) + [PR #23 review-round 2026-05-09 Finding 5](../../tasks/phase-b/T-016-mmu-activation.md#review-history).
- **`Mmu::map` rejects unrepresentable flag combinations.** OK — `QemuVirtMmu::map` (and `FakeMmu::map` for parity) reject `MappingFlags::DEVICE | MappingFlags::EXECUTE` up-front with `MmuError::InvalidFlags` rather than letting `flags_to_descriptor_bits` silently coerce the EXECUTE bit into PXN=1/UXN=1. MMIO is never executable per [ADR-0027 §Decision outcome (b)](../../../decisions/0027-kernel-virtual-memory-layout.md); the trait surface is now closed against this misuse. **Adversarial probe:** is the validation comprehensive? The check is `flags.contains(DEVICE) && flags.contains(EXECUTE)` — covers the kernel-execute-MMIO and user-execute-MMIO cases. `DEVICE | USER | WRITE` (user-writable MMIO) is *allowed* because future memory-mapped userspace devices are a plausible B5+ scenario; v1 has no userspace so the path is currently unreachable. The check is the minimum that catches the misuse class; tightening to additional combinations is a follow-up if a B5+ task surfaces a different concern.
- **All Phase A + B0 + B1 (2026-05-07 closure-trio) invariants intact.** OK — re-verified:
  - `cap_copy` peer-depth, `cap_derive` saturating depth cap, `cap_revoke` BFS reachability, `cap_take` ordering, `CapRights::from_raw` reserved-bit masking, `CapabilityTable::new` build-time capacity assertion.
  - `ipc_send` cap-take-before-state-mutation atomicity, `ipc_recv` `ReceiverTableFull` pre-flight, `IpcQueues::reset_if_stale_generation` slot-reuse reset, `ipc_notify` `&CapabilityTable` immutability vs `&mut` for send/recv.
  - `Scheduler::start` empty-queue panic preserved (T-014's ADR-0026 idle-fallback still in effect; T-016 does not touch the scheduler).
  - T-015's `ipc_cancel_recv` recovery primitive (PR #17, post-B1) re-verified: still non-generic, still reuses `IpcError::InvalidCapability` for cap-validation failure path, no new audit entry, no new monomorphisation fan-out.

## 2. Trust boundaries

Adversarial frame: *what new boundaries did ADR-0027 / T-016 introduce, and what untrusted bytes (or values) cross them?*

- **No userspace/kernel boundary added.** OK — same as B1. Syscalls remain Phase B5+ work; MMU is on but no userspace yet.
- **The MMU now mediates every kernel memory access** — but no new trust boundary is introduced because every kernel address is identity-mapped to its physical address per ADR-0027 §Decision outcome (a). The MMU is structurally on, semantically transparent for v1 kernel code. The translation regime adds two architectural side effects: (a) MMIO accesses obey the device-nGnRnE attribute (no speculative reads, no write merging); (b) RAM accesses obey the normal-cached attribute (D-cache + I-cache active). Both are *strengthenings* of the architectural contract, not weakenings.
- **The `mmu_bootstrap` routine introduces a single-caller boundary.** OK — `unsafe fn mmu_bootstrap()` is called exactly once per boot, at EL1, with `.boot_pt` pre-zeroed by `_start`'s BSS-zero loop, before any subsequent MMIO step. The function's `# Safety` doc-comment lists every precondition explicitly. **Adversarial probe:** could a future BSP author call `mmu_bootstrap` more than once, or in a context where the preconditions don't hold? The preconditions are enforced by reading the doc-comment and the surrounding `kernel_entry` flow; no runtime check (a runtime check would require atomic state which v1 cooperative single-core boot doesn't have). The discipline is the same as `Pl011Uart::new(MMIO_BASE)` — kernel-boot discipline, not type-system enforcement. Trust boundary at the BSP-author / kernel-discipline level, not user-input level.
- **`Mmu::map`'s `&mut dyn FrameProvider` parameter is the only kernel-internal trust boundary.** OK — the `FrameProvider::alloc_frame` contract requires "page-aligned, exclusively-owned, zero-initialised" frames. v1 has no real PMM; the `&mut dyn FrameProvider` parameter is exercised only by host tests (`VecFrameProvider`). When the future PMM lands (B3 first task), the contract becomes load-bearing — recorded as [B2 closure business retro §Adjustments item "B3 PMM bring-up"](../business-reviews/2026-05-09-B2-closure.md#adjustments). **Adversarial probe:** could a misbehaving `FrameProvider` return aliased frames? The page-table walker would install two paths through the same intermediate table, leading to a translation aliasing hazard. The trait's safety contract is "must return non-aliased frames" — currently informal. The future PMM ADR will need to formalise this contract; flagged forward.
- **No new MMIO surface added by T-016.** OK — T-016 maps the existing GIC + UART MMIO ranges with explicit device-nGnRnE attributes (UNSAFE-2026-0019 / 0020 / 0021's MMIO sites are now translated through the new descriptors), but does not introduce any new MMIO consumer. The previously-flagged "speculative read on device range without device-nGnRnE attribute" theoretical concern (B0/B1 era) is now closed: every MMIO access goes through the strict device-memory mode.

## 3. Memory safety

Adversarial frame: *can a hostile caller, or an unexpected interrupt arrival, cause any audited unsafe region (UNSAFE-2026-0001..0025) to violate the invariants it claims, in light of the new T-016 surface?*

- **Four new audit entries land cleanly under the project's `unsafe-policy.md §3` discipline.** OK:

The "pre-zero discipline" invariant (`.boot_pt` lives inside `[__bss_start, __bss_end)` so the BSS-zero loop pre-zeros every byte before `mmu_bootstrap` runs) was verified by the smoke trace's clean completion.
  - **UNSAFE-2026-0023** (`MSR TTBR0_EL1` in `QemuVirtMmu::activate` + bootstrap-Amendment for MAIR/TCR/TTBR/SCTLR writes): EL1 + non-VHE precondition asserted by `QemuVirtCpu::new`'s self-check (UNSAFE-2026-0016); architected ordering enforced by surrounding ISBs; `options(nostack)` is the post-2026-05-09-Finding-1 conservative annotation (the prior `nomem` was technically correct but the conservative form treats the asm as a memory clobber). The single-core v1 doesn't hit the SMP race; the discipline lands now without forcing a future asm rewrite when secondary cores come online.
  - **UNSAFE-2026-0025** (`Mmu::map`/`unmap` page-table walker): six invariants explicitly named (root-frame validity, index bounds via `(va >> shift) & 0x1FF`, volatile access discipline, write-ordering at the leaf, caller's TLB-invalidate responsibility via `MapperFlush`, encoder correctness via the host-tested `vmsav8::page_descriptor` / `table_descriptor`). The MMIO accesses these audit are now translated through device-nGnRnE attributes — *strengthening* the safety argument, not weakening it. Their `Pending QEMU smoke verification` status notes carry forward unchanged (the IRQ-take / dispatch path remains structurally unverified pending a future deadline-arming caller).
  - UNSAFE-2026-0001..0011 / 0013..0018 / 0022..0025: each re-read against the post-PR-#23 source; no in-place body edits; cross-references all point at the post-rename `hal/src/mmu/mod.rs` path (verified by grep).
- **The post-T-016 smoke trace produces no observable aliasing or UB.** OK — Miri 185/185 clean across HEAD `b0035ce`; the 12 new `mmu::vmsav8::tests` (descriptor encoders) + 6 new `MapperFlush` token-semantics tests + 3 contract-parity tests (`map_rejects_unaligned_va`, `unmap_rejects_unaligned_va`, `map_rejects_device_plus_execute`) all pass under Miri's Stacked Borrows. ## 4. Kernel-mode discipline

Adversarial frame: *does any new code path violate the principle that kernel-mode work be minimal, deterministic, and panic-only-on-impossibility?*

- **`mmu_bootstrap` is constant-time.** OK — Step 1 has two bounded loops (9 iterations for the device range + 64 iterations for the RAM range; both constants); Step 2 is five system-register writes + one ISB; Step 3 is six instructions (TLBI / DSB / IC IALLU / DSB / ISB / SCTLR RMW / ISB). No allocation; no MMIO; no panic on the structurally-reachable path. The function is `unsafe fn` with a `# Safety` block listing every precondition; structural unreachability (e.g., calling `mmu_bootstrap` twice) is a kernel-discipline error that produces a Translation Fault on the second flip-already-flipped, caught by the EL1 panic vector.
- **`QemuVirtMmu::map` / `unmap` are bounded by VA depth.** OK — the four-level walk visits at most 4 page-table frames (one per level); each level is a 4 KiB read + optional 8-byte write. No loops; no allocation; the only allocation source is the caller's `&mut dyn FrameProvider`, which the kernel owns and bounds. **Adversarial probe — could a malicious VA cause the walker to escape the frame?** The per-level index is `(va >> shift) & 0x1FF`, bounded to `[0, 511]` by construction; `parent.add(idx)` therefore stays within the 4 KiB (`512 × 8 B`) frame. A `debug_assert!(idx < ENTRIES_PER_TABLE)` mirrors this (release builds skip the assert; the bit-mask makes it provably impossible to overrun anyway).
- **`QemuVirtMmu::map`'s flag validation is constant-time.** OK — two `Option`-equality compares (`flags.contains(DEVICE)` + `flags.contains(EXECUTE)`); single boolean AND; conditional `Err` return. The flag-validation is the only new pre-walk check; all other validations (VA alignment via `va.0.is_multiple_of(PAGE_SIZE)`) are single arithmetic compares.
- **`MapperFlush::flush` is a single TLBI + barrier sequence.** OK — `mmu.invalidate_tlb_address(va)` issues `DSB ISHST + TLBI VAE1, x + DSB ISH + ISB` (post-Finding-4). Constant-time at the architectural level; sub-microsecond on Cortex-A72.
- **`#![deny(clippy::panic)]` etc. in `kernel/src/lib.rs` is preserved.** OK — T-016 did not touch the kernel crate's lint configuration. The HAL crate gained one `assert!` in `MapperFlush`'s test (host-only; kernel-internal panics at unrecoverable invariant violations remain governed by the kernel-stricter denies). **Adversarial probe — does T-016 introduce any kernel-side panic on a structurally-reachable path?** `bsp-qemu-virt`'s `kernel_entry` calls `mmu_bootstrap()` once; if the routine were to fault during Step 3, the panic vector takes over (per the [B2 closure business retro §Axis 4 limit-of-defence note](../business-reviews/2026-05-09-B2-closure.md): the defence covers ~80 % of failure surface; corrupt `L2_high[0]` or `L2_low[72]` descriptors would break even the panic vector's fetch / UART write). No new panic-on-success-path is introduced.
- **The `kernel_entry` re-ordering preserves the K3-12 DAIF discipline.** OK — interrupts remain masked from `_start`'s first instruction (per [boot.s §K3-12](../../../../bsp-qemu-virt/src/boot.s)); the post-T-016 `kernel_entry` order is `hello print → VBAR_EL1 install → boot_ns snapshot → mmu_bootstrap → "mmu activated" print → GIC init → DAIF.I unmask → timer banner → demo`. DAIF.I is unmasked AFTER `mmu_bootstrap` returns and AFTER `gic.init()` completes — no IRQ can fire while the MMU is being staged. **Adversarial probe — could a non-IRQ exception fire during `mmu_bootstrap`?** Synchronous exceptions (Translation Fault on a bad descriptor write, etc.) would route to the EL1 sync vector at `tyrne_vectors+0x000` (Current EL with SP_EL0) — installed before `mmu_bootstrap` runs. The defence covers everything except recursive faults from corrupt kernel-image / UART descriptors, which are caught by the host-tested encoders + ADR-0027 §Simulation review discipline.

- **No new threat-model claim introduced by T-016.** OK — T-016 is *infrastructure* (MMU on with identity layout) for future security wins (B5+ userspace isolation, B5+ kernel-image section permissions, B5+ guard-page stacks). - **The MMU activation deliberately leaves several future-security items as named placeholders.** OK — ADR-0033 (high-half kernel migration) and ADR-0034 (kernel-image section permissions) are named-but-unallocated forward-flags. The deferral is well-flagged in the ADR + business retro + threat-model document. - **The ADR §Simulation discipline is now load-bearing for state-machine ADRs.** ADR-0027's five-row table walked the worst-case interaction (SCTLR.M=1 transition) end-to-end; the implementation hit zero descriptor / encoding bugs as a result. The pattern-of-record continues from ADR-0026 (queue-state simulation; T-014's regression-guard test) and ADR-0032 (Phase-2 Deadlock simulation; T-015's `cancel_recv` rollback tests). Three data points — the rule is stable. The threat-model benefit is real: simulation tables surface bug classes (off-by-one VA ranges, missing AF bit, EPD0 accidentally `1`) that prose-only ADRs miss.
- **The bot-review-round arc surfaced one architectural concern: SMP-readiness of the TLBI sequence.** The Finding-4 "DSB ISHST before TLBI" suggestion was applied to `QemuVirtMmu::activate` + `invalidate_tlb_address` + `invalidate_tlb_all`. Single-core v1 didn't hit the SMP race (peer-core TLB walks during a TLBI broadcast); the fix lands the architectural canonical pattern now without forcing a future asm rewrite when secondary cores come online. **Adversarial probe — does the missing `DSB ISHST` matter for v1 correctness?** No — v1 is single-core; no peer core exists to race. The fix is forward-compat-only; its absence pre-fix was a latent bug for SMP, not an active correctness gap. The post-fix sequence aligns with Linux's `arch/arm64/include/asm/tlbflush.h` exactly.

## Verdict

**Approve.**

All eight axes pass; T-016 / ADR-0027 land MMU activation without introducing any new attack surface, capability widening, memory-safety hazard, or threat-model shift. The audit-log discipline holds (UNSAFE-2026-0022 / 0023 / 0024 / 0025 introduced under Operation / Invariants / Rejected-alternatives shape; 0023 / 0024 carry bootstrap-Amendments + 2026-05-09 smoke-verification Amendments; 0025 retains its `Pending QEMU smoke verification` status note pending the first B3+ runtime caller). Phase A + B0 + B1 inheritance items remain at their previously-recorded severity; no new B2-era item is escalated.

The post-T-016 smoke trace is the single most important verification artefact this closure produces — recorded in the [business retrospective](../business-reviews/2026-05-09-B2-closure.md) §"Smoke trace" and pasted verbatim into the [T-016 review-history row](../../tasks/phase-b/T-016-mmu-activation.md). The MMU is now structurally on; B3 work can build on a live translation regime with MMIO obeying device-nGnRnE attributes and RAM obeying normal-cached attributes.
