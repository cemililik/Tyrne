## What changed in the plan

- **B2 milestone closed without re-opening.** Unlike B1's "rolled back, refilled" arc (B1 was implementation-claimed 2026-04-28, then re-opened 2026-05-06 by smoke regression, then truly closed 2026-05-07 post-T-014), B2 closed cleanly on first attempt. The `write-adr` §Simulation discipline (codified during B1 closure) made the difference: ADR-0027's five-row table walked the SCTLR.M=1 transition through every potential failure mode before the Accept commit, and the implementation hit none of them. Smoke verified clean on first run.
- **B2 prep was bundled into a single integration PR (#22)** consolidating three original PRs (#19 ADR-0027 design, #20 path-drift sweep, #21 P10 harness). The maintainer's choice to integrate rather than land sequentially halved the review-surface count without losing bisectability — each commit on the integration branch was independently reviewable. Pattern worth retaining for future multi-artefact prep arcs.
- **The post-T-015 perf re-baseline became the precondition for measuring B2.** The 2026-05-07 B1 trio's perf baseline included a §Post-T-015 amendment that flagged "single-run boot-to-end timings should not be over-read until P10 lands". P10 (`tools/perf-harness.sh`) landed in PR #22 *before* T-016, so T-016's perf claim is a harness band (3 percentiles, 20 iterations) rather than a single-run anecdote. The discipline-shift is the most important non-implementation change of B2.
- **The MapperFlush flush-token discipline is the first project-wide `#[must_use]` enforced API contract.** Before T-016, the project's `unused_must_use = "deny"` workspace lint had no load-bearing user — `Result<(), Err>` types could be ignored without consequence in the kernel-internal call sites. `MapperFlush` is the first surface where the lint genuinely catches a bug class (forgotten TLB invalidate). Pattern-of-record for future HAL trait surfaces where mutation requires a follow-up step.
- **No B-phase milestone re-shuffling.** B3 (next milestone, no canonical name yet — likely PMM bring-up) becomes active. The phase-b plan's sub-breakdown remains: PMM as a separate B-phase task; per-section kernel-image permissions (ADR-0034) deferred until B5+; high-half migration (ADR-0033) deferred until B5 per-task TTBR0 swap.

## What we learned

### The §Simulation discipline did its job

ADR-0027's five-row table walked the kernel from `SCTLR_EL1.M = 0` (PC at PA `0x4008_0000`) to `SCTLR_EL1.M = 1` (PC at the same identity-mapped VA, caches active). The single-most-likely failure mode (Step 3 — flipping the bit before the regime is staged correctly) was the table's central focus, with explicit citations of every system-register write, every barrier, and the architectural reason for `DSB ISH` rather than `DSB NSH`.

The empirical evidence: smoke ran clean on the first attempt. T-016's review-rounds caught compiler-side concerns (`nomem` on inline asm, missing `DSB ISHST` before TLBI for SMP-readiness, `DEVICE | EXECUTE` flag-validation gap, `MapperFlush::new` visibility documentation drift) but **zero descriptor-encoding bugs**, **zero MAIR / TCR field-encoding bugs**, **zero ordering bugs** at the architecture level. The §Simulation table is the prevention; the host-tested encoders (`tyrne_hal::mmu::vmsav8`) are the verification. Together they made T-016 a "did exactly what the ADR said" implementation — which is the right outcome but worth noting because it's not what happened in B1's first attempt.

This is the §Simulation rule's first productive use (ADR-0026 was the retro-source; ADR-0032 applied it to a recovery primitive; ADR-0027 applies it to a state-machine *design* before any code lands). The pattern is now load-bearing for future state-machine ADRs: ADR-0028 (address-space data structure) and ADR-0030 (syscall ABI) will both need it.

### The integration PR (#22) bundled three concerns; the bot review-rounds caught the integration-discipline drift

PR #22 consolidated three originally-separate concerns: ADR-0027 design, P10 harness, 2026-05-06 path-drift sweep. The integration discipline added ~180 link-instances of relative-path verification across 7 review-track files; coderabbitai's review-round flagged path-drift in **one** location the maintainer had missed, and gemini flagged a stale review-round-claim banner number ("#19/#20/#21" → "#22"). Both findings were factual-mechanical (the integration shifted the surface; the docs lagged).

Bot reliability for integration PRs scales with the surface count. Solo PRs land with bot signal on style + factual findings; integration PRs benefit from a wider sweep because the cross-cutting effects are exactly what the per-file reviewer mode misses. **Discipline note:** when integrating multi-artefact prep, expect ~2–3 cross-cutting findings the bots catch that the maintainer wouldn't have without the explicit integration step. The cost is one review-round; the value is closing the integration drift before main.

### The PL011 "data written to disabled UART" warning was pre-existing and is a v1 trade-off, not a T-016 regression

T-016's smoke surfaced 379 `LOG_GUEST_ERROR` events from QEMU's PL011 model (one per byte of UART output). The maintainer almost classified this as a T-016 regression until a comparison run on `main` pre-T-016 showed 358 events (same warning, `+21` exactly = the new `tyrne: mmu activated\n` line bytes). The warning is a long-standing trade-off: Tyrne's `Pl011Uart` rides QEMU's reset state (UARTEN=0 at reset; QEMU's stdout-passthrough still works regardless) rather than performing the canonical PL011 init sequence (UARTLCR_H + UARTIBRD/UARTFBRD + UARTCR.UARTEN=1). The trade-off is intentional for v1 (reset-state UART works on QEMU and is one fewer init step in the boot sequence) but the warning noise dilutes any future signal in `-d guest_errors`.

**Adjustment:** queue a B-phase BSP task to land proper PL011 init (clear UARTCR + configure baud + UARTEN=1). Not a T-016 blocker; the noise has been present since the very first kernel boot. Recorded in PR #23's review-round commit message and in this retrospective for tracking.

### The §Simulation table → host-test pairing is now codifiable

ADR-0027's §Simulation table walked five steps; T-016 added 12 new `mmu::vmsav8::tests` host tests pinning every `MappingFlags` permutation against the encoding table in [`memory-management.md`](../../../architecture/memory-management.md) §"Page-table entry encoding". The pairing — *ADR's simulation table as design checklist; host tests as encoding-correctness checklist* — is the same pattern T-014 used to good effect (`unblock_after_yield_dispatches_unblocked_receiver_not_idle` was the host-test form of ADR-0026's queue-state simulation table). At three data points (T-014, ADR-0032 / T-015, ADR-0027 / T-016), the pattern is no longer a lone observation.

**Codification candidate:** extend the `write-adr` skill's §Simulation step with an explicit "every simulation row should map to a host-test or audit-log line in the implementing task". Trigger: when ADR-0028 (address-space data structure) opens, draft the test-mapping alongside the simulation table.
