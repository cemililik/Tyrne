# Track B — HAL & test-HAL

- **Agent run by:** Claude general-purpose agent, 2026-05-06
- **Scope:** HAL trait surface (`Console`, `Cpu`, `ContextSwitch`, `Mmu`, `Timer`, `IrqController`, `IrqGuard`, `IrqState`, `FmtWriter`, free `current_el`, tick-arithmetic helpers) + the five `tyrne-test-hal` fakes; trait-contract drift across HAL impls.
- **HEAD reviewed:** 214052d

## Findings

### Blocker

*No blockers.* The trait surface as it stands is consistent with the ADRs that pinned each trait (0007/0008/0009/0010/0011/0020); the dev-dep posture is correct; the `Mmu` placeholder is genuinely unconsumed by kernel or BSP and so cannot regress anything yet.

### Non-blocking

- [test-hal/src/cpu.rs:94](../../../../test-hal/src/cpu.rs#L94) `FakeCpu` does not implement `ContextSwitch`, so the kernel's scheduler tests cannot use the test-HAL's `FakeCpu` and have grown a parallel local `FakeCpu` inside `kernel/src/sched/mod.rs:850-908` that re-implements both `Cpu` and `ContextSwitch` (with associated `FakeTaskContext`). This is the exact "generation drift / fake behind its real-trait counterpart" footgun the Track B brief calls out: ADR-0020 §"`TestHal` fake" sketches a `FakeCpu: ContextSwitch` impl with a `FakeTaskContext { switched_to: usize }`, but `tyrne-test-hal` was never updated. Concrete consequence — every scheduler test reimplements a no-op switch by hand, and the fake exported as the canonical "Cpu fake for host tests" omits the operation that is *the* hot path of the kernel under test.
  - Suggested resolution: lift the `FakeCpu`/`FakeTaskContext` from `kernel/src/sched/mod.rs` into `tyrne-test-hal::cpu`, make the `FakeTaskContext` capture the `switched_to` index per ADR-0020, and have the kernel's sched tests import it. Tracked as a follow-up; do **not** ship test-HAL as the documented "all five fakes present" ([test-hal/src/lib.rs:18-20](../../../../test-hal/src/lib.rs#L18)) while the most important behavioural axis (context switch) lives elsewhere.

- [hal/src/context_switch.rs:25](../../../../hal/src/context_switch.rs#L25) `ContextSwitch` lacks a `Send + Sync` bound. `Cpu` is `Send + Sync` ([hal/src/cpu.rs:44](../../../../hal/src/cpu.rs#L44)), and every current consumer in the kernel writes `C: ContextSwitch + Cpu` so the Cpu side carries the auto-traits, but the trait alone does not encode the multi-core invariant the rest of HAL is consistent about. This is the kind of drift that bites when a future caller bounds on `<C: ContextSwitch>` alone (e.g. a bring-up test or a future per-core scheduler initialiser).
  - Suggested resolution: add `: Send + Sync` to the trait declaration, matching the discipline ADR-0020 §Decision-drivers calls "compiler-checked, so multi-core safety is not left to convention." Same convention `Cpu`, `Console`, `Timer`, `IrqController`, `Mmu` already carry.

- [hal/src/cpu.rs:22](../../../../hal/src/cpu.rs#L22) `IrqState(pub usize)` synthetic-construction concern, recheck: the Phase A review flagged this as non-blocking; it is still non-blocking, but with one new fact worth recording. Two test-only `Cpu` impls in the kernel now construct `IrqState(0)` directly ([kernel/src/sched/mod.rs:868](../../../../kernel/src/sched/mod.rs#L868) and [kernel/src/sched/mod.rs:1174](../../../../kernel/src/sched/mod.rs#L1174)) — i.e. the `pub` field is being exercised by safe code, not just by the BSP per the doc-comment's "BSP implementations can construct the value from raw architecture bits" intent. Either the doc-comment widens to acknowledge "BSP impls and test-only Cpu impls" or the field eventually narrows to `pub(crate)` with a `BspConstruct` helper as the Phase A review suggested. Either is fine; today's drift is documentation lagging usage by a notch.

- [hal/src/timer.rs](../../../../hal/src/timer.rs#L31) The Track B per-trait checklist names the `Timer` methods as `now_ticks` / `freq_hz` / `arm_deadline` / `cancel_deadline`, but the trait surface as landed in T-009/T-012 is `now_ns` / `arm_deadline` / `cancel_deadline` / `resolution_ns` (no `now_ticks`, no `freq_hz`). This is benign — ADR-0010 §Decision-outcome unambiguously chose nanoseconds at the trait boundary and pushed ticks into BSP-internal helpers (`ticks_to_ns` / `ns_to_ticks` / `resolution_ns_for_freq`) — but the plan's drafting had an outdated tick-shaped vocabulary. Flagging so the merge step does not chase a phantom regression. The actual landed trait matches ADR-0010 cleanly.

- [hal/src/irq_controller.rs:37](../../../../hal/src/irq_controller.rs#L37) Same drafting-vs-landed gap on `IrqController`: the Track B checklist names `enable_irq` / `disable_irq` / `claim` / `complete` / `set_priority`, but the trait surface per ADR-0011 §Decision-outcome is `enable` / `disable` / `acknowledge` / `end_of_interrupt`. There is **no** `set_priority` on the trait — ADR-0011 §Open-questions explicitly defers priority configuration to a future ADR, and §Negative consequences notes "all IRQs are delivered at the BSP's default priority." So the checklist's "verify `GIC_MAX_IRQ` range check lives in BSP impl, not the trait" question has the obvious answer — the only check that *could* live on the trait is for a method that does not exist. The actual `GIC_MAX_IRQ` check is at [bsp-qemu-virt/src/gic.rs:78](../../../../bsp-qemu-virt/src/gic.rs#L78) and is asserted inside `enable`/`disable` impls (lines 318, 344). That is the right place.

- [hal/src/cpu.rs:44](../../../../hal/src/cpu.rs#L44) The Track B checklist mentions an `enable_irqs` method on `Cpu`. That method does not exist; the trait is intentionally `disable_irqs` + `restore_irq_state` only, because the `IrqGuard` RAII pattern and the symmetric "save-prev-then-restore" idiom make a free-standing `enable_irqs` redundant (and dangerous — it would let callers stomp a saved mask). ADR-0008 settled this. Flagging as drafting drift in the plan, not a code finding.

### Observation

- [hal/src/cpu.rs:86-91](../../../../hal/src/cpu.rs#L86) The `IrqGuard<'a, C: Cpu>` post-mortem comment from Phase A is preserved verbatim ("Dynamic dispatch (`&dyn Cpu`) is also avoided because coercing a concrete type to a trait object at certain inlining depths can produce vtable references that alias unrelated data in `.rodata`; using a concrete type parameter eliminates the coercion site entirely"). Confirms the maintainer's request that this comment survive into the B1 era. It does.

- [hal/src/console.rs:34](../../../../hal/src/console.rs#L34) `Console` trait surface: one method (`write_bytes(&self, &[u8])`), `Send + Sync` bound at the trait level, contract spelled out (synchronous, infallible, no allocation, best-effort under contention). `FmtWriter<'a>(pub &'a dyn Console)` provides `core::fmt::Write` without forcing the HAL to depend on it. Matches ADR-0007 exactly. The `FakeConsole` ([test-hal/src/console.rs:58](../../../../test-hal/src/console.rs#L58)) preserves byte-level fidelity (Mutex-protected `Vec<u8>` capture, `captured_str()` UTF-8 decode for assertion ergonomics). No drift.

- [hal/src/context_switch.rs:25-69](../../../../hal/src/context_switch.rs#L25) The ADR-0020 split is real: `Cpu` is object-safe (no associated types), `ContextSwitch` carries the associated `TaskContext: Default + Send`. The two `unsafe fn` methods (`context_switch`, `init_context`) carry tight `# Safety` blocks: interrupts-disabled invariant, `current` validity, `next` initialisation invariant for switch; stack-top alignment + size + lifetime, `entry: fn() -> !` non-return invariant for init. An adversarial impl could of course do anything (the trait does not enforce save-restore symmetry), but every callsite in the kernel passes the impl through an `unsafe fn` already and the BSP's `Aarch64TaskContext` impl is the only production target. The contract is as tight as a non-`unsafe trait` can be without crossing into unfriendly-API territory. Note: the checklist phrases this as "the `unsafe` trait split"; the trait is not `unsafe trait`, only its methods are `unsafe fn`. That matches ADR-0020 §"`unsafe` placement" Option E.

- [hal/src/mmu.rs:208](../../../../hal/src/mmu.rs#L208) `Mmu` is a *real* trait with a complete v1 surface (associated `AddressSpace`, `create_address_space` `unsafe`, `address_space_root`, `activate`, `map`, `unmap`, `invalidate_tlb_address`, `invalidate_tlb_all`, `MmuError`, `FrameProvider`, `MappingFlags`, `PhysFrame::from_aligned`). Phase A review described it as "placeholder"; that was true at A6. The current surface is fully fleshed out per ADR-0009. **Crucial: no real BSP impl exists yet** — `grep "impl Mmu"` against the BSP and kernel source returns zero hits; only `tyrne_test_hal::FakeMmu` implements it. So the Track B brief's worry "if any A6/B-era code accidentally consumed it, that's a finding" — checked, no consumption. The trait is dormant, ready for B2 / ADR-0027 work, and the fake is functional enough to gate the kernel tests that B2 will add.

- [hal/src/timer.rs:31-55](../../../../hal/src/timer.rs#L31) `Timer` surface (`now_ns`, `arm_deadline`, `cancel_deadline`, `resolution_ns`) lines up with ADR-0010 §Decision-outcome and with the BSP impl at [bsp-qemu-virt/src/cpu.rs:484](../../../../bsp-qemu-virt/src/cpu.rs#L484) (`arm_deadline` → `ns_to_ticks` → `MSR CNTV_CVAL_EL0` + `MSR CNTV_CTL_EL0` + GIC `enable(TIMER_IRQ)`). The BSP correctly uses the host-testable `ns_to_ticks` from the HAL.

- [hal/src/timer.rs:198-213](../../../../hal/src/timer.rs#L198) `ns_to_ticks` is **ceiling-rounded** (`intermediate.div_ceil(nanos)`), matching ADR-0010's contract that the IRQ "fires at-or-after the requested time." The PR #10 review-round-2 sub-tick test the brief asks about lives at [hal/src/timer.rs:422-436](../../../../hal/src/timer.rs#L422) (`ns_to_ticks_rounds_up_on_subtick`) — confirmed present and exercises the discriminating `freq=3 Hz, ns=333_333_334` case where floor would give 1 tick but ceiling correctly gives 2. Saturating cast at `u128 → u64` is also tested ([hal/src/timer.rs:461-477](../../../../hal/src/timer.rs#L461)). Solid coverage; no regression risk.

- [hal/src/irq_controller.rs:37-59](../../../../hal/src/irq_controller.rs#L37) `IrqController` four-method surface (`enable`/`disable`/`acknowledge`/`end_of_interrupt`), `Option<IrqNumber>` for spurious/race semantics on `acknowledge`, `IrqNumber(pub u32)` newtype. Matches ADR-0011. The `FakeIrqController` ([test-hal/src/irq_controller.rs:93](../../../../test-hal/src/irq_controller.rs#L93)) captures every observable (enabled set, FIFO pending queue, EOI history) — fidelity is genuine, not degenerate.

- [test-hal/Cargo.toml:13-14](../../../../test-hal/Cargo.toml#L13) `tyrne-test-hal` declares `tyrne-hal` as a regular dep (no dev-dep gating needed because the crate's *entire purpose* is host-test fakes), and the kernel pulls it in only as `[dev-dependencies]` ([kernel/Cargo.toml:16-17](../../../../kernel/Cargo.toml#L16)). The kernel's library code at `lib.rs` only depends on `tyrne-hal`. Production builds (the BSP) depend on `tyrne-kernel` and `tyrne-hal` but never on `tyrne-test-hal` — confirmed by `grep -r "tyrne_test_hal\|tyrne-test-hal" bsp-qemu-virt`: no matches. The `[dev-dependencies]`-only linkage holds.

- [test-hal/src/cpu.rs:99-108](../../../../test-hal/src/cpu.rs#L99) `FakeCpu::disable_irqs` returns `IrqState(usize::from(state.irqs_enabled))` (so `1` if previously enabled, `0` if previously disabled). `restore_irq_state` flips to `state.0 != 0`. This is faithful to the Phase A "save and restore" semantics — the nested-guard test at [test-hal/src/cpu.rs:158-172](../../../../test-hal/src/cpu.rs#L158) verifies inner-guard drop leaves the outer-section mask in place. Fidelity is good for the EL1/host-mock contract. (The fake is degenerate w.r.t. *what specific bits* go into the IrqState — which is correct, because the real BSP's DAIF bit pattern is meaningless on the host, and the fake's job is the save/restore *protocol*, not the bits.)

- [test-hal/src/timer.rs:90-107](../../../../test-hal/src/timer.rs#L90) `FakeTimer` deliberately does **not** simulate deadline expiry — the doc-comment at [test-hal/src/timer.rs:9-13](../../../../test-hal/src/timer.rs#L9) calls this out. Tests assert on `armed_deadline()` and `cancel_count()` directly. Fidelity matches the real impl's *observable* surface (the manual clock + arm/cancel/replace semantics) without taking on the impossible job of simulating a hardware IRQ. Reasonable.

- [test-hal/src/mmu.rs:128-174](../../../../test-hal/src/mmu.rs#L128) `FakeMmu` stores mappings as a `HashMap` keyed by VirtAddr, returns `AlreadyMapped` on duplicate insert, `NotMapped` on missing unmap, and exposes `tlb_address_invalidations` / `tlb_all_count` for assertions. Tests at [test-hal/src/mmu.rs:177-318](../../../../test-hal/src/mmu.rs#L177) cover map/unmap round-trip, alignment rejection in `PhysFrame::from_aligned`, double-map rejection, and TLB-invalidation ordering. Fidelity is appropriate for a fake that does not pretend to model `VMSAv8`. One thing the fake does *not* exercise — but no real consumer exists to need it — is `OutOfFrames`, since the fake never calls into `frames` (no intermediate-table walk). Acceptable for v1.

- [hal/src/lib.rs:33-49](../../../../hal/src/lib.rs#L33) Crate posture: `#![no_std]`, modules cleanly partitioned, public re-exports flat (`Console`, `FmtWriter`, `ContextSwitch`, `CoreId`, `Cpu`, `IrqGuard`, `IrqState`, `IrqController`, `IrqNumber`, `MappingFlags`, `Mmu`, `MmuError`, `PhysAddr`, `PhysFrame`, `VirtAddr`, `PAGE_SIZE`, `FrameProvider`, `Timer`). The `cpu` and `timer` modules are `pub mod` (so `tyrne_hal::timer::ns_to_ticks` and `tyrne_hal::cpu::current_el` resolve); the others are `mod` with selective re-exports. Sensible.

- [hal/src/cpu.rs:164](../../../../hal/src/cpu.rs#L164) `current_el()` is gated `#[cfg(all(target_arch = "aarch64", target_os = "none"))]` so it is *absent* from host builds — host tests cannot accidentally trap with SIGILL. The doc-comment, the `# Safety` block, and the `SAFETY:` comment are all in good order; this matches UNSAFE-2026-0018 in the audit log.

- [hal/Cargo.toml](../../../../hal/Cargo.toml) Workspace-derived metadata, no dependencies. The crate is genuinely traits-only (no logic), per the `lib.rs` doc-comment's promise. Clean.

## Sub-verdict

**Comment**

Three-line summary: the HAL trait surface is consistent with its ADRs, the test-HAL is `[dev-dependencies]`-isolated and the four exercised fakes are appropriately faithful, but `FakeCpu` is missing the `ContextSwitch` impl ADR-0020 sketches and has been silently shadowed by a local re-implementation inside `kernel/src/sched/mod.rs`. Closing that gap (lifting `FakeTaskContext` into `tyrne-test-hal`) is the highest-value follow-up; tightening `ContextSwitch` with `: Send + Sync` is a small consistency win matching the rest of HAL. No blockers, no security-shaped findings; trait↔BSP wiring is clean.
