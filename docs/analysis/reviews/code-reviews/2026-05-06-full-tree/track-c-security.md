## 1. Capability correctness

Adversarial frame: *can a caller invoke `cap_copy` / `cap_derive` / `cap_revoke` / `cap_drop` / `cap_take` / `ipc_send` / `ipc_recv` / `ipc_notify` / `ipc_send_and_yield` / `ipc_recv_and_yield` without holding the authority `security-model.md` requires?*

- **Phase A + B0 + B1 invariants intact.** **OK** — every "OK" finding in the three prior consolidated reviews re-verified against the post-`214052d` source. No regressions in `cap_copy` ([`cap/table.rs:185-187`](../../../../../kernel/src/cap/table.rs#L185-L187)), `cap_derive` ([`cap/table.rs:260-262`](../../../../../kernel/src/cap/table.rs#L260-L262)), `cap_revoke` ([`cap/table.rs:322-324`](../../../../../kernel/src/cap/table.rs#L322-L324)), `cap_drop`, `cap_take`, `ipc_send` rights enforcement ([`ipc/mod.rs:266`](../../../../../kernel/src/ipc/mod.rs#L266)), or `ipc_recv` rights enforcement ([`ipc/mod.rs:343`](../../../../../kernel/src/ipc/mod.rs#L343)).
- **Capability is move-only at the type level.** **OK** — `Capability` derives `Debug` only; not `Copy`, not `Clone` ([`cap/mod.rs:100-104`](../../../../../kernel/src/cap/mod.rs#L100-L104)). Duplication is structurally impossible without going through `cap_copy`, which requires `DUPLICATE`.
- **Reserved-bit masking at the ABI boundary.** **OK** — `CapRights::from_raw` silently masks bits outside `KNOWN_BITS` ([`cap/rights.rs:67-70`](../../../../../kernel/src/cap/rights.rs#L67-L70)). A hostile untrusted caller cannot smuggle reserved flags past `contains` checks. Test pinned at [`cap/rights.rs:179-188`](../../../../../kernel/src/cap/rights.rs#L179-L188).
- **Generation-tagged handles defeat slot-reuse confusion.** **OK** — `free_slot` bumps `generation` via `wrapping_add(1)` ([`cap/table.rs:548`](../../../../../kernel/src/cap/table.rs#L548)); stale handle fails `resolve_handle` ([`cap/table.rs:506-518`](../../../../../kernel/src/cap/table.rs#L506-L518)). **Inherited flag (latent):** `u32` generation wraps after 2³² reuse cycles per slot — Phase-A note carried forward unchanged; not exploitable in v1 (reuse counts are single digits). Tracked as a B-late ADR follow-up.
- **Cross-table revocation gap.** **flagged — inherited, unchanged.** `install_cap_if_some` calls `insert_root` on the receiver's table ([`ipc/mod.rs:370`](../../../../../kernel/src/ipc/mod.rs#L370)), so an IPC-transferred capability lands as a fresh root in the receiver's tree — sender's `cap_revoke` does not reach it. Tracked as ADR-0023 (deferred per B0 closure plan); `security-model.md`'s *Invariants* now carries the v1 scope qualification (see §8 below). No new code in B0/B1 makes the gap worse or better.
- **Bridge resolves cap rights *before* materialising any momentary `&mut`.** **OK** — `resolve_ep_cap` runs against the caller's `&CapabilityTable` ([`sched/mod.rs:273-284`](../../../../../kernel/src/sched/mod.rs#L273-L284)) before the inner block in `ipc_send_and_yield` / `ipc_recv_and_yield` allocates any `&mut`. Failure path returns `SchedError::Ipc(InvalidCapability)` with the scheduler untouched.
- **Capability transfer atomicity through the bridge.** **OK** — `ipc_send_and_yield`'s Phase 1 block reuses the bare `ipc_send` path ([`sched/mod.rs:651`](../../../../../kernel/src/sched/mod.rs#L651)); the cap moves out of the sender's table inside the same momentary-`&mut` block as the endpoint-state transition. The bridge adds yield/unblock around the existing flow but does not introduce a new cap-handling code path.
- **`ReceiverTableFull` pre-flight retains the in-flight cap.** **OK** — pre-flight check at [`ipc/mod.rs:356-363`](../../../../../kernel/src/ipc/mod.rs#L356-L363) tests `pending_has_cap && caller_table.is_full()` *before* `core::mem::replace` moves state to `Idle`. A failed install would otherwise drop the cap silently. Regression test `recv_with_full_table_preserves_pending_cap` exists in the IPC test module.
- **Slot-reuse generation guard with `Some(cap)` leak detection.** **OK** — `IpcQueues::reset_if_stale_generation` at [`ipc/mod.rs:211-228`](../../../../../kernel/src/ipc/mod.rs#L211-L228) fires a `debug_assert!` if a stale-generation reset would silently drop an in-flight `Capability`. The destruction-path drain-before-reset invariant is the future-proof bound the assertion enforces; release-mode behaviour acceptably leaks (debug_assert is a no-op) but the bug would surface as a generation-bump test failure in production.
- **`SchedError::Deadlock` rollback scope.** **OK** — `ipc_recv_and_yield` restores `s.current` and `s.task_states[current_idx]` at [`sched/mod.rs:773-778`](../../../../../kernel/src/sched/mod.rs#L773-L778) before returning Err. Endpoint state stays at `RecvWaiting` (documented in `SchedError::Deadlock`'s doc-comment, [`sched/mod.rs:160-180`](../../../../../kernel/src/sched/mod.rs#L160-L180)); a hostile retry sees `QueueFull`, not `Pending`. No capability leak; no escalation.
- **`PendingAfterResume` typed return.** **OK** — `ipc_recv_and_yield`'s Phase 3 decodes a resume-time `Pending` into `Err(SchedError::Ipc(IpcError::PendingAfterResume))` at [`sched/mod.rs:828`](../../../../../kernel/src/sched/mod.rs#L828) instead of returning `Ok(Pending)` (which a downstream `let RecvOutcome::Received { .. } else { panic!() }` would convert into a panic).
- **GIC `enable` / `disable` / `acknowledge` / `end_of_interrupt` are not capability-gated in v1.** **N/A — deliberate.** These are kernel-internal hardware operations; they correspond to MMU page-table ops in spirit, not to capability-table ops. The `IrqController` trait sits below the capability layer (mirroring `Console::write_bytes`'s lack of `ConsoleCap` check). Future userspace IRQ delivery is gated by the `IrqCap` pattern in [`exceptions.md`](../../../../architecture/exceptions.md)'s "No userspace IRQ delivery in v1" §; that lands with Phase C work. **Adversarial probe — can a hostile caller reach `IrqController::enable`?** No; the only callers are `arm_deadline` / `cancel_deadline` (kernel-internal) and `irq_entry` itself, none of which is reachable from userspace in v1.
- **`Timer::arm_deadline` / `cancel_deadline` are not capability-gated in v1.** **N/A — deliberate.** Same reasoning. There is *no caller* in v1 — the cooperative IPC demo arms no deadline; idle uses `wait_for_interrupt` which goes via `Cpu::wait_for_interrupt`, not `Timer`. The first userspace-reachable arm path will be Phase B5+ `time_sleep_until` syscall, gated by `TimerCap`.
- **GIC range assertions on `enable` / `disable`.** **OK** — `assert!(irq.0 < GIC_MAX_IRQ)` at [`gic.rs:317-322`](../../../../../bsp-qemu-virt/src/gic.rs#L317-L322) and [`:343-348`](../../../../../bsp-qemu-virt/src/gic.rs#L343-L348) gates offset arithmetic against ARM IHI 0048B's 1020-INTID architectural cap. The assertion is *before* any MMIO; defence-in-depth against future regressions (only caller passes `IrqNumber(27)` today).
- **`acknowledge` folds spurious INTID 1023 to `None`.** **OK** — [`gic.rs:369-376`](../../../../../bsp-qemu-virt/src/gic.rs#L369-L376) masks the bottom 10 bits of `GICC_IAR` and folds `1023` to `None` per the trait contract. The spurious value cannot leak as a real `IrqNumber` to a downstream caller.
- **`end_of_interrupt` writes the raw IRQ ID.** **OK** — [`gic.rs:378-384`](../../../../../bsp-qemu-virt/src/gic.rs#L378-L384) writes `irq.0` to `GICC_EOIR` without further validation; GICv2 EOIR ignores writes that don't match a currently-active priority level. No state corruption from a mis-paired call.

## 2. Trust boundaries

Adversarial frame: *what new boundaries did B0/B1 add, and what untrusted bytes (or values) cross them?*

- **No userspace/kernel privilege boundary in code yet.** **N/A — by design.** Phase B5+ work. `kernel_entry` and task entry functions are still called from within the same privilege level (EL1) and the same Rust module tree. No `svc`/`smc`/`hvc`/`ecall` instruction appears anywhere in the tree (verified by grep across `bsp-qemu-virt/src`). No syscall dispatch table, no enter-user routine, no EL0-bound code.
- **No premature userspace surface introduced by B0/B1.** **OK** — vectors.s explicitly defends against IRQs taken from `SP_EL0` mode or any lower EL ([`vectors.s:174-186`](../../../../../bsp-qemu-virt/src/vectors.s#L174-L186)) by routing them to `panic_entry` with class id 1. The architectural surface is reserved for Phase C; no B0/B1 code reaches it.
- **EL2 → EL1 boot transition (T-013, UNSAFE-2026-0017).** **OK** — `boot.s`'s EL drop sequence configures `HCR_EL2 = 0x80000000` (RW=1; everything else literally zero — explicitly non-VHE), `SPSR_EL2 = 0x3c5` (mode=EL1h + DAIF masked), `ELR_EL2 ← post_eret`. The post-`eret` PSTATE inherits the masked DAIF from `SPSR_EL2`. Adversarial probe: a firmware that delivered at EL2 with `HCR_EL2.IMO = 1` (route IRQs to EL2) — the literal write at [`boot.s:76-77`](../../../../../bsp-qemu-virt/src/boot.s#L76-L77) clears `IMO` so the kernel runs with EL1-bound IRQ delivery. Verified safe.
- **EL3 entry path.** **OK** — `boot.s` halts in a named-label `wfe`-loop on EL3 entry ([`boot.s:60-62`](../../../../../bsp-qemu-virt/src/boot.s#L60-L62)). No Rust runtime is up; no panic frame attempted. Future BSPs needing the EL3→EL2→EL1 chain are tracked in ADR-0024 §Open questions.
- **`CurrentEL` boot-time post-condition assertion.** **OK** — `QemuVirtCpu::new` calls `tyrne_hal::cpu::current_el()` and asserts `== 1` at [`bsp-qemu-virt/src/cpu.rs:135-139`](../../../../../bsp-qemu-virt/src/cpu.rs#L135-L139) *before* any timer-related MRS. A regression in `boot.s`'s EL drop becomes a loud, named boot panic with the observed EL value — useful for triage, not an information leak. The assertion is now load-bearing (post-T-013 Amendment to UNSAFE-2026-0016).
- **`tyrne_hal::cpu::current_el()` cfg-gating.** **OK** — function is `cfg(all(target_arch = "aarch64", target_os = "none"))`-gated at [`hal/src/cpu.rs:164`](../../../../../hal/src/cpu.rs#L164). On hosted targets (including `aarch64-apple-darwin` test runs) the function is structurally absent — there is no fallback EL0 read; the `MRS` would `SIGILL`. The Amendment to UNSAFE-2026-0018 corrects the original "or yield `EL0`" prose; current source aligns with the corrected reasoning.
- **GIC MMIO surface (UNSAFE-2026-0019).** **OK** — every read/write goes through `core::ptr::read_volatile` / `write_volatile` against typed offsets within the distributor (`0x0800_0000`) and CPU interface (`0x0801_0000`) windows. Adversarial offset analysis in the [B1 closure review §2](../../security-reviews/2026-04-28-B1-closure.md#2-trust-boundaries) re-verified at HEAD: `GICD_ITARGETSR + byte_idx` for `byte_idx < 1024` produces a maximum offset of `0xC00` — well within a 4 KiB distributor window; `ISENABLER` / `ICENABLER` indexed by `n / 32 < 32` produces offsets in `[0x100, 0x180)` and `[0x180, 0x200)` respectively. CPU interface offsets are fixed (0x000 / 0x004 / 0x00C / 0x010). No window escape possible.
- **`gic.init` runs with `DAIF.I` masked.** **OK** — `kernel_entry` sequences "MSR `VBAR_EL1` → GIC construct → GIC init" before "MSR `DAIFClr, #0x2`" at [`bsp-qemu-virt/src/main.rs:548-598`](../../../../../bsp-qemu-virt/src/main.rs#L548-L598). A fault during `init` (e.g. wrong base) would land at `tyrne_vectors+0x200` (sync, current EL with SP_ELx) which trampolines to `panic_entry(class=0, ESR_EL1)` — loud failure.
- **EL1 vector table install via `MSR VBAR_EL1` (UNSAFE-2026-0020).** **OK** — address read via `core::ptr::addr_of!(tyrne_vectors) as u64` at [`main.rs:555`](../../../../../bsp-qemu-virt/src/main.rs#L555) (no deref of the zero-byte symbol), written to `VBAR_EL1` with `ISB`. Two-layer 2-KiB alignment: `.balign 2048` in [`vectors.s:39`](../../../../../bsp-qemu-virt/src/vectors.s#L39) plus the linker's `. = ALIGN(2048)` directive. Either alone is sufficient; both is defence-in-depth.
- **IRQ trampoline frame layout invariant.** **OK** — `const _: () = assert!(core::mem::size_of::<TrapFrame>() == 192);` at [`exceptions.rs:77`](../../../../../bsp-qemu-virt/src/exceptions.rs#L77) is the strongest possible safety net for the asm ↔ `repr(C)` ABI invariant. A future field reorder or padding change fails the build before it ships. `repr(C)` for `[u64; 2]`-typed fields is layout-stable.
- **`tyrne_vectors_end` exported but unused by linker.ld.** **N/A — defensive.** Recorded in vectors.s as a future-tools convenience for audit-region bound checks. No security impact.
- **Vector entry padding fills with `0x00` = `udf` on aarch64.** **OK** — `.balign 0x80` after each `b <label>` pads with the assembler's default `0x00` byte, which decodes as `udf #0x0`. A CPU that lands inside the padding faults loudly to the same trampolines. Self-stabilising.
- **No kernel deref of raw userspace pointers.** **OK** — verified by grep; zero `*const u8` / `*mut u8` dereferences in `kernel/src/`. The `stack_top: *mut u8` argument forwarded to `init_context` is written into a `u64` field, not dereferenced. Grow this surface once syscalls land.
- **`Message` validation.** **OK** — `Message { label: u64, params: [u64; 3] }` is plain pass-by-value data ([`ipc/mod.rs:62-68`](../../../../../kernel/src/ipc/mod.rs#L62-L68)). The kernel does not interpret message contents; there is no parser, hence no parser-state bug surface.
- **`CapHandle` opaqueness.** **OK** — `index()` and `generation()` accessors are `#[doc(hidden)]` and only used by tests ([`cap/table.rs:46-60`](../../../../../kernel/src/cap/table.rs#L46-L60)). Out-of-range indices returned by a fabricated handle hit `.get(idx)` not `[idx]` — `None`-returning, not panic.

## 4. Kernel-mode discipline

Adversarial frame: *can a call wedge the kernel, force a kernel panic, or starve another task's work?*

- **No allocation in ISR.** **OK** — `irq_entry`'s body ([`exceptions.rs:144-228`](../../../../../bsp-qemu-virt/src/exceptions.rs#L144-L228)) is straight-line: read GICC_IAR (1 volatile read), branch on spurious, branch on IRQ ID, MSR `CNTV_CTL_EL0`, GICC_EOIR write, return. ~10 instructions of dispatch logic plus the 31 instructions of trampoline save/restore. No heap, no scheduler touch, no momentary `&mut Scheduler<C>` (per ADR-0021 2026-04-28 Amendment to UNSAFE-2026-0014, v1 vacuously satisfies the discipline).
- **No new heap allocation anywhere.** **OK** — `grep -rE 'alloc::|Box::new|Vec::new' kernel/ hal/ bsp-qemu-virt/` returns zero hits. Kernel crate enforces this with `#![deny(clippy::alloc_instead_of_core)]` at the workspace level ([`Cargo.toml`](../../../../../Cargo.toml)).
- **Bounded kernel state.** **OK** — `CAP_TABLE_CAPACITY = 64`, `TASK_ARENA_CAPACITY = 16`, `ENDPOINT_ARENA_CAPACITY = 16`, `NOTIFICATION_ARENA_CAPACITY = 16`, `MAX_DERIVATION_DEPTH = 16`, `SchedQueue` capacity pinned to `TASK_ARENA_CAPACITY`. Every over-capacity path returns a typed error.
- **Allocation paths return typed errors, not panics — kernel scope.** **OK in cap/ipc/obj/sched bridge.** Three production-kernel `panic!`s remain in `kernel/src/sched/mod.rs`:
  1. [`sched/mod.rs:307`](../../../../../kernel/src/sched/mod.rs#L307) — `unblock_receiver_on` "ready queue full on unblock". `#[allow(clippy::panic, reason = ...)]` justifies why it's an unreachable invariant.
  2. [`sched/mod.rs:440`](../../../../../kernel/src/sched/mod.rs#L440) — `start_prelude` "scheduler start called with empty ready queue". Annotated as a kernel programming error.
  3. [`sched/mod.rs:559`](../../../../../kernel/src/sched/mod.rs#L559) — `yield_now` "ready queue full on yield re-enqueue". Annotated; unreachable invariant.
  
  The Phase A review's flagged "deadlock panic in `ipc_recv_and_yield`" is **resolved at HEAD** — replaced by `Err(SchedError::Deadlock)` per ADR-0022, with rollback discipline at [`sched/mod.rs:773-778`](../../../../../kernel/src/sched/mod.rs#L773-L778). Verified.
- **Allocation paths return typed errors — IPC.** **OK** — `IpcError` is `#[non_exhaustive]` with seven variants ([`ipc/mod.rs:71-97`](../../../../../kernel/src/ipc/mod.rs#L71-L97)); every over-capacity / invalid-handle path is an `Err`. The single `unreachable!()` at [`ipc/mod.rs:314`](../../../../../kernel/src/ipc/mod.rs#L314) is preceded by an explicit `peek_state` pre-check that excludes the Send/Recv-pending arms; correctness depends on that pre-check, which is in the same function and audit-able.
- **Allocation paths return typed errors — capabilities.** **OK** — `CapError` is `#[non_exhaustive]` with six variants ([`cap/mod.rs:138-161`](../../../../../kernel/src/cap/mod.rs#L138-L161)). `cap_revoke`'s descendant array is bounded by `CAP_TABLE_CAPACITY` with explicit `desc_len < CAPACITY` guards at [`cap/table.rs:346-352, 370-376`](../../../../../kernel/src/cap/table.rs#L346-L376) — release-mode skips overflowing children rather than panicking; debug-mode trips the `debug_assert!`. Defensive.
- **Unbounded loops.** **OK** — `cap_revoke`'s BFS bounded by `CAP_TABLE_CAPACITY`. `unblock_receiver_on` iterates `0..TASK_ARENA_CAPACITY` (16). `SchedQueue` ops are O(1). `init_context` is straight-line. `gic.init`'s loops are bounded by `irq_count` derived from `GICD_TYPER` (capped at 1024 per architecture). The PL011 `write_bytes` `UARTFR_TXFF` spin remains unbounded — Phase-A note carried forward unchanged; acceptable for QEMU console, low priority for non-QEMU BSPs.
- **Critical sections minimised.** **OK** — `IrqGuard` is held only across `cpu.context_switch` in `start`, `yield_now`, `ipc_recv_and_yield`. The rest of each function runs at caller-state DAIF.
- **No new kernel panic on a *userspace-reachable* path.** **OK** — `irq_entry`'s panic at [`exceptions.rs:227`](../../../../../bsp-qemu-virt/src/exceptions.rs#L227) is the structurally-unreachable "unhandled IRQ" guard (only `IrqNumber(27)` is enabled at the GIC). The GIC range-assertions are kernel-internal contract guards. None are reachable from a userspace caller in v1 because there is no userspace.
- **Idle's WFI activation.** **OK** — [`bsp-qemu-virt/src/main.rs:255-286`](../../../../../bsp-qemu-virt/src/main.rs#L255-L286) — `cpu.wait_for_interrupt() + yield_now`. In v1 the cooperative IPC demo's two application tasks are permanently `Ready`, so the FIFO never reaches `idle_entry` — WFI is structurally unreachable in the demo path. Documented at [`main.rs:268-274`](../../../../../bsp-qemu-virt/src/main.rs#L268-L274).
- **GIC + DAIF unmask ordering.** **OK** — `kernel_entry` sequences "`VBAR_EL1` install → GIC construct + init → `DAIFClr, #0x2`" at [`main.rs:548-598`](../../../../../bsp-qemu-virt/src/main.rs#L548-L598). Reordering to "GIC init before VBAR install" would silently hang on a fault; reordering to "DAIF unmask before GIC init" is benign because GIC's CPU interface is disabled. The current order is safe-by-construction.
- **Scheduler does not touch the new IRQ surface.** **OK** — `kernel/src/sched/mod.rs` is unchanged by T-012; the cooperative-only model is preserved. Future preemption arc adds `sched::on_timer_irq` per the ADR-0021 2026-04-28 Amendment.
- **DAIF.I-only unmask.** **OK** — `MSR DAIFClr, #0x2` clears bit 1 (`I`) only; bits 0 (`F`), 2 (`A`), 3 (`D`) stay masked. v1 has no FIQ/SError-recovery/debugger surface.
- **Arithmetic-side-effect lint coverage.** **OK** — `#![deny(clippy::arithmetic_side_effects)]` in `kernel/src/lib.rs`; explicit `saturating_add` / `wrapping_add` with `#[allow]` reason annotations where used (e.g. [`sched/mod.rs:83-90, 100-108`](../../../../../kernel/src/sched/mod.rs#L83-L108) `SchedQueue` head/len wrap; [`cap/table.rs:266-277`](../../../../../kernel/src/cap/table.rs#L266-L277) derivation depth saturation; [`gic.rs:166-169, 174-186, 197-210, 219-230`](../../../../../bsp-qemu-virt/src/gic.rs#L166-L230) GIC offset math).
- **Float arithmetic ban.** **OK** — `#![deny(clippy::float_arithmetic)]` in the kernel lib. The d8–d15 register saves are integer `STP d8, d9` instructions; no FP *operations*.
- **BSP task-body `.expect`/`panic!`.** **flagged — inherited, unchanged.** `task_a`/`task_b` still call `.expect("task X: ipc_send failed")` / `panic!("task X: expected Received outcome ...")` at multiple sites in [`bsp-qemu-virt/src/main.rs`](../../../../../bsp-qemu-virt/src/main.rs). Same severity as the Phase A review: not a v1 blocker because there is no userspace; **pre-requisite for Phase B's first userspace-driver task**, where supervisor-endpoint design + `error-handling.md §8` fault containment must be wired up.

## 8. Threat-model impact

Adversarial frame: *does any code path at HEAD change what the system defends against vs. the statements in `security-model.md`?*

- **Capability-based microkernel shape.** **OK** — v1 implements the capability table, rights, derivation tree, revocation, transfer, and the `SEND`/`RECV`/`NOTIFY`/`TRANSFER`/`DUPLICATE`/`DERIVE`/`REVOKE` initial right set. `MemoryRegion` deferred to Phase B as `security-model.md` §"Capability types" promises; [`cap/mod.rs:58-60`](../../../../../kernel/src/cap/mod.rs#L58-L60) carries the explicit deferred marker. No divergence.
- **"No privileged operation without authorising capability".** **OK** — every IPC and capability operation rejects on missing rights before any side effect (verified in §1).
- **"Capabilities move with consent".** **OK** — `cap_take` consumes the sender's handle; IPC transfers materialise in the receiver only after the move succeeds.
- **"Capability rights narrow, never broaden, on derivation".** **OK** — enforced structurally by `rights.contains(new_rights)` checks in `cap_copy` / `cap_derive`.
- **"Revocation is transitive over a derivation subtree".** **flagged — inherited single-table qualification.** `security-model.md §Invariants` now carries the v1 scope qualification ([`security-model.md:289-294`](../../../../architecture/security-model.md#L289-L294)) — IPC transfer creates a new root in the receiver's table; cross-table CDT is open per ADR-0023. No B0/B1 code worsens or improves this.
- **"A sender cannot transfer authority it does not hold".** **OK** — `cap_take` validates handle + rights; `TRANSFER` right required; verified in §1.
- **"The kernel never dereferences raw userspace pointers".** **OK today** — no userspace pointers exist. Becomes live at Phase B5+ syscall introduction.
- **"Drivers are userspace tasks".** **OK as design intent; v1 deviates by necessity** — PL011 driver in BSP runs at kernel privilege because there is no userspace. The B0 / B1 reviews flagged this as a `not a v1 divergence; a note` carry-forward; same disposition at HEAD.
- **"`unsafe` is audited".** **OK** — 21 entries (20 Active; 0012 Removed; 0019/0020/0021 Active with Pending QEMU smoke). Every `unsafe` block in the tree has a `// SAFETY:` comment citing an audit tag. Workspace-wide `#![deny(clippy::undocumented_unsafe_blocks)]` enforces it. **§3 audit-table verified row by row at HEAD; no drift.**
- **"No proprietary binary blobs".** **OK** — build has no binary blobs.
- **"Fault containment does not leak authority".** **N/A** — no userspace, no fault containment surface yet. Phase B5+ work.
- **"Bounded kernel state".** **OK** — covered in §4.
- **"DMA is capability-scoped where hardware permits".** **N/A — v1 enables no DMA-capable peripherals.** PL011 is PIO. Re-examine at first DMA-capable driver.
- **Threat-model load-bearing shifts (B0/B1 inherited):**
  - *EL1 as structural property post-T-013.* `boot.s`'s EL drop sequence promotes "we run at EL1" from a hoped-for emulator default to an actively-driven invariant. UNSAFE-2026-0016 is now the post-condition assertion.
  - *Aliasing model post-T-006.* The raw-pointer scheduler bridge (UNSAFE-2026-0013/0014) replaces `&mut Scheduler<C>` with `*mut Scheduler<C>` plus a momentary-`&mut` discipline that Stacked Borrows (Miri) verifies on every host-test run.
  - *IRQ delivery as structural property post-T-012.* The kernel actively drives `DAIF.I` unmasked after `VBAR_EL1` install + GIC init; the misbehaving-emulator stray-IRQ silent-hang surface is closed.
