It does **not** run `cargo audit` or `cargo vet check` — both listed as required gates in [`infrastructure.md` lines 67-74](../../../standards/infrastructure.md). This is consistent with the workspace having zero external dependencies today (so neither tool would find anything), but the standard treats these as "block merge" gates regardless. Either (a) wire the gates in (no-op today, future-safe) or (b) update [infrastructure.md](../../../standards/infrastructure.md) to mark them as conditional-on-first-extern-dep.
  - Suggested resolution: add a `supply-chain` job that runs `cargo install cargo-audit && cargo audit` and notes "no advisories possible until first extern dep" in its output; or amend `infrastructure.md` §Continuous integration to make these gates conditional on `Cargo.lock` containing at least one external entry.

- [[.github/workflows/ci.yml](../../../../.github/workflows/ci.yml):44-48, 80-85] **`lint-and-host-test` and `kernel-build` jobs install stable Rust, then implicitly defer to `rust-toolchain.toml` for the actual nightly.** The `rustup default stable` + `rustup component add rustfmt clippy` lines look like the jobs run on stable, but [rust-toolchain.toml](../../../../rust-toolchain.toml) pins `nightly-2026-01-15` workspace-wide and rustup honors that pin in the workspace. Net effect: stable is installed (and ignored), nightly is downloaded by rustup-on-demand inside the workspace, and the components added to *stable* are not the components that actually get used. This works, but it is misleading and pollutes the cache key. Either drop the stable install + component-add lines (the toolchain file pulls everything), or `rustup override set` the nightly explicitly so the cache reflects what is actually executed.
  - Suggested resolution: replace the bespoke `rustup` block with `dtolnay/rust-toolchain@stable` or `actions-rust-lang/setup-rust-toolchain@v1` honoring `rust-toolchain.toml`, and drop the orphan `rustup component add rustfmt clippy` against `default stable`.
