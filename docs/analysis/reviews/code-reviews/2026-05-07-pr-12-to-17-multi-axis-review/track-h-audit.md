### Minor

- **MIN-1 (PR #14, commit `9b77f3f`).** UNSAFE-2026-0016's body fields were edited in-place to swap `Yüksek` → `High` in two locations (§Introduced and §Rejected alternatives → "Skip the check, document only"). This is technically an in-place edit of an entry whose introducing-commit boundary is `f289d4d` (T-013, 2026-04-27); per `unsafe-policy.md §3` and UNSAFE-2026-0017's "Discipline note for future readers" paragraph, "additions to body fields after the introducing commit belong in Amendment blocks even within the same PR" — and the same logic applies to *modifications*. The localisation sweep treated this as a repo-wide find-and-replace (similar to the rest of PR #14's `Yüksek` → `High` and `cemililik/TyrneOS` → `cemililik/Tyrne` work) which produced a defensible result *semantically* (Yüksek and High denote the same severity in Tyrne's review vocabulary), but the discipline expects an Amendment block. **Severity rationale: minor — no semantic change, no information loss; the rider would be a one-line "Yüksek → High localization sweep" Amendment that a future reader could verify against `git blame`. Suggested closure: append a small Amendment to UNSAFE-2026-0016 documenting the localization edit; or accept that PR #14 was a pure-mechanical localization sweep and codify in the unsafe-policy that mechanical localization is exempt (i.e. amend `unsafe-policy.md §3` itself rather than the entry).** Either is acceptable; the latter is the cleaner standard-side fix and would unblock future repo-wide localization sweeps without adding ceremony to every entry they touch.

- **MIN-2 (PR #12 + PR #17).** UNSAFE-2026-0014's 3rd Amendment (PR #12) and 4th Amendment (PR #17) both reference the introducing commit by **PR number** ("commit on PR #12" / "commit on PR #17") rather than by SHA. Per UNSAFE-2026-0017's "Discipline note for future readers" — codified in PR #9 review-round 2 and intended to be the project's standing rule — every Amendment should carry an `(YYYY-MM-DD, commit SHA)` tuple in its header so future readers can reconstruct the entry's state at any past commit (the very property the policy lists as the *raison d'être* of Amendment append-only). The actual introducing commits are `c30f4ee` (T-014 feat) for the 3rd Amendment and `7a402cb` (T-015 feat) for the 4th. The Amendment headers themselves do correctly carry `(2026-05-06, T-014 / ADR-0026)` and `(2026-05-07, T-015 / ADR-0032)` — but the SHA is intentionally absent. There is project precedent for back-filling the SHA after merge (commit `6d2f725 docs(audits): back-fill T-012 commit-3 SHA — 28c5ce9`; commit `80e2be3 docs(audits): back-fill UNSAFE-2026-0006 / 0015 Amendment commit SHAs`) which suggests the absence is a known pattern, not an oversight, and the back-fill is the closure path. **Severity rationale: minor — the Amendment is recoverable from `git log -p docs/audits/unsafe-log.md` without the SHA, and the project precedent for post-merge back-fill is established. Suggested closure: a small `docs(audits): back-fill T-014 + T-015 Amendment commit SHAs` follow-up on the same shape as `6d2f725` / `80e2be3`.**

- **NIT-2 (PR #17, audit-log Amendment text).** UNSAFE-2026-0014's 4th Amendment header reads *"Amendment (2026-05-07, T-015 / ADR-0032): scope extended to the Deadlock-rollback `&mut EndpointArena` + `&mut IpcQueues` block."* The body then notes that the cancel block also carries *"`&*caller_table` (immutable), reflecting that recovery does not mutate caller-table state"* — a careful and correct call-out. Suggested closure: leave as-is; future readers reaching the body will see the complete enumeration. Alternatively, a small body-side rider could note "title omits the `&CapabilityTable` borrow for brevity; see Additional locations".**
