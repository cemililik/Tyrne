## 1. Capability rights & validation in `ipc_cancel_recv`

Adversarial frame: *can a caller invoke `ipc_cancel_recv` without holding the right that authorised the original `ipc_recv` registration?*

- **Two-layer cap validation present.** **OK** — the first line of [`ipc_cancel_recv`](../../../../../kernel/src/ipc/mod.rs#L470) calls `validate_ep_cap(caller_table, ep_cap, CapRights::RECV)` ([`kernel/src/ipc/mod.rs:485`](../../../../../kernel/src/ipc/mod.rs#L485)). `validate_ep_cap` performs (a) `lookup` on the caller's table → `InvalidCapability` on stale handle, (b) `rights().contains(RECV)` → `InvalidCapability` on missing right, (c) object-kind discriminator match (`CapObject::Endpoint`) → `InvalidCapability` on a notification cap that happens to coexist at the same handle. The arena `get` re-check at [`:472-474`](../../../../../kernel/src/ipc/mod.rs#L472-L474) catches a generation mismatch *between* the lookup and the `state_of` call (TOCTOU window discussed in §5). Symmetric to `ipc_recv`'s validation — the same RECV right authorises the cancel as authorised the original registration. The doc-comment at [`:455-459`](../../../../../kernel/src/ipc/mod.rs#L455-L459) makes the contract explicit.
- **Rights enforcement test pinned.** **OK** — [`cancel_recv_without_recv_right_fails`](../../../../../kernel/src/ipc/mod.rs#L1234-L1246) sets up a SEND-only cap and asserts `IpcError::InvalidCapability`. Symmetric to the existing `recv_without_recv_right_fails`.
- **`Idle`-state cancel is structurally indistinguishable from `RecvWaiting`-state cancel.** **Major (forward-flag) — tracked, no v1 action.** `EndpointState::RecvWaiting` is a unit variant ([`ipc/mod.rs:157`](../../../../../kernel/src/ipc/mod.rs#L157)) — the slot itself implies "the (single) registered receiver" under v1's depth-1 single-waiter discipline. The kernel **records no waiter identity**, so any holder of a RECV cap can cancel any other holder's pending receive on the same endpoint. In v1 this is unobservable: the only production caller is the same `ipc_recv_and_yield`'s Phase 2 Deadlock branch as the task that just performed the Phase 1 `ipc_recv`; no userspace exists; `ipc_cancel_recv` is `pub` in `kernel::ipc` ([`ipc/mod.rs:464`](../../../../../kernel/src/ipc/mod.rs#L464)) but only reachable from the kernel + BSP linker scope. ADR-0032 §Decision outcome's "no userspace caller in v1" framing is correct *for v1*, and the doc-comment at [`ipc/mod.rs:447-453`](../../../../../kernel/src/ipc/mod.rs#L447-L453) explicitly forward-flags the signature change to `caller: TaskHandle` when multi-waiter wake-up lands. **The risk is real for B5+ syscall exposure, not for v1.** This is not a regression introduced by PR #17 — it is a property of the unit-variant `RecvWaiting` that has been there since T-003 — but PR #17's `ipc_cancel_recv` is the first primitive that *makes that property observable* by giving a holder of the RECV cap the ability to *erase* a registration without observing it. ADR-0032 documents this in §Negative ("kernel-internal-only in v1; no userspace observer") and the forward-flag is structurally correct. Tracked, no v1 action; **gating item for the future syscall-ABI ADR (currently ADR-0030 placeholder)**.
- **No "third-party cancel" privilege escalation in v1.** **OK** — even with the unit-variant-implies-identity property above, v1 has no path that materialises a hostile RECV-cap holder. The cooperative two-task IPC demo's `task_a` / `task_b` each hold their own SEND-only and RECV-only caps; no task holds another's RECV cap. The potential escalation surface is forward-only (B2+ userspace, B5+ syscall).

## 2. Endpoint-state discipline

Adversarial frame: *the cancel arc must only fire from `RecvWaiting`. What does it do on `Idle` / `SendPending` / `RecvComplete`? Should be `Err(IpcError::WrongState)` or similar — verify.*

- **`Idle` is a deliberate no-op + `Ok(())`.** **OK by ADR-0032 design.** The implementation at [`ipc/mod.rs:476-480`](../../../../../kernel/src/ipc/mod.rs#L476-L480) is `if matches!(state, EndpointState::RecvWaiting) { *state = EndpointState::Idle; } Ok(())`. Tested by [`cancel_recv_on_idle_is_noop`](../../../../../kernel/src/ipc/mod.rs#L1180-L1192) and [`cancel_recv_is_idempotent`](../../../../../kernel/src/ipc/mod.rs#L1248-L1262). The doc-comment at [`:438-439`](../../../../../kernel/src/ipc/mod.rs#L438-L439) explicitly names this contract: "**`Idle`**: no-op; returns `Ok(())`. Idempotent — a redundant cancel call is harmless." Adversarial probe: could a hostile caller use the no-op to probe endpoint state? Yes — but `peek_state`/`state_of` already provide the same observability through `ipc_send`/`ipc_recv` typed errors (`QueueFull` vs `Pending` vs `Delivered`); the cancel no-op leaks no new information.
- **`SendPending { .. }` and `RecvComplete { .. }` are deliberate no-ops + `Ok(())`.** **OK with caveat.** Implementation: same `if matches!(... RecvWaiting)` predicate ⇒ no-op for these branches. Doc-comment at [`:440-445`](../../../../../kernel/src/ipc/mod.rs#L440-L445) records the rationale: "Phase 1 has already advanced past the recv-only state, so the cancel has nothing to undo. (In v1 this branch is unreachable from the Deadlock path because Phase 1's `ipc_recv` returned `Pending` only after observing `Idle`; the broader IPC surface may reach it once preemption / multi-waiter shapes land.)" Test [`cancel_recv_on_send_pending_does_not_drop_message`](../../../../../kernel/src/ipc/mod.rs#L1194-L1232) verifies the parked sender's message survives the cancel — important because a naive "blanket clear to Idle" would drop the in-flight send. **Caveat — Minor.** The `SendPending { cap: Some(_) }` / `RecvComplete { cap: Some(_) }` branches are also unreachable in the v1 cooperative single-thread discipline, but a future test that constructs `SendPending { cap: Some(_) }` and calls `ipc_cancel_recv` is the natural pin against a future bug where the predicate gets widened ("cancel any non-Idle state"). Today the predicate is correctly narrow (only `RecvWaiting` matches), so the cap-bearing states are protected by exactly the same single-pattern arm. The forward-test would harden the contract; not a v1 blocker.
- **No `IpcError::WrongState` variant introduced.** **OK by design.** The review prompt asked whether non-`RecvWaiting` states should return `Err(IpcError::WrongState)`. ADR-0032 §Decision outcome chose "`Idle`/`SendPending`/`RecvComplete` no-op + `Ok(())`" deliberately — the recovery primitive is *idempotent* by contract. Returning `Err(WrongState)` would couple every future caller (preemption-rollback, userspace-destroy drain) to remember which state the endpoint was in *before* calling cancel — defeating the purpose of a recovery primitive. The idempotency contract is the right choice; the test set pins it.

## 3. Generation invalidation

Adversarial frame: *if the endpoint slot was destroyed and reused between recv and cancel, does generation tracking catch it?*

- **`ep_arena.get(ep_handle.slot())` is the live-handle gate.** **OK** — at [`ipc/mod.rs:472-474`](../../../../../kernel/src/ipc/mod.rs#L472-L474), `ep_arena.get(...)` returns `None` if the slot's current generation does not match `ep_handle`'s generation; `.ok_or(IpcError::InvalidCapability)?` short-circuits the cancel with no state mutation. This is the same gate `ipc_send` and `ipc_recv` use ([`:286-288`](../../../../../kernel/src/ipc/mod.rs#L286-L288), [`:350-352`](../../../../../kernel/src/ipc/mod.rs#L350-L352)) — uniform discipline.
- **`IpcQueues::reset_if_stale_generation` second-line defence.** **OK** — `state_of` calls `reset_if_stale_generation` first ([`ipc/mod.rs:235-238`](../../../../../kernel/src/ipc/mod.rs#L235-L238)). If the queue's recorded generation lags `ep_handle`'s, the slot is reset to `Idle` — and the cancel's `if matches!(state, RecvWaiting)` branch is *not* taken, so the cancel correctly observes "no registration to undo" rather than blanket-clearing a state that belonged to a different endpoint. The `debug_assert!` at [`:220-228`](../../../../../kernel/src/ipc/mod.rs#L220-L228) catches the destruction-path leak hazard ("destroy forgot to drain a `Some(cap)` state") — defensive, future-proofing for the B2+ destroy path. Adversarial probe: could a sequence "destroy ep slot k → realloc same slot k with fresh generation → call `ipc_cancel_recv` with the *old* handle" reach into the new endpoint's state? **No** — `ep_arena.get` rejects the stale handle at line 472 before `state_of` is touched. The two checks are belt-and-braces.
- **Generation wrap-around (inherited).** **flagged — inherited, unchanged.** `u32` slot generation wraps after 2³² reuses per slot; the prior tracks recorded this as a B-late item. Not introduced or worsened by PR #17.

## 4. Ambient authority leakage

Adversarial frame: *does any new code path materialise authority a caller didn't have? Is `ipc_cancel_recv` exposed in a way userspace could reach in v1?*

- **`ipc_cancel_recv` is `pub` in `kernel::ipc`.** **OK in v1.** The function is `pub fn ipc_cancel_recv(...)` at [`ipc/mod.rs:464`](../../../../../kernel/src/ipc/mod.rs#L464); the module is `pub mod ipc` at [`kernel/src/lib.rs:52`](../../../../../kernel/src/lib.rs#L52). The symbol is therefore reachable to anyone linking `tyrne-kernel`. v1 has only one such linker — the `tyrne-bsp-qemu-virt` crate — and the BSP's `task_a` / `task_b` bodies do not call it. There is no syscall surface; there is no userspace; the symbol is "exposed" only in the same kernel-internal sense `ipc_send` and `ipc_recv` already are. No ambient authority is materialised.
- **Deadlock-branch use of `ipc_cancel_recv` is kernel-internal.** **OK** — the only production caller is [`kernel/src/sched/mod.rs:983-988`](../../../../../kernel/src/sched/mod.rs#L983-L988) inside `ipc_recv_and_yield`'s Phase 2 Deadlock branch. Phase 1 has already validated `ep_cap` against `caller_table` with the RECV right (at [`sched/mod.rs:912-913`](../../../../../kernel/src/sched/mod.rs#L912-L913)); the Deadlock-branch cancel call uses **the same `ep_cap` and `caller_table`**, so the cancel cannot surface a new `InvalidCapability` under v1's cooperative single-thread invariant. The `debug_assert!` at [`sched/mod.rs:989-993`](../../../../../kernel/src/sched/mod.rs#L989-L993) pins this expectation in debug builds; the typed-`Result` discard in release is acceptable because the only possible `Err` (a `caller_table` mutation between Phase 1 and the Deadlock cancel — impossible under cooperative single-thread discipline) is itself a kernel invariant violation, not a security failure.
- **No new syscall, no new ABI surface.** **OK** — PR #17 adds no `svc`/`smc`/`hvc`/`ecall` instruction; no syscall dispatch table; no exit-to-EL0 path. The `ipc_cancel_recv` Rust symbol is reachable from userspace only when (a) a syscall layer lands (B5+) and (b) that ABI ADR explicitly exposes the recovery primitive. ADR-0032 §Negative says "kernel-internal-only in v1; no userspace observer" — verified.
- **Symmetric to ADR-0017 surface.** **OK** — ADR-0017's *Decision outcome* enumerates `send` / `recv` / `notify` as the *user-observable* set; the §Revision notes rider added by PR #17 ([`docs/decisions/0017-ipc-primitive-set.md:212-215`](../../../../decisions/0017-ipc-primitive-set.md#L212-L215)) explicitly records `cancel_recv` as a *recovery primitive* that does not extend the user-observable surface. The wording is accurate: in v1 it is not user-observable. Track E (docs) verifies the rider lands; this track verifies the *security claim* the rider makes.

## 5. TOCTOU windows

Adversarial frame: *any window between cap validation and state mutation? The momentary `&mut` discipline (drop scheduler `&mut` before taking arena/queue `&mut`) — does it open a TOCTOU?*

- **Phase 1 → Deadlock-branch cancel: no preemption window in v1.** **OK** — between Phase 1's `validate_ep_cap` ([`sched/mod.rs:912-913`](../../../../../kernel/src/sched/mod.rs#L912-L913)) and the Deadlock-branch cancel ([`sched/mod.rs:983-988`](../../../../../kernel/src/sched/mod.rs#L983-L988)), no `cpu.context_switch` runs (the Deadlock branch is structurally the *failure-to-switch* path) and no `IrqGuard` is taken/dropped (the cancel happens before the switch attempt's `IrqGuard`). v1 is cooperative single-thread; no preemption arrives mid-flow. The cap and the endpoint state observed by Phase 1 are the same cap and state observed by the cancel. **Forward-flag — Major when preemption lands.** With B5+ preemption, an IRQ between Phase 1 and the cancel could in principle revoke the cap, destroy the endpoint, or schedule another task that drains the `RecvWaiting` slot. The cancel's `validate_ep_cap` + `ep_arena.get` + `reset_if_stale_generation` chain handles all three (revocation → `InvalidCapability`; destroy → `InvalidCapability`; drain by sender → state is no longer `RecvWaiting`, cancel no-ops). The preemption-rollback ADR (B5+) inherits the same primitive; the symmetric-rollback claim of ADR-0032 is preserved. *Tracked, no v1 action.*
- **The momentary `&mut` discipline does not open a new TOCTOU.** **OK** — the Deadlock branch's borrow scope structure (verified at [`sched/mod.rs:939-995`](../../../../../kernel/src/sched/mod.rs#L939-L995)) is:
  1. Phase 1 inner block: `&mut EndpointArena` + `&mut IpcQueues` + `&mut CapabilityTable` — acquired together, dropped together at the close-brace before line 916. Phase 1's `validate_ep_cap` and `ipc_recv` happen inside this single block.
  2. Phase 2 dispatch inner block: `&mut Scheduler<C>` only — acquired at line 944, dropped at line 969.
  3. Deadlock-branch cancel inner block: `&mut EndpointArena` + `&mut IpcQueues` + `&CapabilityTable` (immutable) — acquired at line 983, dropped at line 988.
  
  No two `&mut` borrows on the same arena/queue/scheduler are alive at the same instant. Stacked Borrows verifies this on every Miri run. **The cancel's `&CapabilityTable` is intentionally immutable** — recovery does not mutate caller-table state, which is the right semantic for a recovery primitive (UNSAFE-2026-0014 fourth Amendment captures this explicitly).
- **No re-validation gap inside `ipc_cancel_recv`.** **OK** — within the cancel function body, the sequence is `validate_ep_cap` → `ep_arena.get` → `state_of` → mutate. Each step holds the same `&mut`/`&` borrows for the entire body; no point in the body releases a borrow before the mutation completes. v1's cooperative single-thread discipline makes interleaving impossible.
- **Two-table TOCTOU not reachable.** **OK** — `ipc_cancel_recv` reads only `caller_table` (immutable `&`) and mutates only `IpcQueues`. The cross-table cap-revocation gap (ADR-0023) is unaffected; cancel does not transfer or install any cap.

## 6. ADR-0032's symmetric-rollback claim

Adversarial frame: *ADR-0032 says the symmetric rollback closes the ADR-0017 asymmetry. Verify the claim against the merged source.*

- **Scheduler-side rollback unchanged from pre-T-015.** **OK** — `s.task_states[current_idx] = prior_state; s.current = Some(current_handle);` at [`sched/mod.rs:965-966`](../../../../../kernel/src/sched/mod.rs#L965-L966). Identical shape to the pre-PR-#17 rollback (the Track A 2026-05-06 non-blocker that ADR-0032 cites). Restores `current` and `task_states[current_idx]` from `Blocked → Ready`; the ready queue is left untouched (the caller was never in it).
- **Endpoint-side rollback newly added.** **OK** — the cancel block at [`sched/mod.rs:983-988`](../../../../../kernel/src/sched/mod.rs#L983-L988) reverses Phase 1's `Idle → RecvWaiting`. The tests prove the symmetry: [`ipc_recv_and_yield_returns_deadlock_when_ready_queue_empty`](../../../../../kernel/src/sched/mod.rs#L1318-L1378) (extended by T-015 with the post-cancel `RecvOutcome::Pending` assertion at [`:1372-1377`](../../../../../kernel/src/sched/mod.rs#L1372-L1377)) and the dedicated regression guard [`ipc_recv_and_yield_deadlock_rolls_back_endpoint_state`](../../../../../kernel/src/sched/mod.rs#L1387-L1431). Both replay Simulation table row 3b from ADR-0032.
- **State-machine doc matches code.** **OK** — [`docs/architecture/ipc.md:41`](../../../../architecture/ipc.md#L41) adds the `RecvWaiting --> Idle: ipc_cancel_recv (recovery)` Mermaid edge to the state diagram; the prose at [`:63`](../../../../architecture/ipc.md#L63) names the v1 consumer (Deadlock branch only) and the future B2+/B5+ consumers explicitly.
- **`SchedError::Deadlock`'s doc-comment §Rollback scope expanded.** **OK** — [`sched/mod.rs:188-206`](../../../../../kernel/src/sched/mod.rs#L188-L206) records both halves of the rollback; the wording "atomic relative to the caller's observation" is accurate because the caller cannot interleave between the two `&mut` blocks (cooperative single-thread).
- **The symmetric-rollback claim holds for v1 *and* for the v2 surfaces ADR-0032 §Negative enumerates.** **OK** — multi-waiter wake (ADR-0019 §Open) gets the cancel primitive ready-made; preemption-rollback (B5+) gets the cancel primitive ready-made; userspace-destroy drain (B2+) gets the cancel primitive ready-made. The signature change ADR-0032 forecasts (gain a `caller: TaskHandle` parameter on multi-waiter landing) is forward-flagged in [`ipc/mod.rs:447-453`](../../../../../kernel/src/ipc/mod.rs#L447-L453); the v1 surface remains a clean step toward the v2 surface.

## 7. Forward-arc compatibility

Adversarial frame: *does the new arc leave the future userspace-destroy drain path (B2+) in a sound shape, multi-waiter wake (ADR-0019 §Open), and preemption-rollback (B5+) in a sound shape?*

- **B2+ userspace endpoint destroy drain.** **OK forward — but Major forward-flag for the destroy-path ADR.** ADR-0032 §Positive ("Forward-cleanup primitive for endpoint destroy") promises that userspace-driven destroy can call `ipc_cancel_recv` as part of a "drain receivers" sweep, eliminating the silent-drop hazard `IpcQueues::reset_if_stale_generation`'s `debug_assert!` currently catches in tests but tolerates in release. **The promise is structurally correct** — the cancel primitive is the right shape for the drain sweep. **The forward-flag the destroy-path ADR must address:** today's `ipc_cancel_recv` only handles `RecvWaiting`; the destroy sweep also needs to handle `SendPending { cap: Some(_) }` and `RecvComplete { cap: Some(_) }` *with* their carried capabilities. The current `ipc_cancel_recv` doc-comment correctly names these as no-op branches, which is the *wrong* behaviour for a destroy drain (the destroy path needs to *return the cap to the sender's table* or *destroy it*, not no-op). PR #17 does not introduce this gap — the destroy path doesn't exist yet — but ADR-0032's "Forward-cleanup primitive for endpoint destroy" wording should not be read as "the destroy path is done"; the destroy path needs its own state-machine ADR + a *separate* primitive (or an extended `ipc_cancel_recv`) for the cap-bearing branches. *Tracked for the destroy-path ADR; no v1 action.*
- **Multi-waiter wake (ADR-0019 §Open).** **OK forward.** The doc-comment at [`ipc/mod.rs:447-453`](../../../../../kernel/src/ipc/mod.rs#L447-L453) explicitly says "When multi-waiter wake-up lands, this signature gains a `caller: TaskHandle` parameter to remove the named caller from the waiter list rather than blanket-clearing the slot." This is the right forecast. Until that lands, the unit-variant `RecvWaiting` is a v1-only simplification; the multi-waiter ADR will need to introduce a `RecvWaiting { waiters: BoundedSet<TaskHandle> }` or equivalent and re-pin the cancel test set against the per-waiter form. *Tracked, no v1 action.*
- **Preemption-rollback (B5+).** **OK forward.** A preempted Phase-2-bound task whose Phase 1 has already transitioned the endpoint, but whose Phase 2 has not yet executed, leaves the endpoint in `RecvWaiting` across the preemption boundary — the same shape as the Deadlock case, with the same fix (call `ipc_cancel_recv`). The preemption ADR inherits ADR-0032's primitive directly. The TOCTOU §5 forward-flag is the gating consideration: the preemption ADR must specify whether the cancel runs in the IRQ handler (kernel-mode discipline §4 in the prior security review forbids `&mut Scheduler<C>` in IRQ handlers per UNSAFE-2026-0014) or is deferred to the resumption point. The current `ipc_cancel_recv` body takes `&mut EndpointArena` + `&mut IpcQueues` + `&CapabilityTable` — *not* `&mut Scheduler<C>` — so the preemption ADR has freedom to arrange the call. *Tracked, no v1 action.*
- **Future-test-coverage gap.** **Minor forward-flag.** The B5+ preemption-rollback ADR will surface the question "does cancel ever observe `SendPending { cap: Some(_) }` from a preempted task that already advanced past Phase 1?" — a v1-unreachable scenario. Today's no-op branch is the correct v1 default but the wrong v2 default for that specific scenario. The cancel-on-cap-bearing-state test does not exist (only cancel-on-`SendPending { cap: None }` is tested via [`cancel_recv_on_send_pending_does_not_drop_message`](../../../../../kernel/src/ipc/mod.rs#L1194-L1232)). Adding a `cancel_recv_on_send_pending_with_cap_does_not_drop_cap` test today pins the no-op semantics for the cap-bearing branch; the destroy-path ADR will then know whether to extend or replace. *Track F (tests) follow-up; not a security blocker.*

## 8. Smoke-trace claim — security implication

Adversarial frame: *PR #17 claims smoke is "byte-for-byte unchanged from post-T-014". If no new code path is exercised by smoke, the cancel arc is unverified end-to-end. Is the residual risk acceptable for v1?*

- **Smoke does not exercise the cancel path.** **OK with caveat.** The PR #17 description states: "byte-for-byte identical message sequence to the post-T-014 baseline (T-015 adds no v1-reachable code path; Deadlock is structurally unreachable with `register_idle` installed per ADR-0026)." Verified by reading the merged scheduler dispatch chain ([`sched/mod.rs:956`](../../../../../kernel/src/sched/mod.rs#L956)): `s.ready.dequeue().or(s.idle)` returns `Some(idle)` whenever `s.idle` is set; the BSP `kernel_entry` calls `register_idle` before `start`, so `s.idle.is_some()` holds for the entire lifetime of the kernel; the Deadlock branch (and therefore the cancel call) is structurally unreachable.
- **Host-test coverage substitutes for end-to-end.** **OK** — [`ipc_recv_and_yield_returns_deadlock_when_ready_queue_empty`](../../../../../kernel/src/sched/mod.rs#L1318-L1378) constructs a scheduler with no idle registered, confirming the Deadlock branch fires *and* asserts both halves of the rollback. The dedicated [`ipc_recv_and_yield_deadlock_rolls_back_endpoint_state`](../../../../../kernel/src/sched/mod.rs#L1387-L1431) test exercises the same arc with a focused endpoint-state assertion. Both run under `cargo +nightly miri test` (158/158 clean per PR #17 description) — Stacked Borrows verifies the momentary-`&mut` discipline of the new cancel block. The five IPC-side tests ([`cancel_recv_clears_recv_waiting_back_to_idle`](../../../../../kernel/src/ipc/mod.rs#L1158-L1178), [`cancel_recv_on_idle_is_noop`](../../../../../kernel/src/ipc/mod.rs#L1180-L1192), [`cancel_recv_on_send_pending_does_not_drop_message`](../../../../../kernel/src/ipc/mod.rs#L1194-L1232), [`cancel_recv_without_recv_right_fails`](../../../../../kernel/src/ipc/mod.rs#L1234-L1246), [`cancel_recv_is_idempotent`](../../../../../kernel/src/ipc/mod.rs#L1248-L1262)) pin the state-machine contract directly.
- **Residual risk.** **acceptable for v1.** The cancel arc is exercised under Miri (Stacked Borrows + UB checking) but not under hardware-realistic conditions. There is no cache-coherency, no MMIO, no IRQ delivery in the cancel path — all of those are absent by construction (the cancel runs at scheduler-bridge level, post-Phase-1, pre-switch, with no hardware handshake). The host-test coverage is a sound substitute for v1; the QEMU smoke is the right next-line defence when (a) preemption lands or (b) userspace-destroy lands. **Forward-flag:** when the first userspace-driven endpoint destroy task lands (B2 or later), its acceptance criteria should include a QEMU-smoke variant that exercises the cancel-on-destroy drain path. Tracked for the B2+ task; not a v1 blocker.

## Verdict

**Approve.**

PR #17 closes the 2026-05-07 consolidated security review's forward-flagged item ("`ipc_recv_and_yield` Deadlock endpoint-rollback asymmetry") in the way the review expected: ADR-first per ADR-0025 §Rule 1, primitive-based per ADR-0032 Option A, ahead of the B2 userspace-destroy work that would have surfaced the asymmetry as a live bug. The other five PRs in the arc (#12–#16) introduce no security regressions; PR #15's `debug_assert! → assert!` upgrade on `register_idle` and PR #14's SECURITY.md cleanup are net-positive.

The merged source at HEAD `c258ee3` enforces:
1. **Capability rights validation on cancel** — `validate_ep_cap(..., RECV)` + arena-live-handle gate + `state_of`'s generation reset (§1, §3).
2. **Idempotent state-machine contract** — `RecvWaiting → Idle` only; all other states no-op + `Ok(())` (§2).
3. **No ambient authority** — kernel-internal `pub`; no syscall surface; no userspace observer in v1 (§4).
4. **No new TOCTOU** — momentary `&mut` discipline preserved; cooperative single-thread invariant unchanged (§5).
5. **Symmetric rollback** — both scheduler *and* endpoint state restored on `SchedError::Deadlock` (§6).
6. **Forward-arc compatibility** — primitive shape supports multi-waiter, preemption-rollback, and userspace-destroy drain (§7).
7. **Host-test coverage substitutes for unreachable smoke path** — 5 IPC tests + 1 sched test + extended T-007 test, all Miri-clean (§8).

### Top-3 findings (this track)

1. **`RecvWaiting` unit-variant identity gap (Major forward-flag, tracked).** §1 / §7. The kernel records no waiter identity; v1's depth-1 single-waiter discipline makes the gap unobservable, but the future syscall-ABI ADR (currently ADR-0030 placeholder) and the multi-waiter ADR must address it before either lands. ADR-0032 forecasts the signature change correctly.
2. **Cancel-on-cap-bearing-state destroy-drain semantics (Minor forward-flag, tracked).** §2 / §7. The current no-op on `SendPending { cap: Some(_) }` and `RecvComplete { cap: Some(_) }` is correct for v1 Deadlock-recovery but is the wrong default for B2+ userspace-destroy drain. ADR-0032 §Positive's "Forward-cleanup primitive for endpoint destroy" wording should not be misread as "destroy is done"; the destroy-path ADR needs its own primitive (or an extended `ipc_cancel_recv`) for the cap-bearing branches.
3. **Cancel-on-cap-bearing-state regression test (Nit, Track F follow-up).** §7. A `cancel_recv_on_send_pending_with_cap_does_not_drop_cap` test today would pin the no-op semantics for the cap-bearing branch; not a v1 security blocker, but cheap forward-protection.

### Forward-flagged items (carry-forward; non-blocking)

- `IpcError::WrongObjectKind` / `MissingRight` variant split (ADR-0030, B5).
