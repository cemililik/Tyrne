## Executive summary

The HAL/BSP surface across PRs #12–#17 is **clean**. Only PRs #12 and #15 carry meaningful HAL/BSP code; #13 / #16 / #17 are doc- or kernel-only; #14 is also doc-only on the HAL/BSP axis (the bash-3.2 `tools/run-qemu.sh` empty-array fix the brief attributed to #14 actually landed earlier, in PR #12 — see [Cross-PR observations](#cross-pr-observations) §1 — and is also reachable from PR #14's history). The two changes that *do* touch HAL/BSP are both Track-B comprehensive-review-2026-05-06 follow-ups landing precisely as recommended:

1. PR #15 lifts `ContextSwitch` to `Send + Sync` (Track B Non-blocking #2 of [`track-b-hal.md`](../2026-05-06-full-tree/track-b-hal.md)) and adds two compile-time size-asserts (`TrapFrame == 192` already existed; `Aarch64TaskContext == 168` is new) plus prose-only comments on the IRQ-spurious `compiler_fence`, `irq_entry`'s `_frame` rationale, and the `start(...) -> !` no-defensive-loop rationale (Track G Non-blocking #1 / #3 / #4 of [`track-g-bsp.md`](../2026-05-06-full-tree/track-g-bsp.md)).
2. PR #12's BSP-side change is a single-site rewire of `kernel_entry`'s task-add sequence: idle is now registered via `register_idle(...)` instead of `add_task(...)`. This implements ADR-0026 on the BSP side without touching `boot.s`, `vectors.s`, the GIC, the EL-drop sequence, or any HAL trait.

The IRQ-dispatch path (`vectors.s+0x280` → `tyrne_irq_curr_el_trampoline` → `irq_entry` → `gic.acknowledge()` → timer mask + EOI) is **unchanged** at the byte level since 2026-04-28's T-012 commit `a043079`; PR #15 added one prose comment and one redundant-but-deliberately-kept `compiler_fence` rationale and changed nothing else. The 2026-05-06 *partial-verification* + *post-T-014 smoke* Amendments on UNSAFE-2026-0019 / 0020 / 0021 still match reality post-PR #15 / #17 (no new BSP code touches the audited lines).

The `Cpu::wait_for_interrupt` discipline is preserved end-to-end: PR #12's idle-dispatch rewire keeps `cpu.wait_for_interrupt()` as the single HAL touchpoint inside [`bsp-qemu-virt::idle_entry`](../../../../../bsp-qemu-virt/src/main.rs#L233), and the scheduler never calls it. The BSP boot checklist's six rows all still pass.

**Sub-verdict: Approve, with three small follow-ups (all Minor / Nit).** No Blocker, no Major.

### Nit

- **B-2 (Nit — comment-only, optional):** The new prose comment on `irq_entry`'s spurious branch ([`bsp-qemu-virt/src/exceptions.rs:177–186`](../../../../../bsp-qemu-virt/src/exceptions.rs#L177)) explicitly names the `compiler_fence(SeqCst)` as "structurally redundant on this branch … kept as a guard against a future refactor that adds shared-state writes between the spurious-detect and the return — drop only when that risk is closed by an ADR." This is exactly the resolution the 2026-05-06 Track-G review proposed (Non-blocking #1 of [`track-g-bsp.md`](../2026-05-06-full-tree/track-g-bsp.md) — "drop the fence, or keep it and add a one-line note"). The wording is slightly stronger than the original suggestion (it asserts the fence is structurally redundant, then keeps it on defence-in-depth grounds), and it bakes the requirement that any future removal be ADR-gated. That is fine for v1 — the cost of a `compiler_fence` on a path that fires only when `gic.acknowledge()` returns INTID 1023 (architecturally rare; never observed in v1 smoke per the audit-log Amendments) is sub-pico-second on aarch64. **Suggested follow-up (defer):** if a future code-review round wants to actually drop the fence, the ADR gate is now explicit; until then the comment turns the discrepancy into documentation rather than mystery. No action.

- **B-3 (Nit — `irq_entry`'s `_frame` parameter justification):** The new comment block at [`bsp-qemu-virt/src/exceptions.rs:144–150`](../../../../../bsp-qemu-virt/src/exceptions.rs#L144) ("`_frame` is unused in v1: the asm trampoline at `vectors.s+0x280` constructs the `TrapFrame` and passes its pointer in `x0` for ABI uniformity; future arcs … will read or modify the frame.") closes Track-G Non-blocking #3. The justification correctly cites UNSAFE-2026-0014's third Amendment (which names `irq_entry` as a future site of the momentary-`&mut Scheduler<C>` pattern), so a future reader who wants to remove the parameter sees both *why* it is shaped this way and *what audit chain* governs the future read. The comment lives directly above the `#[unsafe(no_mangle)]` attribute so it is visually attached to the function rather than to the unrelated `# Safety` doc-comment two paragraphs above. **No action.**

- **B-4 (Nit — `Aarch64TaskContext` size assert is healthy, but its discipline is not yet codified):** PR #15 adds `const _: () = assert!(core::mem::size_of::<Aarch64TaskContext>() == 168);` at [`bsp-qemu-virt/src/cpu.rs:326`](../../../../../bsp-qemu-virt/src/cpu.rs#L326). The motivation comment at lines 321–325 ("Mirror the discipline applied to `TrapFrame` so the drift fails the build rather than the first IRQ.") is correct and matches the existing `TrapFrame == 192` assert at [`exceptions.rs:77`](../../../../../bsp-qemu-virt/src/exceptions.rs#L77). However, the `vectors.s` side of the trampoline still encodes `sub sp, sp, #192` and the `stp` offsets `0x00..0xA0` as raw literals — the asm side has no equivalent guard. PR #15's `vectors.s` comment block ([`bsp-qemu-virt/src/vectors.s:114–120`](../../../../../bsp-qemu-virt/src/vectors.s#L114)) does explicitly cross-reference the Rust-side assert ("the compile-time guard `const _: () = assert!(core::mem::size_of::<TrapFrame>() == 192)` next to the `TrapFrame` definition in `bsp-qemu-virt/src/exceptions.rs` fails the build before the drift can reach an IRQ. Update both sides together if the frame ever grows."), which addresses Track-G Non-blocking #2's *cheaper* remedy. The ideal long-term shape (a single source of truth — e.g. `TYRNE_TRAP_FRAME_SIZE` as a `.equ` consumed by both sides via `global_asm!` substitution) remains a future ergonomics improvement, not a regression. **Tracked, no action.** The exact same shape now applies to `Aarch64TaskContext`'s offsets in `context_switch_asm` ([`cpu.rs:374–402`](../../../../../bsp-qemu-virt/src/cpu.rs#L374)): the asm encodes literals 0/16/32/48/64/80/96/104/120/136/152, the assert pins the total size, and the comment chains the two together. Same caveat: a future change that grows the context (e.g. saving FPSR/FPCR, adding TPIDR_EL0) needs to update the literal offsets *and* the asserted size *and* potentially the unrelated boot stack reservation in `linker.ld`. The follow-up shape is identical to Track-G Non-blocking #2; treating both as one consolidated future ergonomics task is correct.

The Amendments' "still not verified" claims continue to match reality. No new audit-log Amendment is owed by these PRs on the IRQ-dispatch axis.

5. **PR #12's idle-dispatch rewire respects the `Cpu::wait_for_interrupt` HAL contract.** Verified by reading [`bsp-qemu-virt/src/main.rs:233–290`](../../../../../bsp-qemu-virt/src/main.rs#L233) (idle_entry body, post-PR #12) and `kernel/src/sched/mod.rs` for callers of `wait_for_interrupt`: the only call site in the entire workspace is `idle_entry`'s loop body. The scheduler does not call `wait_for_interrupt` directly. ADR-0026's *Decision outcome* preserves this property (the dispatcher routes *to* idle_entry when `s.ready.dequeue() == None && s.idle.is_some()`; idle_entry then calls `cpu.wait_for_interrupt()` itself). Brief expectation upheld.

6. **PR #15's prose-only `start(...)` no-defensive-loop justification at [`bsp-qemu-virt/src/main.rs:737–743`](../../../../../bsp-qemu-virt/src/main.rs#L737) responds to Track-G Non-blocking #4 with a deliberate "no" rather than the suggested `loop { spin_loop(); }`.** The justification — that `-> !` is the type-system belt-and-braces and `unreachable_code` would flag the addition — is correct. A reasonable alternative would have been `core::hint::unreachable_unchecked()` after `start(...)`, but that would require `unsafe` and the rationale "the type system already proves nothing follows" is the right minimal-`unsafe` posture. **Disposition:** Track-G #4 is now closed via "documented rejection" rather than "applied"; this is a valid resolution form per the comprehensive-review master plan.
