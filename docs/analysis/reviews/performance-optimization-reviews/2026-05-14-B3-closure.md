**Observations:**

- **`.text` grew by +1,624 bytes** for the entire B3 surface:
  - `Pmm<N, R>` body (`new` constructor with reservation-tracking loop; `alloc_frame` bitmap-scan with hint pointer + UNSAFE-2026-0026 zero-fill; `free_frame` with reset-on-already-free guard; `stats` accessor; `Pmm`'s `FrameProvider` trait-impl bridge): ~480 bytes. The bitmap walk is `u64`-word-stepping with `trailing_zeros` so the per-frame cost is amortised; the binary cost of the scan loop is small.
  - `AddressSpace<M>` body (`wrap_bootstrap` + `from_mmu_address_space` constructors; `root_frame` / `inner` / `inner_mut` accessors; the per-type-arena `create_address_space` / `destroy_address_space` / `get_address_space` / `get_address_space_mut` free fns; the `AddressSpaceError` enum's `Debug` impl): ~280 bytes. The arena methods inline aggressively under `-C opt-level=3`; the symbol table shows their `#[inline]` annotations took effect.
  - `cap_create_address_space` body (eight-step preflight chain: lookup → kind → DERIVE → no-widening → depth → arena/table capacity → PMM alloc → arena commit → `cap_derive` insert): ~360 bytes. The doc-comment is large but the compiled body is bounded by the linear preflight sequence; no recursion, no loops.
  - `cap_map` + `cap_unmap` bodies (cap resolution → arena lookup → `Mmu::map` / `unmap` delegation → flush-token discharge): ~200 bytes total. Mostly delegation through the trait surface; the wrapper itself is thin.
  - `address_space_activation_target` pure helper + the activation-hook closure-threading in `yield_now` / `start` / `ipc_recv_and_yield` / `ipc_send_and_yield`: ~240 bytes net. The closure-as-FnOnce parameter is monomorphised per call site; the BSP closure's `Mmu::activate` body is the same as in T-016.
  - `CapabilityTable::depth_of` `pub(crate)` accessor: ~16 bytes (a single `entry_of(handle)?.depth` line; trivial).
  - LLVM cross-function re-inlining residuals from the activation-hook closure threading: ~48 bytes net per disassembly inspection (some scheduler entry points re-inlined their `next_handle` lookup site).
  - Net: +1,624 bytes ≈ sum of the contributions above.
- **`.rodata` grew by +592 bytes** — mostly the new `AddressSpaceError::*` and `PmmError::*` enum-`Debug` strings + a few new doc-derived constants. The cap-wrapper rejection messages introduced by the depth-preflight (round-3 fix) added a small chunk of static string content; the rest is enum-variant `Debug` impl tables expanding.
- **`.bss` grew by +1,872 bytes (+4.7 %)** — this is overwhelmingly the new `AddressSpaceArena<QemuVirtMmu, 8>` `StaticCell` (8 slots × sizeof(AddressSpace<QemuVirtMmu>) + generation tags + the `Arena`'s free-list head + arena occupancy bitmap). The `Pmm<32768, 4>` `StaticCell` was already accounted-for in T-017's partial landing (the static was published with the PR #26 banner; the `.bss` measurement at T-018 PR-merge captures both T-017's PMM + T-018's AddressSpaceArena because the B2 baseline didn't include either). **No new linker-script reservation** — both statics live in the regular `.bss` section, not a dedicated reservation like T-016's `.boot_pt`.

## Hotspot

The +6 to +7 ms p10/p50/p90 increase versus B2 closure is dominated by two new one-time-at-boot costs, both amplified by QEMU TCG's translation-cache miss behaviour:

### Hotspot 1 — `Pmm::new`'s bitmap initialisation pass

`Pmm::new` walks the full 32 768-frame extent to mark the reservation list (kernel image + `.boot_pt` + boot stack = 161 frames). The bitmap is 512 `u64` words; the mark loop touches each word. The constructor is called once at `kernel_entry`'s PMM stage. Under QEMU TCG, the first time the bitmap is touched the translation cache misses on the `.bss` page containing it; the translation cost is **roughly 1–2 ms per fresh TCG block** on the Cortex-A72 model with this host (Rosetta 2 / Apple Silicon). The §"Methodology" note about per-run variance reflecting "translation-cache behaviour and host scheduler jitter, not real hardware performance" is now load-bearing.

**Real-hardware projection:** the bitmap initialisation is ~512 `u64` writes + ~161 bit-set operations; on a Cortex-A72 at 1.4 GHz that's sub-microsecond. The QEMU TCG cost is an emulation artifact.

### Hotspot 2 — `AddressSpaceArena` bootstrap-slot publish + bootstrap-AS cap mint

The BSP wiring in `kernel_entry` does: (a) call `<QemuVirtMmu as Mmu>::wrap_existing_root(l0_root)` to construct the BSP-AS value, (b) call `AddressSpace::wrap_bootstrap(inner)` to wrap it with kernel metadata, (c) call `create_address_space(&mut AS_ARENA, bootstrap_as)` to publish into slot 0, (d) mint the kernel-init's bootstrap-AS cap via `CapabilityTable::insert_root`. The full sequence is ~6 function calls + 3 trait-method dispatches + 1 cap-table allocation. Under QEMU TCG, each first-time-translated function adds the per-block TCG cost; the total cost of the AS-arena ramp-up is **roughly 1–2 ms** per the same model.

**Real-hardware projection:** the AS-arena ramp-up is ~50 instructions on real Cortex-A72; sub-microsecond. The QEMU TCG cost is again an emulation artifact.

### Hotspot 3 — Sustained TCG translation-cache churn

The post-T-018 kernel image is +5.4 KiB larger (text + rodata + bss combined) than post-T-016. QEMU TCG's translation cache (default size ~32 MiB) is far larger than any single Tyrne image; the cache is not the limit. However, the *first-time-translated* footprint grows linearly with `.text`, and the boot-to-end window now includes more first-time-translated code (PMM scan + AS arena init + activation hook closure threading + cap-wrapper preflight chains). Each new first-time-translated function adds **roughly 0.2–0.5 ms** of QEMU TCG translation cost per run.

**Across 20 iterations:** the TCG cache warms after iteration 1, but each iteration is a fresh QEMU process invocation (the harness wraps `tools/run-qemu.sh` in a `perl -e 'alarm ...'` watchdog); the cache is destroyed between iterations. So **every iteration pays the full translation cost** — this is what produces the +6 to +7 ms band.

**Real-hardware projection:** translation-cache cost is zero on real hardware; the per-iteration cost on a Cortex-A72 would be the actual code execution time. ADR-0027's §Consequences projection ("sub-100 µs for MMU activation on real hardware") + ADR-0035's §Consequences projection ("sub-millisecond PMM init on real hardware") + the T-018 review-round arc's audit notes all converge on the same prediction: real-hardware boot-to-end should remain in the **sub-5 ms** range, dominated by UART output bandwidth (PL011 at 9600 baud transmits ~960 bytes/sec = ~32 ms for the 31-line boot trace; the kernel's `Mmu::activate` + PMM init + AS arena publish are sub-millisecond in aggregate).

### Why this is a baseline, not a regression

- The Δ is **purely QEMU TCG translation overhead** on a single emulated host; no real-hardware regression is projected.
- The cost is **one-time-at-boot**; per-iteration steady-state IPC + scheduler costs are unchanged (the activation hook short-circuits in v1's bootstrap-shared topology).
- The pattern is **consistent with B2's TCG-amplification observation** (B2's debug-build harness showed +1.5 ms per percentile for the MMU activation; that overhead is preserved in the release build and now compounded by T-017's PMM scan + T-018's AS arena publish).
- The harness band is **the canonical measurement convention** per [`infrastructure.md` §"Performance harness"](../../../standards/infrastructure.md#performance-harness); single-run claims are deprecated.

## Proposal

None this cycle. This artefact records the baseline for future B4+ regression checks; it does not propose an optimisation.

If a B4 task surfaces a real-hardware perf concern (the projected sub-5 ms boot-to-end claim is testable only when real hardware lands), a follow-up perf review can hypothesis-test the projection. Until then, the QEMU TCG band is informational, not normative.

**Rejected proposals (with reasoning):**

- *"Reduce `.bss` by shrinking `AddressSpaceArena<QemuVirtMmu, 8>` to N=4 since v1's demo only uses 1 slot."* Rejected — the arena is sized for B4 userspace bring-up (1 kernel-init + N userspace tasks); shrinking to N=4 now would require re-bumping in B4, churning the `.bss` shape needlessly. The +1,872-byte `.bss` cost is well within ADR-0035 + ADR-0028's bounded-footprint budgets.
- *"Optimise `Pmm::new`'s bitmap-mark loop to skip whole-word zeros."* Rejected — the loop already word-steps the `[u64; 512]` bitmap with `trailing_zeros` for hint advancement; the mark phase is O(R) where R is the reservation count (currently 4 ranges in v1); the bottleneck is QEMU TCG cold-start, not the loop. Real-hardware optimisation gains nothing here.
- *"Cache the activation closure invocation."* Rejected — the closure is `FnOnce` per scheduler invocation; `address_space_activation_target` already short-circuits to `None` when AS handles match (the v1 common case). No caching opportunity remains.
- *"Move the PMM bitmap to a `static mut [u64; 512]` instead of `StaticCell`."* Rejected — `StaticCell`'s `take()` discipline is load-bearing for the project's no-`unsafe`-on-the-bring-up-path policy; the cost difference is negligible and the safety property is real.

## Verdict

**Baseline — no proposal under measurement.**

The B3 closure baseline records a **+6 to +7 ms p10/p50/p90 boot-to-end increase under QEMU TCG** versus the B2 closure baseline. The increase decomposes cleanly into (a) `Pmm::new`'s bitmap initialisation pass over 32 768 frames, (b) the AddressSpaceArena bootstrap-slot publish + bootstrap-AS cap mint, and (c) sustained TCG translation-cache churn from the larger `.text` footprint (+1,624 bytes). All three are **one-time-at-boot costs amplified by QEMU TCG emulation**; real-hardware projections place the absolute boot-to-end well under 5 ms on a Cortex-A72 at 1.4 GHz.

The baseline is recorded as the canonical reference for B4+ regression checks. Cite the band (p10 / p50 / p90 = 10.311 / 11.884 / 13.823 ms; release; 20-iteration harness; QEMU 10.2.2 on Apple Silicon / Rosetta) when comparing later changes against this snapshot. Single-run boot-to-end claims in PR bodies should be replaced with a fresh harness run when a non-trivial perf-relevant change lands; see [`docs/standards/infrastructure.md` §"Performance harness"](../../../standards/infrastructure.md#performance-harness).

### Forward-flagged items

- **Real-hardware perf measurement is the load-bearing missing data point.** The QEMU TCG band is informational; the project's first real-hardware boot will produce the first real-hardware boot-to-end baseline. **Trigger:** opens when the first BSP for actual hardware (Raspberry Pi 4 per [README.md](../../../../README.md)'s mention) lands.
- **The `Pmm::new` bitmap init cost is bounded but linear in extent.** v1's 128 MiB extent makes the cost trivial; a future 4 GiB / 16 GiB / 64 GiB extent would scale the init cost linearly (4 GiB = 1,048,576 frames = 16 384 `u64` words; init pass would still be sub-millisecond on real hardware, but worth re-measuring at the new scale). **Trigger:** opens when the first BSP with >128 MiB RAM lands.
- **AddressSpaceArena slot count revisit.** Currently sized N=8 for v1; B4 userspace bring-up may justify N=16 or N=32. **Trigger:** opens with the B4 first task that lands actual userspace AS allocation.
- **Activation-hook cost when AS handles differ.** v1's hook short-circuits because all tasks share `BOOTSTRAP_ADDRESS_SPACE_HANDLE`; the *differ-path* cost (`Mmu::activate` invocation + `TLBI VMALLE1` global flush) is unmeasured at runtime. Host tests + Miri pin the path; runtime measurement gates on the first multi-AS task scenario. **Trigger:** opens with the first B4+ task that owns a non-bootstrap AS.
