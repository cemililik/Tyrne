## Baseline

### Methodology

- ELF section sizes via `rust-readobj --elf-output-style=GNU -S` (workspace nightly toolchain), release profile.
- Source LOC via `find … -name '*.rs' -o -name '*.s' | xargs wc -l`.
- Boot-to-end timing measured via [`tools/perf-harness.sh`](../../../../tools/perf-harness.sh) (P10 wall-clock harness, landed in PR #22). 20 iterations, 5 s per-run timeout, release build, single host. Single-run smoke trace recorded for the canonical record.
- Cross-build verification: `cargo +nightly miri test` on every host crate confirms no UB / Stacked-Borrows violation across the new code paths.

### Metric 1 — Kernel image size

| Section | post-T-015 (2026-05-07 baseline-of-record) | post-T-016 (2026-05-09) | Δ bytes | Δ % |
|---------|--------------------------------------------|-------------------------|---------|-----|
| `.text` | 22,020 (`0x5604`) | **22,384** (`0x5770`) | **+364** | **+1.7 %** |
| `.rodata` | 2,928 (`0xb70`) | **2,944** (`0xb80`) | +16 | +0.5 % |
| `.bss` | 22,256 (`0x56f0`) | **40,208** (`0x9d10`) | +17,952 | +80.7 % |

**Observations:**

- **`.text` grew by +364 bytes** for the entire MMU-activation surface:
  - `mmu_bootstrap` body (Step 1 loops + Step 2 sysreg writes + Step 3 invalidate-and-enable sequence): ~140 bytes.
  - `QemuVirtMmu::activate` (MSR TTBR0 + DSB ISHST + TLBI + DSB ISH + ISB sequence with the post-Finding-4 ordering): ~32 bytes.
  - `QemuVirtMmu::invalidate_tlb_address` + `invalidate_tlb_all` (TLBI + barrier sequences): ~48 bytes total.
  - `QemuVirtMmu::map` + `unmap` + `walk_and_install_leaf` + `walk_or_alloc_table` (the page-table walker; **dead-code-allow** in v1 because no kernel caller post-bootstrap): ~144 bytes — the `dead_code` annotation does NOT remove the symbols from the release binary because they're `pub` and reachable through the trait surface.
  - LLVM re-inlining residuals from the `kernel_entry` reorder (`boot_ns` snapshot moved earlier; GIC init moved later): ~0 bytes net per disassembly inspection.
  - Net: +364 bytes ≈ sum of the contributions above.
- **`.rodata` grew by +16 bytes** — the static descriptor / panic strings introduced by the page-table walker (`MmuError::AlreadyMapped` / `NotMapped` / `MisalignedAddress` / `OutOfFrames` / `InvalidFlags` are all `#[non_exhaustive]` enum variants whose `Debug` strings already lived in `.rodata` from B1; the +16 bytes is the new flag-validation panic message strings introduced by the round-2 Finding-3 fix and the `MapperFlush::new` doc-string-derived const data).
- **`.bss` grew by +17,952 bytes (+80.7 %)** — overwhelmingly from the new `.boot_pt` linker reservation (4 × 4 KiB = **16,384 bytes**) plus the `4 KiB alignment slack` between the existing `.bss` end and the `.boot_pt` start (`. = ALIGN(4096); __boot_pt_start = .;` introduces up to 4 KiB of zero-fill padding; the actual measured padding is ~1,568 bytes after the BSS-original tail). The `.boot_pt` reservation is **the v1 cost of static page-table allocation** — explicit and bounded per ADR-0027 §"Bounded bootstrap frame budget"; once the future PMM lands the bootstrap can shrink to a single L0 frame + dynamic L1/L2/L3 allocation, but that's a B5+ revisit.
- **No new kernel-allocated stack.** T-016 reuses every existing TaskStack from T-007 / T-014; the dispatch refactor changes nothing about per-task RAM. The +17,952 byte `.bss` growth is **exactly** the `.boot_pt` reservation + alignment, not per-task state.

**A6 → 2026-04-28 → 2026-05-07 → 2026-05-09 cumulative trajectory** (relative to A6 baseline of `.text 13,940` / `.rodata 1,960` / `.bss 17,872`):

| Section | A6 | B1 (2026-04-28) | post-T-014 (2026-05-07) | post-T-015 (2026-05-07) | **post-T-016 (2026-05-09)** | A6 → today total Δ |
|---------|----|-----|-----|-----|-----|--------------------|
| `.text` | 13,940 | 21,908 | 21,792 | 22,020 | **22,384** | **+8,444 (+60.6 %)** |
| `.rodata` | 1,960 | 2,784 | 2,928 | 2,928 | **2,944** | **+984 (+50.2 %)** |
| `.bss` | 17,872 | 22,248 | 22,256 | 22,256 | **40,208** | **+22,336 (+125 %)** |

The total kernel image (text + rodata + bss) is now ~64.2 KiB, up from ~46.5 KiB at B1 closure. The `.bss` surge is the dominant cost; it reflects the explicit, bounded MMU bootstrap budget. Within the implicit non-hypothesis budget set by [ADR-0027 §Consequences §Bounded bootstrap frame budget](../../../decisions/0027-kernel-virtual-memory-layout.md): "Four 4 KiB frames (16 KiB total) for the entire boot-time mapping" — measured `.bss` delta matches this exactly.

The debug-build Δ of **+1.5 ms p10/p50** is the QEMU TCG translation-cache overhead at MMU bring-up — translations that previously hit a flat passthrough path now go through the TLB, and TCG re-translates the hot path once on first execution. **Real-hardware activation cost is sub-100 µs** per [ADR-0027 §Consequences](../../../decisions/0027-kernel-virtual-memory-layout.md); the QEMU figure is ~15× the real-hardware figure due to TCG's instruction-by-instruction emulation. **No real-hardware regression projected** — the release-build closure-baseline numbers above are the v1 reference for any future regression check.

The release build is ~750 µs faster than debug at p10 / p50 (smaller binary footprint per inlining + LTO + `opt-level=3`). The `.text 22,384` release vs `.text` debug (typically ~1.5–2× larger) ratio is consistent with prior baselines' release-vs-debug gap.

### Methodology notes

The post-T-016 release-build harness band (Source B above) is the **first baseline-of-record** measuring kernel performance with the MMU active under release codegen. Future cycles use this as the reference for hypothesis-driven measurement; the existing debug-build baselines remain useful for debug-build regression checks but are not the closure-trio's canonical baseline.

The QEMU `-d int,unimp,guest_errors` window is **not empty** — it contains the pre-existing PL011 "data written to disabled UART" warnings (one per UART byte). This is a v1 trade-off (Tyrne rides QEMU's reset state rather than performing the canonical PL011 init sequence), well-documented in the [business retrospective §"What we learned"](../business-reviews/2026-05-09-B2-closure.md#what-we-learned) and queued as a follow-on B-phase BSP task. **The closure-trio criterion of "smoke trace pasted into the closure retrospective"** (codified at B1 closure) is satisfied: the trace is recorded above; the warning noise is pre-existing and tracked separately.

## Hotspot

No new hotspot was identified for measurement this cycle. The structural cost growth from T-016 is bounded:

| Hot path | Pre-T-016 | Post-T-016 | Notes |
|----------|-----------|------------|-------|
| `kernel_entry` boot sequence (cold path; runs once) | hello → VBAR_EL1 → GIC init → DAIF unmask → timer banner → boot_ns → demo | hello → VBAR_EL1 → boot_ns → `mmu_bootstrap()` → "mmu activated" → GIC init → DAIF unmask → timer banner → demo | +`mmu_bootstrap()` (~600 ns single-run on QEMU TCG; sub-100 µs on real hardware per ADR-0027). +1 UART line. boot_ns moved earlier so the recorded baseline includes MMU activation cost. |
| `mmu_bootstrap` Step 1 (boot-time only) | n/a | 9 + 64 = 73 `core::ptr::write_volatile` to bootstrap frames + 3 table descriptors | Bounded by constant loop counts; 8-byte stores; total ~590 ns single-run on QEMU TCG. |
| `mmu_bootstrap` Step 2 (boot-time only) | n/a | 4 `MSR` + 1 `ISB` | 5 instructions; sub-microsecond. |
| `mmu_bootstrap` Step 3 (boot-time only) | n/a | TLBI + DSB ISH + IC IALLU + DSB ISH + ISB + SCTLR RMW + ISB | 8 instructions; the SCTLR RMW + final ISB drains the pipeline; the next instruction-fetch goes through the new translation regime. |
| `Mmu::map` / `unmap` (no v1 caller) | n/a | 4-level page-table walk + descriptor encode + `core::ptr::write_volatile` | Constant-bounded by 4 page-table levels; each level is one read + optional write. **Dead-code-allow in v1**; first runtime exercise lands with the future B3+ post-bootstrap caller. |
| `Mmu::invalidate_tlb_address` (no v1 caller) | n/a | DSB ISHST + TLBI VAE1, x + DSB ISH + ISB | 4 instructions; sub-microsecond on Cortex-A72. **Dead-code-allow in v1**; first runtime exercise lands with the future `MapperFlush::flush()` caller. |
| `MapperFlush::flush` consumer | n/a | dispatches to `Mmu::invalidate_tlb_address(va)` | Same as above; the token is `#[must_use]` so the lint catches "forgot to flush" at compile time. |
| MMIO accesses (post-MMU) | passthrough (MMU off) | translated through device-nGnRnE attribute | Sub-cycle architectural difference on real hardware (the device attribute disables speculation + write merging — slightly *faster* on systems where speculation would have been wasteful). On QEMU TCG: zero observable difference (TCG doesn't model the cache hierarchy). |

The `Mmu::map` / `unmap` / `invalidate_tlb_*` paths are **not** hot in v1 (no caller). They become hot when:
- B3 PMM lands and starts mapping pages dynamically.
- B5+ userspace bring-up adds per-task `Mmu::map` for grant operations.
- Future preemption / fault-handling lands and adds `Mmu::invalidate_tlb_address` from the IRQ path.

The 11 performance proposals queued by [Track D's 2026-05-06 paper review](../code-reviews/2026-05-06-full-tree/track-d-performance.md) remain open. P3 (const-eval invariant assertions) was **partially landed by γ at B1 closure** (the `Aarch64TaskContext` size guard + `SchedQueue<0>` assertion + inline `const { assert!(...) }` in `cap/table.rs`) and **further extended by T-016's vmsav8 module** (12 new `const fn` encoder tests pin every bit-field at compile time; the `MAIR_EL1_VALUE` / `TCR_EL1_VALUE` / `SCTLR_EL1_MMU_ENABLE_MASK` constants are `const fn` composed). P10 (multi-run wall-clock benchmark harness) **landed in PR #22** and is the source of this baseline's Source B. P1 (`#[cold]` annotations on error paths) and P4 (`core::hint::assert_unchecked` migration) remain open; they become candidates for explicit hypothesis-driven cycles when B3's PMM bring-up surfaces a measurement target.

## Verdict

**Merge** — re-baseline complete; no proposal to land in this cycle.

The post-T-016 image absorbs the explicit, well-bounded cost of MMU activation: +364 `.text` / +16 `.rodata` / +17,952 `.bss` (the dominant cost being the `.boot_pt` reservation, exactly 16 KiB + 4 KiB alignment slack). The kernel image total is now ~64.2 KiB versus B1 closure's ~46.5 KiB — a +38 % growth that reflects a structural addition (page-table backing storage), not a code-bloat regression.

The release-build harness band (4.262 / 4.642 / 6.456 ms p10 / p50 / p90 across 20 iterations) is the new baseline-of-record. Single-run boot-to-end of 4.891 ms is within this band. The debug-build comparison shows ~1.5 ms p10/p50 cost from MMU activation under QEMU TCG translation-cache overhead; real-hardware activation is sub-100 µs per ADR-0027 §Consequences. **No measurable performance regression** for the v1 cooperative IPC demo; future hypothesis-driven cycles (preemption, IPC throughput, userspace mapping) become measurable against this baseline.

The `MapperFlush` flush-token discipline is the project's first load-bearing `#[must_use]` API contract — pattern-of-record for future HAL trait surfaces where a mutation requires a follow-up step. Cost: zero (the lint catches the bug class at compile time, with no runtime overhead). Win: the bug class "forgot to flush after `Mmu::map`" is now structurally impossible to land.
