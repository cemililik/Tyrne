## Baseline

### Methodology

- ELF section sizes via `rust-readobj --elf-output-style=GNU -S` (workspace nightly toolchain).
- Source LOC via `find ‚Ä¶ -name '*.rs' -o -name '*.s' | xargs wc -l`.

### Metric 1 ‚Äî Kernel image size

| Section | 2026-04-28 (B1 baseline) | 2026-05-07 (post-T-014) | Œî bytes | Œî % |
|---------|--------------------------|--------------------------|---------|-----|
| `.text` | 21,908 | **21,792** (`0x5520`) | **‚àí116** | **‚àí0.5 %** |
| `.rodata` | 2,784 | **2,928** (`0xb70`) | +144 | +5.2 % |
| `.bss` | 22,248 | **22,256** (`0x56f0`) | +8 | +0.04 % |

**Observations:**

- **`.text` is slightly *smaller* despite +192 LOC of source.** The reduction comes from Œ≥'s `kernel_entry` simplification (the rejected defensive `loop {}` would have added ~16 bytes; Œ≥ removed the conditional `add_task(handle_idle, ...)` call and replaced it with `register_idle(...)` which inlines slightly tighter under release-profile LLVM). T-014's `register_idle` body is ~80 bytes; the dispatch chain rewrites in `start_prelude` / `yield_now` / `ipc_recv_and_yield` add ~60 bytes total; the idle-handle storage in `Scheduler<C>` adds 8 bytes. Counterweighted by the removal of `Scheduler::add_task`'s idle-specific call site (~30 bytes), the GIC range-assertion comment cleanups, and LLVM re-inlining decisions across the function-call graph. Net: ‚àí116 bytes.
- **`.rodata` grew by +144 bytes** ‚Äî almost entirely from the new panic message strings (`"register_idle called twice ‚Äî idle slot is set-once by boot-time discipline"`, `"scheduler start called with empty ready queue and no idle task"`, `"SchedQueue requires N > 0"`, the `Aarch64TaskContext` size-guard message, and the comprehensive-review-derived assertion strings). Documentation-quality strings live in `.rodata`; the size growth is the cost of the panic-clarity / build-time-assertion discipline this closure trio formalises.
- **`.bss` grew by +8 bytes** ‚Äî the new `idle: Option<TaskHandle>` field on `Scheduler<C>`. `Option<TaskHandle>` on aarch64 is 8 bytes (TaskHandle is `u32` + 4 bytes of `Option` discriminant rounding, packed into the struct's existing alignment). The field is the *only* new RAM cost from T-014.
- **No new kernel-allocated stack.** ADR-0026 / T-014 reuses the `TASK_IDLE_STACK` allocated by T-007; the dispatch refactor changes *where* the idle TaskHandle is stored, not whether idle has a stack. RAM footprint identical except for the +8 bytes above.

**B0 ‚Üí 2026-04-28 ‚Üí 2026-05-07 cumulative trajectory** (relative to A6 baseline of `.text 13,940` / `.rodata 1,960` / `.bss 17,872`):

| Section | A6 | B1 (2026-04-28) | B1 (2026-05-07) | A6 ‚Üí today total Œî |
|---------|----|-------|-------|--------------------|
| `.text` | 13,940 | 21,908 | 21,792 | +7,852 (+56.3 %) |
| `.rodata` | 1,960 | 2,784 | 2,928 | +968 (+49.4 %) |
| `.bss` | 17,872 | 22,248 | 22,256 | +4,384 (+24.5 %) |

The total kernel image (text + rodata + bss) is now ~46.0 KiB, up from ~32.7 KiB at A6. Within the implicit non-hypothesis budget set by the [2026-04-28 baseline](2026-04-28-B1-closure.md) (~5‚Äì10 KiB `.text` + ~5 KiB `.bss` from B0+B1's structural surface ‚Äî vector table 2 KiB, GIC driver ~1.5 KiB, idle stack 4 KiB).

## Hotspot

No new hotspot was identified for measurement this cycle. The structural cost growth from T-014 is bounded:

| Hot path | Pre-T-014 | Post-T-014 | Notes |
|----------|-----------|------------|-------|
| `start_prelude` dispatch | `s.ready.dequeue()?` | `match s.ready.dequeue() { Some(h) => h, None => match s.idle { Some(h) => h, None => panic!() } }` | +1 conditional branch on the cold path (queue-empty AND idle-None); zero cost on the hot path (queue-non-empty). |
| `yield_now` dispatch | `s.ready.dequeue()` then conditional re-enqueue | Adds `current_is_idle` check + skip re-enqueue if idle | +1 `Option`-equality compare per yield; no allocation, no MMIO. |
| `ipc_recv_and_yield` Phase 2 | `s.ready.dequeue()` | `s.ready.dequeue().or(s.idle)` | +1 `Option::or` call per Phase-2 dispatch. |
| `register_idle` (boot-time) | n/a (was `add_task(idle_entry, ...)`) | `unsafe { &mut *sched }` + `init_context` + `s.idle = Some(handle)` | Boot-time only; runs exactly once; ~80 bytes of code, ~10 instructions of dispatch. |

The dispatch-chain additions are *single-instruction* on the hot path (the `Option::or(Option<T>)` call compiles to a conditional move on aarch64 release builds, not a branch). The boot-time cost of `register_idle` is sub-microsecond at QEMU's emulation rate.

The 11 performance proposals queued by [Track D](../code-reviews/2026-05-06-full-tree/track-d-performance.md) at the comprehensive review remain open; one (P3 ‚Äî const-eval invariant assertions) was already complete when Track D ran, and Œ≥ extended it (the new `Aarch64TaskContext` size guard, the inline `const { assert!(...) }` migration in `cap/table.rs`, the `SchedQueue<0>` assertion). The other ten (P1, P2, P4, P5, P6, P7, P8, P9, P10, P11) remain open. **None are blocked by today's closure.** P1 (`#[cold]` annotations on error paths), P10 (IPC round-trip benchmark harness), and P4 (`core::hint::assert_unchecked` migration) remain the highest-ROI near-term picks; P10 is a precondition for measuring P1 and P4. They become candidates for explicit hypothesis-driven cycles when B2's MMU activation work needs IPC throughput as a measurement target.

## Verdict

**Merge** ‚Äî re-baseline complete; no proposal to land in this cycle.

The post-T-014 image is essentially footprint-neutral relative to the 2026-04-28 baseline. The +144 bytes `.rodata` growth is the cost of the new panic-message clarity and build-time-assertion strings (an explicit value-for-money trade per the closure trio's discipline-codifying tone). The +8 bytes `.bss` is the `Option<TaskHandle>` idle-fallback slot that ADR-0026 / T-014 introduced. The ‚àí116 bytes `.text` reduction is LLVM's re-inlining-after-call-graph-changes plus Œ≥'s small simplifications.

## Post-T-015 amendment (2026-05-07, PR #17)

Append-only addendum after T-015 / ADR-0032 landed on the same day. The original re-baseline above was authored before PR #17's `ipc_cancel_recv` + symmetric Deadlock rollback merged; the numbers below replace the headline `.text` figure for any future regression check.

| Section | post-PR-#16 (`95b15aa`, original re-baseline) | post-PR-#17 (`8dc433e`, T-015 merged) | Œî vs re-baseline | Œî vs 2026-04-28 |
|---------|-----------------------------------------------|---------------------------------------|------------------|------------------|
| `.text` | 21,792 (`0x5520`) | **22,020 (`0x5604`)** | **+228** | **+112** |
| `.rodata` | 2,928 (`0xb70`) | **2,928 (`0xb70`)** | 0 | +144 |
| `.bss` | 22,256 (`0x56f0`) | **22,256 (`0x56f0`)** | 0 | +8 |

**Per-symbol attribution of the +228 `.text` bytes** (release-profile, measured live via `rust-readobj --symbols` at both commits):

| Symbol | post-#16 | post-#17 | Œî |
|--------|----------|----------|---|
| `tyrne_kernel::sched::ipc_recv_and_yield<QemuVirtCpu>` | 740 | 752 | +12 (cold cancel-call arm) |
| `tyrne_kernel::ipc::ipc_cancel_recv` (non-generic) | ‚Äî | 216 | +216 (new function body) |
| **Sum** | ‚Äî | ‚Äî | **+228** |

The bookkeeping closes byte-exactly: +12 + 216 = +228 = section-level delta. Zero LLVM-side reshuffling residual; layout is byte-stable around the change.

**Genericity / monomorphization.** `ipc_cancel_recv` is non-generic (signature: `fn(&mut EndpointArena, &mut IpcQueues, CapHandle, &CapabilityTable) -> Result<(), IpcError>` ‚Äî no `<C: ContextSwitch + Cpu>` parameter). One symbol in the release ELF; no `Scheduler<C>` instantiation chain; no monomorphization fan-out when a second BSP comes online. `ipc_recv_and_yield<QemuVirtCpu>` remains a single instantiation; future `<C2>` BSPs add +12 bytes per instantiation but share the single `ipc_cancel_recv` body.

**`Result` / error surface.** `IpcError` unchanged ‚Äî no new variant, no discriminant size growth, no new `Debug` impl bytes. Cancel reuses `IpcError::InvalidCapability` for the cap-validation failure path. The `let cancel_result = unsafe { ‚Ä¶ }; debug_assert!(cancel_result.is_ok());` shape compiles to a discard in release (no branch on the success path; `debug_assert!` is the entire consumer of the `Result`).

**Coverage trade.** The 216-byte `ipc_cancel_recv` body and the +12-byte cold arm in `ipc_recv_and_yield` are pure forward-looking footprint cost ‚Äî the demo's `register_idle` + cooperative-IPC configuration makes `SchedError::Deadlock` structurally unreachable per ADR-0026, so `ipc_cancel_recv` is never called in v1 production. The path is exercised entirely by host tests + miri (5 IPC `cancel_recv_*` unit tests + 1 sched `ipc_recv_and_yield_deadlock_rolls_back_endpoint_state` regression test + the existing T-007 test's new endpoint-state assertion). The path becomes live as soon as preemption (B5+), multi-waiter (ADR-0019 ¬ßOpen), or userspace endpoint destroy (B2+) lands. The cost is structurally appropriate for v1 closing-out a forward-looking invariant gap; a future P1 (`#[cold]` annotation pass) on the cancel call site + `ipc_cancel_recv`'s body would yield real i$ wins on the success path of `ipc_recv_and_yield<QemuVirtCpu>`.

**The closure trio's "footprint-neutral" framing was correct for the PR #12 ‚Üí PR #15 window.** Post-T-015 it is +228 bytes ‚Äî a real but bounded growth, in the same order of magnitude as a single ADR-0026-style structural addition. The 2026-05-07 [pr-12-to-17 multi-axis review's Track D ¬ßD1](../code-reviews/2026-05-07-pr-12-to-17-multi-axis-review/track-d-perf.md) is the source of this amendment.

**Smoke variance band.** PR #17 reports a single-run boot-to-end of ~4.1 ms ‚Äî at the bottom edge of the [`current.md`](../../../roadmap/current.md) "~4‚Äì6.5 ms typical" band and ~25‚Äì30 % below the closure-trio's lowest recorded run (5.5 ms). Almost certainly host-cache variance (cold-vs-warm host), not a genuine speedup ‚Äî T-015 makes zero changes that could shorten the hot path. Promotes [Track D's P10 proposal (multi-run wall-clock harness)](../code-reviews/2026-05-06-full-tree/track-d-performance.md) from "queued" to "load-bearing before B2 ADR-0027 implementation". Single-run boot-to-end timings should not be over-read until P10 lands.

The 11 P-numbered proposals from Track D's [2026-05-06 paper review](../code-reviews/2026-05-06-full-tree/track-d-performance.md) remain queued (P3 partially landed by Œ≥; P1 / P10 / P4 are the highest-ROI near-term picks and ripe for an explicit cycle once B2's MMU work surfaces a measurement target).
