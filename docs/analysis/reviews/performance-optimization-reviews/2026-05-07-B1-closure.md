## Baseline

### Methodology

- ELF section sizes via `rust-readobj --elf-output-style=GNU -S` (workspace nightly toolchain).
- Source LOC via `find … -name '*.rs' -o -name '*.s' | xargs wc -l`.
- Boot-to-end timing observed across the γ-verification smoke runs (~5.8 ms typical).
- Single-run measurements; no iteration / variance recorded because the build is deterministic.

### Metric 1 — Kernel image size

| Section | 2026-04-28 (B1 baseline) | 2026-05-07 (post-T-014) | Δ bytes | Δ % |
|---------|--------------------------|--------------------------|---------|-----|
| `.text` | 21,908 | **21,792** (`0x5520`) | **−116** | **−0.5 %** |
| `.rodata` | 2,784 | **2,928** (`0xb70`) | +144 | +5.2 % |
| `.bss` | 22,248 | **22,256** (`0x56f0`) | +8 | +0.04 % |

**Observations:**

- **`.text` is slightly *smaller* despite +192 LOC of source.** The reduction comes from γ's `kernel_entry` simplification (the rejected defensive `loop {}` would have added ~16 bytes; γ removed the conditional `add_task(handle_idle, ...)` call and replaced it with `register_idle(...)` which inlines slightly tighter under release-profile LLVM). T-014's `register_idle` body is ~80 bytes; the dispatch chain rewrites in `start_prelude` / `yield_now` / `ipc_recv_and_yield` add ~60 bytes total; the idle-handle storage in `Scheduler<C>` adds 8 bytes. Counterweighted by the removal of `Scheduler::add_task`'s idle-specific call site (~30 bytes), the GIC range-assertion comment cleanups, and LLVM re-inlining decisions across the function-call graph. Net: −116 bytes.
- **`.rodata` grew by +144 bytes** — almost entirely from the new panic message strings (`"register_idle called twice — idle slot is set-once by boot-time discipline"`, `"scheduler start called with empty ready queue and no idle task"`, `"SchedQueue requires N > 0"`, the `Aarch64TaskContext` size-guard message, and the comprehensive-review-derived assertion strings). Documentation-quality strings live in `.rodata`; the size growth is the cost of the panic-clarity / build-time-assertion discipline this closure trio formalises.
- **`.bss` grew by +8 bytes** — the new `idle: Option<TaskHandle>` field on `Scheduler<C>`. `Option<TaskHandle>` on aarch64 is 8 bytes (TaskHandle is `u32` + 4 bytes of `Option` discriminant rounding, packed into the struct's existing alignment). The field is the *only* new RAM cost from T-014.
- **No new kernel-allocated stack.** ADR-0026 / T-014 reuses the `TASK_IDLE_STACK` allocated by T-007; the dispatch refactor changes *where* the idle TaskHandle is stored, not whether idle has a stack. RAM footprint identical except for the +8 bytes above.

**B0 → 2026-04-28 → 2026-05-07 cumulative trajectory** (relative to A6 baseline of `.text 13,940` / `.rodata 1,960` / `.bss 17,872`):

| Section | A6 | B1 (2026-04-28) | B1 (2026-05-07) | A6 → today total Δ |
|---------|----|-------|-------|--------------------|
| `.text` | 13,940 | 21,908 | 21,792 | +7,852 (+56.3 %) |
| `.rodata` | 1,960 | 2,784 | 2,928 | +968 (+49.4 %) |
| `.bss` | 17,872 | 22,248 | 22,256 | +4,384 (+24.5 %) |

The total kernel image (text + rodata + bss) is now ~46.0 KiB, up from ~32.7 KiB at A6. Within the implicit non-hypothesis budget set by the [2026-04-28 baseline](2026-04-28-B1-closure.md) (~5–10 KiB `.text` + ~5 KiB `.bss` from B0+B1's structural surface — vector table 2 KiB, GIC driver ~1.5 KiB, idle stack 4 KiB).

| Run | `boot-to-end elapsed` | Notes |
|-----|----------------------|-------|
| 2026-05-06 (post-T-014 first verification) | 6,275,008 ns (~6.3 ms) | First post-T-014 smoke; included in T-014 review-history. |

The figures reflect *boot-to-Task-A-tail*, not steady-state throughput. The dispatch chain for the demo's full A↔B IPC handshake completes within ~6 ms on QEMU virt with `-cpu cortex-a72`; this is order-of-magnitude consistent with the [A6 baseline](2026-04-21-A6-baseline.md)'s "boot finishes in ~5–10 ms on QEMU" framing. **No performance regression observable.**

The `-d int,unimp,guest_errors` window is empty for the entire run — zero exceptions, zero unimplemented-instruction trapses, zero guest errors. The 2026-05-06 morning's smoke (pre-T-014) had hung silently in `WFI`; the post-T-014 traces complete the demo cleanly. Functional regression: closed.

### Methodology notes

Single-core, single-run figures only. The QEMU host-clock gives a ~10–15 % variance on boot-to-end timing across cold-vs-warm host caches; future cycles that need finer numbers should adopt the P10 proposal (IPC round-trip benchmark harness with iteration + variance, gated by a `bench` feature flag). For a closure-trio re-baseline, single-run is sufficient — order-of-magnitude is the unit of interest.

## Hotspot

No new hotspot was identified for measurement this cycle. The structural cost growth from T-014 is bounded:

| Hot path | Pre-T-014 | Post-T-014 | Notes |
|----------|-----------|------------|-------|
| `start_prelude` dispatch | `s.ready.dequeue()?` | `match s.ready.dequeue() { Some(h) => h, None => match s.idle { Some(h) => h, None => panic!() } }` | +1 conditional branch on the cold path (queue-empty AND idle-None); zero cost on the hot path (queue-non-empty). |
| `yield_now` dispatch | `s.ready.dequeue()` then conditional re-enqueue | Adds `current_is_idle` check + skip re-enqueue if idle | +1 `Option`-equality compare per yield; no allocation, no MMIO. |
| `ipc_recv_and_yield` Phase 2 | `s.ready.dequeue()` | `s.ready.dequeue().or(s.idle)` | +1 `Option::or` call per Phase-2 dispatch. |
| `register_idle` (boot-time) | n/a (was `add_task(idle_entry, ...)`) | `unsafe { &mut *sched }` + `init_context` + `s.idle = Some(handle)` | Boot-time only; runs exactly once; ~80 bytes of code, ~10 instructions of dispatch. |

The dispatch-chain additions are *single-instruction* on the hot path (the `Option::or(Option<T>)` call compiles to a conditional move on aarch64 release builds, not a branch). The boot-time cost of `register_idle` is sub-microsecond at QEMU's emulation rate.

The 11 performance proposals queued by [Track D](../code-reviews/2026-05-06-full-tree/track-d-performance.md) at the comprehensive review remain open; one (P3 — const-eval invariant assertions) was already complete when Track D ran, and γ extended it (the new `Aarch64TaskContext` size guard, the inline `const { assert!(...) }` migration in `cap/table.rs`, the `SchedQueue<0>` assertion). The other ten (P1, P2, P4, P5, P6, P7, P8, P9, P10, P11) remain open. **None are blocked by today's closure.** P1 (`#[cold]` annotations on error paths), P10 (IPC round-trip benchmark harness), and P4 (`core::hint::assert_unchecked` migration) remain the highest-ROI near-term picks; P10 is a precondition for measuring P1 and P4. They become candidates for explicit hypothesis-driven cycles when B2's MMU activation work needs IPC throughput as a measurement target.

## Verdict

**Merge** — re-baseline complete; no proposal to land in this cycle.

The post-T-014 image is essentially footprint-neutral relative to the 2026-04-28 baseline. The +144 bytes `.rodata` growth is the cost of the new panic-message clarity and build-time-assertion strings (an explicit value-for-money trade per the closure trio's discipline-codifying tone). The +8 bytes `.bss` is the `Option<TaskHandle>` idle-fallback slot that ADR-0026 / T-014 introduced. The −116 bytes `.text` reduction is LLVM's re-inlining-after-call-graph-changes plus γ's small simplifications.

The 11 P-numbered proposals from Track D's [2026-05-06 paper review](../code-reviews/2026-05-06-full-tree/track-d-performance.md) remain queued (P3 partially landed by γ; P1 / P10 / P4 are the highest-ROI near-term picks and ripe for an explicit cycle once B2's MMU work surfaces a measurement target).
