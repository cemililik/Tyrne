## User story

As the Tyrne kernel, I want a **task loader** that consumes an embedded raw-flat userspace binary (per [ADR-0029](../../../decisions/0029-initial-userspace-image-format.md)), allocates physical frames via [`Pmm`](../../../../kernel/src/mm/pmm.rs), creates a fresh [`AddressSpace`](../../../../kernel/src/mm/address_space.rs) via [`cap_create_address_space`](../../../../kernel/src/mm/address_space.rs), copies the binary's bytes into the new frames, maps them into the new AS via [`cap_map`](../../../../kernel/src/mm/address_space.rs) with `USER | EXECUTE` flags, sets up a userspace stack region under `USER | WRITE` flags, and produces a `LoadedImage { as_cap, entry_va, stack_top_va, image_bytes, stack_bytes }` metadata struct that **describes** the loaded address space — so that B5's syscall ABI (ADR-0030) and B6's first userspace "hello" can compose `LoadedImage` with the still-to-be-built `task_create_from_image` wrapper to produce a runnable `CapHandle{CapObject::Task(TaskHandle)}` capable of executing at EL0.

## Context

[Phase B §B4](../../../roadmap/phases/phase-b.md#milestone-b4--task-loader) defines the loader as the first runtime consumer of the AddressSpace + PMM scaffolds that B3 landed. [ADR-0029](../../../decisions/0029-initial-userspace-image-format.md) (lands with this task at Draft, Accepts before code) settles the byte-level format the loader walks — a raw flat binary embedded into the kernel image via `include_bytes!`. The loader's job is to turn that byte stream into a *populated address space*: a fresh AS with the image's pages mapped under `USER | EXECUTE`, a stack mapped under `USER | WRITE`, and a `LoadedImage` metadata struct describing it.

**Scope boundary (load-complete, not B5/B6-runnable).** T-019 does **not** mint a `CapHandle{CapObject::Task(TaskHandle)}`. The current [`Task` struct](../../../../kernel/src/obj/task.rs) carries only `id: u32` + `address_space_handle: AddressSpaceHandle` — there is **no** PC/SP/context register file on it, and `ContextSwitch::init_context` requires both a stable entry-point function pointer and a kernel-stack-backed `stack_top: *mut u8` that the userspace task does not have until B5+B6 introduce EL0 entry. Beyond the context shape, the loader's new address space contains **only** the image + stack mappings — no kernel mappings. The exception-vector path lives in kernel `.text`; an EL1 exception taken while the userspace AS is active would translation-fault on the vector fetch. Per [ADR-0027 §Decision outcome (a)](../../../decisions/0027-kernel-virtual-memory-layout.md), the kernel-in-userspace-AS problem is the [ADR-0033 high-half migration placeholder](../../../decisions/0027-kernel-virtual-memory-layout.md)'s responsibility, gated on B5 surfacing per-task `TTBR0_EL1` swap. T-019's `LoadedImage` is therefore explicitly a **descriptor of loaded state**, not a runnable-task capability. The `task_create_from_image` wrapper named in [phase-b §B4 §3](../../../roadmap/phases/phase-b.md#milestone-b4--task-loader) lands with B5/B6 after those prerequisites; the rider added to phase-b §B4 in the ADR-0029 propose commit records the split.

**Leak-path closure inherits T-018's discipline, extended for cap_map's intermediate-frame consumption.** T-018's `cap_create_address_space` solved the *creation-side* PMM-leak path via preflight chain. T-019 inherits that property at the `cap_create_address_space` call but must also handle the *additional* leak surface that `cap_map` introduces: each `cap_map` call may allocate up to **three** intermediate page-table frames (L1, L2, L3) via the [BSP's `walk_or_alloc_table`](../../../../bsp-qemu-virt/src/mmu.rs). For a small image (e.g., 4 pages) at a contiguous VA range, the worst case is +3 intermediates (one for each level not already present in the new AS). For an image + stack at *different* VA ranges (the common case — image at low userspace VA, stack at high), the worst case roughly doubles. T-019's preflight must budget the worst-case intermediate frame count *before* the first PMM commitment. The **explicit rollback contract** (which API reverses which commitment, what resources leak in the v1 baseline, why `cap_drop` is the correct cap-side cleanup API instead of `cap_revoke`, and which B5+ ADR formalises the full reclaim path) is documented canonically in §Approach §"Rollback contract (explicit)" below; the §Acceptance criteria `MapFailed` and `OutOfFrames` bullets cite that section's discipline rather than restating it.

The work touches three subsystems already in tree (PMM, AddressSpace, exception-vector path), one new BSP-side glue layer (the embedded blob + the userspace linker layout it implicitly assumes), and zero new ADRs beyond ADR-0029. The complexity sits in **the sequence + the rollback**: preflight every rejectable check → PMM commit (with rollback machinery armed) → cap_create_address_space (delegates to T-018's preflight discipline) → cap_map loop (with the intermediate-frame leak path documented) → return `LoadedImage`. Every step is fallible; the explicit rollback contract (below) names which commits roll back via which API on which error variant.

## Acceptance criteria

A checklist of items that must be true for the task to move from `In Review` to `Done`.

- [ ] **ADR-0029 Accepted** before code lands. Same-day Accept after careful re-read is permitted per [ADR-0025 §Revision notes](../../../decisions/0025-adr-governance-amendments.md); Propose commit is separate from the Accept commit per [`write-adr` skill §10](../../../../.agents/skills/write-adr/SKILL.md).
- [ ] **`pub fn load_image<M: Mmu>(image: &[u8], pmm: &mut Pmm<...>, mmu: &M, table: &mut CapabilityTable, as_arena: &mut AddressSpaceArena<M>, parent_as_cap: CapHandle, new_rights: CapRights, image_base_va: VirtAddr, stack_size_pages: usize) -> Result<LoadedImage, LoadError>`** lands in `kernel/src/obj/task_loader.rs` (new module — Approach decides whether to nest under `obj/` or `mm/`). The signature explicitly takes `&mut Pmm<...>` (**not** `&mut dyn FrameProvider`) because the rollback path requires `Pmm::free_frame` which is not on the trait surface. `parent_as_cap` authorises minting a new child AS via `cap_create_address_space`. **DERIVE rights enforcement** happens inside `cap_create_address_space`'s step 2a (see [`kernel/src/mm/address_space.rs`](../../../../kernel/src/mm/address_space.rs) — `if !parent_cap.rights().contains(CapRights::DERIVE) { return Err(InsufficientRights); }`); T-019's pre-step does a lookup + `CapKind::AddressSpace` sanity check only (the `resolve_address_space_cap` helper at [`kernel/src/mm/address_space.rs`](../../../../kernel/src/mm/address_space.rs) is kind-only by design per its v1 rights-model doc-comment), and lets the DERIVE check surface via `AddressSpaceCreationFailed(CapError(InsufficientRights))` rather than duplicating the check. `new_rights` is the rights set the new AS cap carries. `image_base_va` is the userspace-linker-defined base VA the image is laid out at; `stack_size_pages` is the stack region size in 4 KiB pages (minimum 1 page = 4 KiB ≥ the [`ContextSwitch::init_context`](../../../../hal/src/cpu.rs) trait's 512-byte stack-backing requirement; 16-byte AAPCS64 alignment is satisfied by construction since `stack_top_va = image_base_va + image_pages * PAGE_SIZE + stack_size_pages * PAGE_SIZE` is `PAGE_SIZE`-aligned ⇒ 16-byte aligned).
- [ ] **`pub struct LoadedImage { pub as_cap: CapHandle, pub entry_va: VirtAddr, pub stack_top_va: VirtAddr, pub image_bytes: usize, pub stack_bytes: usize }`** is the loader's success return type. `as_cap` is the `CapHandle` for the newly-minted AS (wraps `CapObject::AddressSpace(handle)` internally); `entry_va` equals `image_base_va` (raw flat format: offset 0 ↔ entry point); `stack_top_va` is **one-past-the-highest mapped VA** of the stack region — i.e., the first VA above the stack that is NOT mapped. The stack mapped range is `[stack_base, stack_top_va)` half-open; `sp = stack_top_va` at task-creation initialisation is correct because the first userspace push decrements `sp` (e.g. `sp -= 16`) which lands inside the mapped range. `image_bytes` is the byte-count of the image as loaded into the AS (may be smaller than `image_pages * PAGE_SIZE` because tail-zeroing happens on the partial last page).
- [ ] **`pub enum LoadError`** with **explicit rollback-discharging** variants:
  - `InvalidImage` — `image.is_empty()` returns this **before** any state change (no rollback needed).
  - `InvalidStackSize` — `stack_size_pages == 0` returns this before any state change.
  - `InvalidParentCap(CapError)` — `parent_as_cap` lookup / kind check failure (DERIVE check is delegated to `cap_create_address_space`; see above and `AddressSpaceCreationFailed` below); wraps [`CapError`](../../../../kernel/src/cap/mod.rs). Pre-PMM preflight; no rollback needed.
  - `FrameBudgetExceeded { needed: usize, available: usize }` — total safe-upper-bound frame budget (1 root AS frame + image pages + stack pages + worst-case intermediates) exceeds `Pmm::stats().free_frames`. Returns **before** any state change; no rollback needed.
  - `InvalidImageBaseVa { base: VirtAddr, end: VirtAddr }` — the requested VA span `[image_base_va, stack_top_va)` (half-open) crosses or exceeds `USERSPACE_VA_LIMIT = 1 << 48` (the architected 48-bit `TTBR0_EL1` range per [ADR-0027 §Decision outcome (a)](../../../decisions/0027-kernel-virtual-memory-layout.md)'s `TCR_EL1.T0SZ = 16`). The variant fires either when `image_base_va` itself is past the limit, when the saturated-add of the span end exceeds it, or when the byte-count multiplication saturates to `usize::MAX` (the documented sentinel for the overflow path). Returns **before** any state change; no rollback needed.
  - `ImageOverlapsAllocatableMemory` — the `image` slice's PA range overlaps a frame `Pmm::alloc_frame()` could yield. Discharges [UNSAFE-2026-0027](../../../audits/unsafe-log.md)'s "source and destination do not overlap" invariant at runtime; practically unreachable under correct BSP wiring (`.rodata`-resident images live in PMM-reserved memory by ADR-0035), but defensive against BSP misconfiguration. Returns **before** any state change; no rollback needed.
  - `AddressSpaceCreationFailed(AddressSpaceError)` — `cap_create_address_space` failure (covers `InsufficientRights` if `parent_as_cap` lacks DERIVE, plus the structurally-unreachable-in-v1 `CapsExhausted` / `DerivationTooDeep` / `ArenaFull` paths). T-018's preflight discipline guarantees no PMM/arena state was committed on failure; no rollback needed at this layer.
  - `OutOfFrames` — direct `Pmm::alloc_frame()` returned `None` mid-loop (image or stack page allocation). Should be structurally unreachable post-FrameBudgetExceeded preflight under v1's single-threaded model, but kept as a defensive variant for budget-calculation bugs and future-concurrency scenarios. **Rollback required:** same shape as `MapFailed` below — undo allocated frames + committed mappings + revoke AS cap.
  - `MapFailed(AddressSpaceError)` — `cap_map` failure (typically `MmuMapError(MmuError::OutOfFrames)` if the intermediate-frame budget was underestimated, `MmuMapError(MmuError::AlreadyMapped)` if the VA range collides with an existing mapping, or `MmuMapError(MmuError::BlockMapped)` if the VA falls inside a block descriptor). **Rollback required:** leaf frames + partial mappings undone, AS cap invalidated via `cap_drop(loaded_as_cap)`; canonical contract + the cap_drop-vs-cap_revoke + v1 leak baseline + B5+ reclaim ADR pointer in §Approach §"Rollback contract (explicit)" below.
- [ ] **Frame budget preflight is a safe upper bound, not exact.** The check is `1 + image_pages + stack_pages + intermediate_budget <= pmm.stats().free_frames`, where the leading `1` accounts for the root L0 frame that `cap_create_address_space` will allocate. `intermediate_budget = 6` for v1's fresh-AS scenario (worst case: 3 intermediate frames per contiguous VA range × 2 ranges for image + stack). A future exact calculation per VA decomposition (e.g., image + stack sharing an L1 if their VA ranges are close enough to land in the same 1 GiB block) is a possible refinement, but the v1 upper-bound of 6 over-allocates at most ~24 KiB of frame headroom under any realistic image/stack VA placement — well within the 128 MiB extent's budget. Documented inline in `load_image`'s rustdoc.
- [ ] **Image page-rounding + tail zeroing.** `image_pages = (image.len() + PAGE_SIZE - 1) / PAGE_SIZE` (ceiling division). The last image page contains `image.len() % PAGE_SIZE` payload bytes followed by zero-fill to the page boundary (the zero-fill is automatic per UNSAFE-2026-0026's PMM contract — `alloc_frame` zero-initialises, then the loader's byte-copy overwrites only the first `image.len()` bytes; the remainder of the last page stays zero).
- [ ] **Mapping flags fixed:** image region maps with `MappingFlags::USER | MappingFlags::EXECUTE`; stack region maps with `MappingFlags::USER | MappingFlags::WRITE`. Per-section flags (`.text` RX-only, `.rodata` R-only, `.data` RW-only, NX on non-text) deferred to ADR-0034 (placeholder; B5+ trigger).
- [ ] **`LoadError` covers every error path; no `panic!` / `unwrap` / `expect` on the kernel-reachable path.** `cargo kernel-clippy -D warnings` clean (the kernel crate's `#![deny(clippy::panic)]` discipline enforces this).
- [ ] **The loader produces a `LoadedImage` but does NOT mint a `CapHandle{CapObject::Task(...)}`.** Per §Context's scope boundary, runnability gates on B5/B6. The smoke trace shows the loader returning a `LoadedImage` + a new banner line (e.g. `tyrne: image loaded (entry = 0x..., sp = 0x...; image bytes N, stack bytes M, AS cap = <handle>)`) inserted in a stable position in the boot sequence. Full demo through `tyrne: all tasks complete` still passes; `-d int,unimp,guest_errors` reports no new event classes beyond the pre-existing PL011-disabled-UART noise.
- [ ] **§Approach §Simulation table — every row mapped to a verification artefact** per the [`write-adr` skill §Procedure step 5 — "Decision outcome → Simulation row-to-verification mapping"](../../../../.agents/skills/write-adr/SKILL.md) discipline codified in commit `3ec94b0`.
- [ ] **`unsafe` discipline for the frame byte-copy site — new audit entry required.** The loader copies `image[i*PAGE_SIZE..]` into the freshly-allocated frame via `core::ptr::copy_nonoverlapping(src_ptr, frame_pa as *mut u8, copy_len)` (NOT `write_volatile` — the target is plain RAM, not a memory-ordered page-table descriptor; volatile is unnecessary and obscures the operation). The `unsafe` block needs **its own** audit entry — UNSAFE-2026-0025's scope is page-table descriptor writes (specific volatile + ordering semantics), which is a different safety argument from "kernel writes bytes into a freshly-PMM-allocated, identity-mapped, exclusively-owned RAM frame". UNSAFE-2026-0026's scope is the PMM's zero-fill at `alloc_frame` (also distinct — same site shape but different operation: zero-fill vs byte-copy). The expected shape is a new entry like **UNSAFE-2026-0027 — `task_loader::load_image` frame byte-copy via `core::ptr::copy_nonoverlapping`** under the Operation / Invariants / Rejected-alternatives discipline, covering: (a) the raw-pointer construction `frame_pa as *mut u8` (PMM's contract guarantees alignment + exclusive ownership + identity-mapping in v1's kernel AS); (b) the `copy_nonoverlapping` itself (source slice has at least `copy_len` bytes from the image bounds check; destination is exactly `copy_len` bytes from the PAGE_SIZE-aligned frame; non-overlapping is satisfied because source is `.rodata` and destination is freshly-allocated PMM RAM); (c) why `copy_nonoverlapping` was chosen over `write_volatile` (RAM-to-RAM byte copy needs no memory ordering; volatile would falsely imply MMIO semantics). The audit-entry shape — whether to extend UNSAFE-2026-0026 via Amendment or open UNSAFE-2026-0027 standalone — is decided at first implementation commit per the [`justify-unsafe`](../../../../.agents/skills/justify-unsafe/SKILL.md) skill's audit-tag scoping discipline.

## Out of scope

- **Running the userspace task.** B5/B6's job. T-019 produces a `LoadedImage` descriptor; the `task_create_from_image` wrapper that turns it into a runnable `CapHandle{CapObject::Task(...)}` requires (a) kernel mappings in the userspace AS — gated on ADR-0033 high-half migration or an interim shared-kernel-mapping shape; (b) EL0-ready context register file — gated on ADR-0030 syscall ABI defining the userspace ↔ kernel register convention; (c) syscall entry path — also ADR-0030 / ADR-0031. The split is recorded in [phase-b §B4 §Revision-notes](../../../roadmap/phases/phase-b.md#milestone-b4--task-loader).
- **Per-section permissions.** Image region is `USER | EXECUTE` globally; stack is `USER | WRITE` globally. Per-section RX/.text + R/.rodata + RW/.data discipline is ADR-0034's job (placeholder; B5+ trigger).
- **Symbols / debug data.** Raw flat carries none. ADR-0029 §Negative consequences accepts this v1 cost.
- **Dynamic loading / filesystem.** The image is `include_bytes!`-embedded at compile time. Filesystem-loaded modules are Phase C / D work.
- **`MemoryRegionCap` for frame ownership.** B5+ ADR per [B3 closure retro §Adjustments](../../reviews/business-reviews/2026-05-14-B3-closure.md#adjustments). T-019 holds frames informally through the AS cap; revocation cascades correctly (T-018 commit 3 wired this) but per-frame ownership tracking arrives with the future MemoryRegionCap ADR.
- **A real `hello` userspace binary.** B6's `userland/hello/` crate produces the real binary; T-019 ships with a *placeholder* embedded blob (a hand-written sequence of bytes — `[0x40, 0x05, 0x80, 0x52, 0xc0, 0x03, 0x5f, 0xd6]` = LE word 0 `0x52800540` (`MOVZ w0, #42`) + LE word 1 `0xd65f03c0` (`RET`) — sufficient to verify the loader-produced mapping behaves correctly; **not** executed in B4).
- **Stack guard page.** v1's stack region is a contiguous mapped range with no unmapped guard page below the lowest stack VA. Stack overflow in B5+ userspace will silently underflow into adjacent VAs. Stack guard page is a B5+ hardening task (no slot reserved today).
- **Intermediate page-table frame reclaim on rollback.** v1 leaks the L1/L2/L3 frames orphaned by `cap_unmap` (per §Context's leak-path-closure paragraph). The future per-AS destroy path (B5+) reclaims them.

## Approach

To be filled out during implementation; this Draft sketches the load-bearing decisions only.

### Module placement

Candidate: a new `kernel/src/obj/task_loader.rs` module that owns the `LoadedImage` struct + the `load_image` function + the `LoadError` enum. Alternative: nest under `kernel/src/mm/` since the bulk of the work is memory-management (PMM + AS). Decision at the first implementation commit; the leaning is `obj/` because the produced `LoadedImage` *informs* a future `obj::task::create` invocation more than it informs further `mm::` work.

### Userspace linker layout

Hardcoded to a fixed VA per [ADR-0027 §Decision outcome (a)](../../../decisions/0027-kernel-virtual-memory-layout.md)'s `TTBR0_EL1` range. Candidate base: `0x0000_0000_0080_0000` (8 MiB from the start of the userspace VA range; matches the layout shape `bsp-qemu-virt/linker.ld` uses for the kernel image so the symmetry is easy to recognise). Stack VA range: `[stack_base, stack_top_va)` where `stack_base = image_base_va + image_pages * PAGE_SIZE` and `stack_top_va = stack_base + stack_size_pages * PAGE_SIZE`. The first userspace crate (lands with B6's `userland/hello/`) carries a linker script that places its `_start` symbol at `image_base_va`; for T-019's v1 hand-coded test blob, the entry-point convention is just "first byte of the blob".

### Embedded image content

For v1 / Draft: a hand-coded 8-byte aarch64 sequence `0x40 0x05 0x80 0x52 0xc0 0x03 0x5f 0xd6` (`mov w0, #42; ret` — LE word 0 = `0x52800540` is the `MOVZ` encoding for `Wd=w0`, `imm16=0x002a`, `hw=00`, `sf=0`; LE word 1 = `0xd65f03c0` is `RET`). Stored as a `static USERSPACE_IMAGE: &[u8] = include_bytes!(...)` or inlined as a `&[u8]` literal in the BSP wiring (which form depends on whether `include_bytes!` is wired up at this point — if not, an inline literal is acceptable as a Draft). **Not** executed in B4 — the loader produces the LoadedImage describing the populated AS; the running-the-task work is B5/B6.

### Error handling

`LoadError` enum + `cargo kernel-clippy`'s `#![deny(clippy::panic)]` discipline. Every fallible step propagates a typed error; no `unwrap` / `expect` on the kernel-reachable path. Mirrors [T-018's `AddressSpaceError`](../../../../kernel/src/mm/address_space.rs) shape. The `MapFailed` variant carries the underlying `AddressSpaceError` (which itself wraps `MmuError`) so callers can distinguish PMM exhaustion-mid-loop (`OutOfFrames` — implies the intermediate-frame budget was wrong) from VA collision (`AlreadyMapped` / `BlockMapped` — implies the caller asked for a VA range that overlaps an existing mapping).

### Rollback contract (explicit)

Listed by error variant for unambiguous reviewer audit. **Critical scope note:** cap-side cleanup uses `cap_drop(loaded_as_cap)`, **not** `cap_revoke`. In v1 `cap_revoke(src)` walks `src`'s *descendants* and `free_slot`s them; it does **not** free `src` itself and additionally requires `CapRights::REVOKE` on `src` (which the caller-supplied `new_rights` may omit). The freshly-minted AS cap is a leaf with no descendants by construction (the loader doesn't derive from it), so the correct rollback API is `cap_drop` — it `free_slot`s the leaf cap-table slot (generation bump + `entry = None`), requires only `!has_children` (satisfied), and is rights-agnostic. **Even with the right API, this is still cap-side only:** `cap_drop` does **not** free the AS arena slot, the L0 root frame, or any intermediate L1/L2/L3 page-table frames inside the AS. Those resources **leak** in the v1 baseline. Full reclaim arrives with the future `MemoryRegionCap` + per-AS destroy ADR (B5+ per [B3 closure §Adjustments](../../reviews/business-reviews/2026-05-14-B3-closure.md#adjustments)). The rollback actions below reverse what is reversible; the documented leaks are the v1 trade-off.

| Error | When raised | Rollback action | What leaks in v1 baseline |
|-------|-------------|-----------------|----------------------------|
| `InvalidImage` / `InvalidStackSize` | Argument preflight (step 1) | None — no state changed yet. | Nothing. |
| `InvalidParentCap(_)` | Cap preflight (step 2) | None — no state changed. | Nothing. |
| `FrameBudgetExceeded` | Frame-budget preflight (step 3) | None — no state changed. | Nothing. |
| `ImageOverlapsAllocatableMemory` | Image-PA-overlap preflight (step 4) | None — no state changed. | Nothing. |
| `AddressSpaceCreationFailed(_)` | `cap_create_address_space` returned `Err` (step 5) | None — T-018's preflight ensures no PMM or arena state was committed. | Nothing. |
| `OutOfFrames` from image or stack loop | `pmm.alloc_frame()` returned `None` mid-loop (defensive — should be unreachable post-FrameBudgetExceeded preflight) | (a) `pmm.free_frame` every leaf frame allocated by the current loop iteration backward; (b) `cap_unmap` every page already `cap_map`-committed, then `pmm.free_frame` the returned PA; (c) `table.cap_drop(loaded_as_cap)` to invalidate the AS cap (leaf cap with no descendants → `cap_drop` is the correct API; not `cap_revoke` which walks descendants and requires `CapRights::REVOKE`). | **Leaks:** the root L0 frame; intermediate L1/L2/L3 page-table frames inside the orphaned AS; the AS arena slot itself. The orphaned-AS persists in `AddressSpaceArena` until B5+ destroy path. |
| `MapFailed(_)` from image loop (step 6) | `cap_map` returned `Err` mid-image-loop | Same rollback shape as `OutOfFrames` above (leaf frames + cap_unmap + `cap_drop(loaded_as_cap)`). | Same leaks as above. |
| `MapFailed(_)` from stack loop (step 7) | `cap_map` returned `Err` mid-stack-loop | Same shape as `MapFailed` image-loop rollback, **plus** image-loop's mappings (which were committed successfully) are also unmapped + freed before `cap_drop(loaded_as_cap)`. | Same leaks as above. |

**Why the leak is acceptable as a v1 baseline:** the rollback path is exercised only on PMM exhaustion (rare in v1 with 32 K frames and a single small image+stack budget) or on a misconfigured `image_base_va` that collides with an existing mapping (a kernel-discipline bug, not a userspace-reachable path). The leak surface is bounded — at most one orphaned AS per failure event — and the v1 demo does not retry-loop on failure. Future-state retries (B5+ when userspace can fail-and-retry an exec) MUST be paired with the reclaim ADR before the retry loop lands; T-019's `MapFailed`/`OutOfFrames` paths today should be treated as **fail-stop** at the caller level until then.
