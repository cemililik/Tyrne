## User story

As the Tyrne kernel, I want a typed `AddressSpace<M>` kernel object that wraps the BSP-specific `<M as Mmu>::AddressSpace` value, lives in an arena alongside `Task` / `Endpoint` / `Notification`, is reachable through a `CapKind::AddressSpace` capability, and gets activated on context switch when the outgoing and incoming tasks have different `address_space_handle`s — so that B5+ userspace tasks can each run in their own translation context without relinking the kernel and without violating the capability-gated discipline.

## Context

[ADR-0028](../../../decisions/0028-address-space-data-structure.md) settles the design: **Option A — Generic `AddressSpace<M: Mmu>` wrapping `M::AddressSpace` inline; per-type `AddressSpaceArena<M>`.** The kernel object holds the BSP-specific page-table topology directly as a field, mirrors the [ADR-0016](../../../decisions/0016-kernel-object-storage.md) per-type fixed-size-block arena pattern, propagates the existing `M: Mmu` generic axis from [ADR-0019](../../../decisions/0019-scheduler-shape.md) / [ADR-0020](../../../decisions/0020-cpu-trait-v2-context-switch.md), and reuses the [`MapperFlush`](../../../../hal/src/mmu/mod.rs) token discipline from [ADR-0027](../../../decisions/0027-kernel-virtual-memory-layout.md). The decision drivers (HAL trait stability, zero-cost activation hook, forward-compat to [ADR-0033 placeholder](../../../decisions/0027-kernel-virtual-memory-layout.md#decision-outcome), capability-system integration cost, audit-discipline minimisation) are recorded in the ADR.

T-018 is the implementation of those decisions per the ADR's §Dependency chain (steps 1–7): a single bundled task covering the new `kernel/src/mm/address_space.rs` module + `AddressSpaceArena<M>` + capability-variant landing (`CapKind::AddressSpace` + `CapObject::AddressSpace(AddressSpaceHandle)`) + cap-gated wrapper surface (`cap_create_address_space` / `cap_map` / `cap_unmap`) + scheduler activation hook (in `yield_now`) + BSP wiring (bootstrap-AS wrap + arena `StaticCell` publication + `Task.address_space_handle` field). The task's shape mirrors [T-017](T-017-physical-memory-manager.md) (which bundled PMM module + BSP wiring + audit-log entry + cross-references in one task); the implementation lands across several bisectable commits within the same task.

T-018 is also the **first runtime exerciser of UNSAFE-2026-0026** (PMM frame-zeroing zero-fill block in `Pmm::alloc_frame` — the path was reached at construction in T-017 but the alloc-fast-path was not executed by the v1 demo because the cooperative IPC demo rides the bootstrap mappings). The `cap_create_address_space` wrapper's `PMM.alloc_frame()` call for the new root frame is what lifts UNSAFE-2026-0026's `Pending QEMU smoke verification` note. T-018 is also the **first post-bootstrap caller of UNSAFE-2026-0025** (`Mmu::map` page-table walker) — the cap-gated map path lifts the corresponding `Pending` note on UNSAFE-2026-0025 (per [ADR-0027 §Decision outcome](../../../decisions/0027-kernel-virtual-memory-layout.md#decision-outcome)).

### Kernel-object module (`kernel/src/mm/address_space.rs`)

- [x] **New file `kernel/src/mm/address_space.rs`** under the existing T-017 `kernel/src/mm/` module. `kernel/src/mm/mod.rs` gains a `pub mod address_space;` declaration and re-exports the public surface (`AddressSpace`, `AddressSpaceHandle`, `AddressSpaceArena`, `AddressSpaceError`, `cap_create_address_space`, `cap_map`, `cap_unmap`).
- [x] **`AddressSpace<M: Mmu>` struct.** Fields:
  - `inner: M::AddressSpace` — the BSP-specific page-table topology value (constructed by `Mmu::create_address_space` or, for slot 0, by `wrap_bootstrap`).
  - `generation: Generation` — capability staleness tag, mirrors the [ADR-0016](../../../decisions/0016-kernel-object-storage.md) pattern in `EndpointArena` / `TaskArena`.
  - **No** forward-compat fields in v1 (no ASID, no reverse-mapping pointers, no per-AS table-walk bookkeeping). Per [ADR-0028 §Decision outcome](../../../decisions/0028-address-space-data-structure.md#decision-outcome) "Adding them speculatively would violate CLAUDE.md non-negotiable #6". When ADR-0033 (high-half migration) opens, those fields land by additive Amendment to this struct definition.
- [x] **`AddressSpaceHandle`** newtype — `Handle<AddressSpace<()>>`-style phantom-typed handle, exact shape mirrors `TaskHandle` / `EndpointHandle` / `NotificationHandle` in [`kernel/src/obj/task.rs`](../../../../kernel/src/obj/task.rs) / [`kernel/src/obj/endpoint.rs`](../../../../kernel/src/obj/endpoint.rs) / [`kernel/src/obj/notification.rs`](../../../../kernel/src/obj/notification.rs). Generation-tagged; stale handles fail lookup with `AddressSpaceError::StaleHandle`.
- [x] **`AddressSpace::wrap_bootstrap(inner: M::AddressSpace) -> Self` constructor** — wraps an already-live `Mmu::AddressSpace` value (the one `mmu_bootstrap` activated post-T-016) into a kernel-object value **without** calling `Mmu::create_address_space`. Generation set to 0 (the bootstrap AS is "always-here" — capabilities to it never go stale via revoke in v1 because the bootstrap AS cannot be destroyed). Per [ADR-0028 §Simulation row 0](../../../decisions/0028-address-space-data-structure.md#simulation).
- [x] **`AddressSpace::root_frame(&self, mmu: &M) -> PhysFrame` accessor** — delegates to `Mmu::address_space_root(&self.inner)`. Diagnostic surface for the bootstrap-banner ("`AS root = 0x<pa>`") and host-test cross-checks.
- [x] **`AddressSpaceArena<M, const N: usize>` struct.** Fixed-size array `[Slot<M>; N]` where `Slot<M> = Option<AddressSpace<M>>` (or a typed enum mirroring `TaskSlot`'s shape — to match existing arena conventions; T-018 picks whichever the existing arena types use). Generation-counter-per-slot. `alloc(&mut self, address_space: AddressSpace<M>) -> Result<AddressSpaceHandle, ArenaFull>`; `get(&self, h: AddressSpaceHandle) -> Option<&AddressSpace<M>>`; `get_mut(&mut self, h: AddressSpaceHandle) -> Option<&mut AddressSpace<M>>`; `free(&mut self, h: AddressSpaceHandle) -> Result<AddressSpace<M>, AddressSpaceError>` (deferred to B4+ — no caller in v1; gated behind a `cfg(...)` if unused-code warns). v1's `bsp-qemu-virt` picks `N = 8` for the bootstrap AS + headroom for T-018's isolation tests; later BSPs may grow this.
- [x] **`AddressSpaceError` enum** — `#[non_exhaustive]`. Variants: `OutOfFrames` (PMM exhausted during `cap_create_address_space` root-frame alloc), `StaleHandle` (capability's generation tag mismatches the arena slot's), `ArenaFull` (no free slot in `AddressSpaceArena`), `CapError(CapError)` (passthrough from cap-resolution — wrong kind, out-of-bounds index, etc.), `MmuMapError(MmuError)` (passthrough from `Mmu::map`), `MmuUnmapError(MmuError)` (passthrough from `Mmu::unmap`). The wrap variants preserve the underlying [`CapError`](../../../../kernel/src/cap/mod.rs) / [`MmuError`](../../../../hal/src/mmu/mod.rs) taxonomy without flattening, so capability-side observers see exactly the cap-side or HAL-side failure. The wrappers (`cap_create_address_space` / `cap_map` / `cap_unmap`) thus expose a single unified return type — every cap-resolution failure surfaces as `Err(AddressSpaceError::CapError(_))`, every map failure as `Err(AddressSpaceError::MmuMapError(_))`, etc.

### Capability integration (`kernel/src/cap/mod.rs`)

- [x] **`CapKind::AddressSpace` variant** added to the `CapKind` enum. Discriminant placement follows the existing enum order (after `Task` / `Endpoint` / `Notification`; before any future variants). The enum's `#[non_exhaustive]` annotation (if present) keeps the addition additive for external matchers; existing match arms inside the kernel gain an `AddressSpace => ...` arm where they exhaustively pattern-match.
- [x] **`CapObject::AddressSpace(AddressSpaceHandle)` variant** added to the `CapObject` enum. Mirrors the `Task(TaskHandle)` / `Endpoint(EndpointHandle)` / `Notification(NotificationHandle)` shape.
- [x] **`Capability::object()` extension** — existing method returns `CapObject`; new arm handles the `AddressSpace` variant. No new surface; one arm in the match.
- [x] **Cap-resolution helper** `resolve_address_space_cap(cap: Capability, table: &CapabilityTable) -> Result<AddressSpaceHandle, CapError>` — mirrors the existing `resolve_ep_cap` / `resolve_task_cap` helpers in `kernel/src/sched/mod.rs`. Looks up the cap by index; validates `cap.kind == CapKind::AddressSpace`; pattern-matches `object()` to extract the typed handle. Returns `Err(CapError::WrongKind)` on mismatch.

### Cap-gated wrapper surface (`kernel/src/mm/address_space.rs`)

- [x] **`cap_create_address_space(parent_cap, mmu, pmm, arena, table) -> Result<AddressSpaceHandle, AddressSpaceError>`** — (i) validates `parent_cap.kind == CapKind::AddressSpace` (or future `Untyped` per Phase-C — in v1 the kernel-init holds an "ambient" cap on slot 0 that grants AS-creation authority; T-018 adopts the simplest v1 shape and forward-flags the Phase-C `Untyped` story); (ii) `pmm.alloc_frame()` → `PhysFrame` (returns `Err(AddressSpaceError::OutOfFrames)` on PMM exhaustion — this is the call that lifts UNSAFE-2026-0026's `Pending` note); (iii) `unsafe { Mmu::create_address_space(root) }` materialises the BSP-specific value; (iv) `arena.alloc(AddressSpace { inner, generation: <new> })` returns a typed `AddressSpaceHandle`; (v) `table.mint(CapKind::AddressSpace, CapObject::AddressSpace(handle))` mints the cap. The `unsafe` is the existing HAL contract per UNSAFE-2026-0023 — caller must guarantee `root` is page-aligned + zero-filled; PMM's `alloc_frame` already does both. Per [ADR-0028 §Simulation row 1](../../../decisions/0028-address-space-data-structure.md#simulation).
- [x] **`cap_map(cap, mmu, pmm, arena, va, pa, flags) -> Result<(), AddressSpaceError>`** — (i) resolves `cap` to `AddressSpaceHandle`; (ii) `arena.get_mut(handle)` → `&mut AddressSpace<M>` (returns `Err(StaleHandle)` on generation mismatch); (iii) `Mmu::map(&mut as_.inner, va, pa, flags, pmm)` → `Result<MapperFlush, MmuError>`; (iv) on success, `token.flush(mmu)` discharges the flush-token (calls `Mmu::invalidate_tlb_address(va)` per ADR-0027 §"`MapperFlush` flush-token discipline"); (v) on `MmuError`, wraps as `AddressSpaceError::MmuMapError(...)` and returns. Per [ADR-0028 §Simulation row 2](../../../decisions/0028-address-space-data-structure.md#simulation). This is the call site that lifts UNSAFE-2026-0025's `Pending` note.
- [x] **`cap_unmap(cap, mmu, arena, va) -> Result<PhysFrame, AddressSpaceError>`** — (i) resolves `cap`; (ii) `arena.get_mut(handle)` → `&mut AddressSpace<M>`; (iii) `Mmu::unmap(&mut as_.inner, va)` → `Result<(MapperFlush, PhysFrame), MmuError>`; (iv) on success, `token.flush(mmu)` discharges; returns the unmapped `PhysFrame` for caller-side handling (typically PMM `free_frame` once `cap_revoke(MemoryRegionCap)` lands in B4+); (v) on `MmuError`, wraps as `AddressSpaceError::MmuUnmapError(...)`. The intermediate L1/L2/L3 frames that become orphaned are deferred to the per-AS destroy path (B4+); T-018 wires the per-page unmap discipline only.

### Scheduler activation hook (`kernel/src/sched/mod.rs::yield_now`)

- [x] **Activation check + `Mmu::activate` call**, just before `cpu.context_switch(cur_ctx, nxt_ctx)` in [`yield_now`](../../../../kernel/src/sched/mod.rs) (per [ADR-0021](../../../decisions/0021-raw-pointer-scheduler-ipc-bridge.md)'s momentary-borrow discipline — the `&mut Scheduler<C>` borrow ends strictly before `cpu.context_switch`; the activation borrow is a *separate* momentary `&AddressSpace<M>` that the activate call consumes immediately).
  - Pre-check: `if next.address_space_handle != current.address_space_handle`. Single field comparison; no allocation; no cap resolution (the `Task` struct caches the handle at construction time so the activation path needs no `CapabilityTable` lookup).
  - On true: resolve `next.address_space_handle` → `&AddressSpace<M>` from the `AddressSpaceArena<M>` (the arena is published as a `StaticCell` next to the existing scheduler / IPC arenas; the momentary borrow is bracketed by an `unsafe` block following the existing ADR-0021 discipline — covered by the existing UNSAFE-2026-0014 umbrella's "momentary `&mut`/`&` across cooperative switches" scope rather than a new entry); call `Mmu::activate(&as_.inner)`. The activate call writes `TTBR0_EL1 = root + ASID=0`, then issues `ISB` + `DSB ISHST` + `TLBI VMALLE1` + `DSB ISH` + `ISB` per `bsp-qemu-virt/src/mmu.rs::QemuVirtMmu::activate` (global TLB invalidate is required because `TCR_EL1.AS = 0` — see the next bullet).
  - On false: skip the activation call entirely. Two tasks sharing the same AS pay zero overhead at switch.
- [x] **Global TLB flush at activate.** v1 has `TCR_EL1.AS = 0` (single ASID; per [ADR-0027 §"ASID"](../../../decisions/0027-kernel-virtual-memory-layout.md)) — no per-task ASID tagging, so every AS switch must globally invalidate the TLB to prevent stale-AS hits. The activation path issues `MSR TTBR0_EL1; ISB; DSB ISHST; TLBI VMALLE1; DSB ISH; ISB` (see [`bsp-qemu-virt/src/mmu.rs::QemuVirtMmu::activate`](../../../../bsp-qemu-virt/src/mmu.rs) and [ADR-0028 §Simulation row 3](../../../decisions/0028-address-space-data-structure.md#simulation)). The original ADR-0028 description called this row "activation-without-TLB-flush"; the implementation landed strictly more conservative to close the stale-TLB risk unconditionally. Cost: a full TLB flush on every AS switch — acceptable for v1's two-AS demo. ADR-0033 (high-half migration) will introduce ASID-based isolation when B5+ surfaces; until then the global flush is the safe choice.
- [x] **`Task` struct extension** — [`kernel/src/obj/task.rs`](../../../../kernel/src/obj/task.rs)'s `Task` struct gains an `address_space_handle: AddressSpaceHandle` field. Existing task constructors gain a parameter; the BSP-side `task_a` / `task_b` creation in `bsp-qemu-virt/src/main.rs` passes the bootstrap AS handle (slot 0) for v1's tasks. The scheduler's `yield_now` reads the field unchanged otherwise.

### BSP wiring (`bsp-qemu-virt/src/main.rs`)

- [x] **Bootstrap-AS wrap.** After `mmu_bootstrap()` returns and after `Pmm::new` initialises the PMM, the BSP:
  - Reads the L0 root frame address from a static (or recomputes it from the same linker symbol `mmu_bootstrap` used).
  - Calls `<QemuVirtMmu as Mmu>::wrap_existing_root(l0_root)` (the BSP-side companion to `Mmu::create_address_space` — does not allocate, does not zero-fill, does not modify any page-table state; simply constructs the `QemuVirtAddressSpace` value naming the already-live root). This is a new BSP-side surface; if `Mmu::create_address_space` already accepts a "this root is already live" path (recognising non-zero L0 entries), T-018 can route through that instead. The Approach §commit-1 picks one shape and documents the choice.
  - Wraps it with `AddressSpace::wrap_bootstrap(inner)` (where `inner` is the `QemuVirtAddressSpace` value from the previous step) → `AddressSpace<QemuVirtMmu>`.
  - Publishes into `AddressSpaceArena<QemuVirtMmu>` slot 0 via the arena's allocation API.
  - Mints the kernel-init's "AS authority" cap at `CapabilityTable[0]` (or wherever the existing init-cap convention places it; T-018 follows the precedent set by `Endpoint` / `Notification` cap-minting in the existing init path).
- [x] **`AddressSpaceArena<QemuVirtMmu, 8>` `StaticCell`** — published before the cooperative IPC demo's `Scheduler::start()` so the activation hook in `yield_now` can resolve the cached handle on every switch. Initialised in `kernel_entry` immediately after the PMM banner and before the GIC init.
- [x] **Banner line** — one new smoke trace line `tyrne: address-space-arena ready (1 / 8 slots used; bootstrap AS root = 0x<pa>)` immediately after the existing `tyrne: pmm initialized (...)` banner and before the existing `tyrne: timer ready (...)` banner. Per [ADR-0028 §Simulation row 0](../../../decisions/0028-address-space-data-structure.md#simulation).
- [x] **No use of `cap_map` / `cap_unmap` / `cap_create_address_space` by the v1 demo.** The cooperative IPC demo rides the bootstrap mappings; the cap-gated wrappers are exercised by T-018's host tests and by the second-AS smoke fixture (see below) but the runtime demo does not depend on them. The activation-on-context-switch hook is exercised by the smoke fixture too (two-task scenario where `task_a.address_space_handle == task_b.address_space_handle == slot 0` — the equality check short-circuits, demonstrating the zero-overhead-on-same-AS path).
- [ ] **Optional: second-AS smoke fixture.** A boot-time fixture that exercises `cap_create_address_space` + `cap_map` + activate for a second AS (then activates back to the bootstrap AS before the demo runs). This lifts UNSAFE-2026-0025 + UNSAFE-2026-0026's `Pending QEMU smoke verification` notes byte-for-byte. T-018 picks whether this lands as a boot-time fixture or only as a host test based on the runtime cost (the bootstrap delay should remain <10 ms on the QEMU virt baseline). If the runtime cost is unacceptable, the host tests alone exercise the path and the audit notes lift to "verified by host tests + Miri, runtime exercise deferred to B5+ first userspace AS creation".

## Approach

The implementation lands across several bisectable commits on a `t-018-address-space-kernel-object` branch (precedent: T-016 / T-017's bisectable-commit-chain pattern). Each commit ends green on `cargo fmt --check`, `cargo host-clippy -D warnings`, `cargo kernel-clippy -D warnings`, `cargo test`, `cargo kernel-build` so any commit can be cherry-picked or reverted without breaking the build:

1. **`kernel/src/mm/address_space.rs` — `AddressSpace<M>` struct + `AddressSpaceHandle` newtype + `AddressSpaceArena<M, N>` arena + `AddressSpaceError` enum.** Pure data-structure landing; no cap integration yet; no scheduler hook yet. Host tests added: arena alloc / get / stale-handle / wrap_bootstrap-shape — the data-structure-only tests from the list above. No `unsafe`. No HAL surface change. The mm/mod.rs gains a `pub mod address_space;` declaration; nothing else moves.
2. **`kernel/src/cap/mod.rs` — `CapKind::AddressSpace` variant + `CapObject::AddressSpace(AddressSpaceHandle)` variant + `resolve_address_space_cap` helper.** Capability-side surface. Existing exhaustive-match sites gain the new arm; host tests for those sites are unchanged (the new arm is added consistently with the existing pattern). Adds the `WrongKind` rejection path. No `unsafe`. No scheduler change.

## Design notes

- **Why one bundled task instead of T-018a + T-018b + T-018c?** Mirrors [T-016](T-016-mmu-activation.md) and [T-017](T-017-physical-memory-manager.md) which both bundled HAL/struct landing + BSP wiring + audit Amendments + cross-references in one task. The split would be artificial: kernel-object module + cap-variant landing + cap-gated wrappers + scheduler hook + BSP wiring are tightly coupled at the implementation level. A single task with several bisectable commits is cleaner than three tasks each with two commits.
- **Why generic `AddressSpace<M>` and not opaque?** Per [ADR-0028 §Decision outcome](../../../decisions/0028-address-space-data-structure.md#decision-outcome). The HAL trait already crosses the BSP boundary at the `Mmu::AddressSpace` associated type; pretending the kernel doesn't know `M` at the object surface would force a HAL trait revision (`reconstruct(&self, root: PhysFrame)`) so soon after the T-016 stabilisation, and force the BSP impl to be stateless w.r.t. AddressSpace.
- **Why a separate bootstrap-AS wrap path?** Per [ADR-0028 §Simulation row 0](../../../decisions/0028-address-space-data-structure.md#simulation). The MMU was activated by T-016's `mmu_bootstrap` using `.boot_pt` frames; calling `Mmu::create_address_space` on the live root would re-zero it (destroying the running translation tables). The wrap path names the already-live topology without modifying any page-table state. The BSP-side `wrap_existing_root` is a one-shot path the BSP must implement; T-018 picks whichever shape the existing `QemuVirtMmu` accepts most cleanly (either a new method or recognising the already-live case inside `create_address_space`).
- **Why no ASID in v1?** Per [ADR-0027 §"ASID"](../../../decisions/0027-kernel-virtual-memory-layout.md). v1 has `TCR_EL1.AS = 0` (single 8-bit ASID; only `ASID = 0` used). The activation hook writes `TTBR0_EL1` without an ASID change; TLB is not flushed on activate. Two AS coexisting + activation rely on T-018's tests pinning "map-before-activate" ordering. ADR-0033 will introduce ASID-based isolation when B5+ surfaces.
- **Why no destroy path in v1?** No v1 caller. The cooperative IPC demo creates two tasks that share the bootstrap AS and never destroy any AS. B4+ userspace introduces the first destroy caller; that ADR/task lands the page-table-tree walk + intermediate-frame return-to-PMM logic.
- **Why is `cap_create_address_space`'s `parent_cap` an `AddressSpaceCap` (not an `UntypedCap`)?** v1 simplification. The kernel-init holds an "ambient" AS cap on slot 0 that grants creation authority; T-018 adopts the simplest v1 shape. Phase-C `Untyped`-style frame ownership (per [seL4](https://sel4.systems/About/seL4-whitepaper.pdf)'s model) lands when `MemoryRegionCap` arrives (B4+); that ADR/task will rev `cap_create_address_space`'s parent-cap discipline.
- **Why is the activation hook in `yield_now` and not at `cpu.context_switch`?** Per [ADR-0021](../../../decisions/0021-raw-pointer-scheduler-ipc-bridge.md). The scheduler owns the borrow discipline; pushing the activate call into `cpu.context_switch` would either force `ContextSwitch` to take `&AddressSpace<M>` (mixing scheduler concerns into the CPU trait) or force the BSP-side `QemuVirtCpu::context_switch` to know about address spaces (mixing kernel-object concerns into the CPU impl). The scheduler-side hook is the right boundary.
